## Chapter 7. The Datadog AI Surface in Q2 2026

On June 10, 2025, Datadog used its annual DASH conference to announce three products simultaneously: AI Agent Monitoring at general availability, LLM Experiments at preview, and AI Agents Console at preview. The announcements came exactly one year after Datadog had made LLM Observability generally available at DASH 2024, and the juxtaposition was deliberate. The 2024 announcement had been about visibility into individual LLM calls. The 2025 announcements were about something structurally different: the capacity to observe, experiment with, and inventory the agents that orchestrate those calls. The platform that had learned to watch a model was learning to watch a mind.

In the week before DASH 2025, a team running a multi-model orchestration pipeline used LLM Observability to correlate a latency regression in their user-facing product with a specific prompt template change introduced three deployments earlier. The correlation required pulling LLM spans, APM traces, and infrastructure metrics into the same view, with a shared trace ID threading through all three signal types. That cross-signal correlation was not available from any specialized LLM observability tool. It was available from Datadog because the LLM spans shared a trace ID with the surrounding APM traffic, which shared a host with the infrastructure metrics. The data were already joined at the collection layer, before any analyst had asked a question about them.

What makes that team's experience relevant to this chapter is what it demonstrates about where Datadog's structural advantage sits and, equally, where it does not. The Datadog AI surface as of Q2 2026 is extensive. It spans from individual token counts to autonomous incident remediation. Understanding it requires separating three distinct waves of product development, examining each layer's mechanics with enough precision to know what it observes and what it misses, and then stepping back to define the edges of the surface honestly.

---

**The Datadog AI Surface.** The full set of AI-related product capabilities Datadog has shipped between August 2023 and Q2 2026, organized across three successive waves: LLM Observability, the agentic platform layer, and the autonomous-operations layer (Bits AI). The surface is not a single coherent product but a portfolio of capabilities built on a shared telemetry substrate. Each wave extends the one before it without replacing it, and each wave introduces its own blind spots alongside its coverage.

The problem, as Pomel described it in a March 2026 conversation with Lê-Quôc, was friction: when an incident happened and everyone was working hard to resolve it, the difficulty was agreeing on the data, on the sheer facts of what was happening (Pomel and Lê-Quôc, This Month in Datadog, March 26, 2026). ### The First Wave: LLM Observability Mechanics

LLM Observability is the foundational layer of the Datadog AI Surface. At its core, it is a tracing system specifically designed for generative AI workloads, integrated into the same data pipeline that serves Datadog APM. The integration is not cosmetic. When an LLM span is emitted by the ddtrace SDK or by an OpenTelemetry-instrumented application, it lands in the same trace store as the APM spans for the surrounding web service, the database calls that service makes, and the infrastructure host metrics for the servers running those services. The correlation is carried through the same trace ID and span ID fields that distributed tracing has used since the W3C Trace Context standard. An engineer investigating a production incident can begin at an LLM span, navigate to the parent APM trace that contains it, see the latency distribution for the HTTP endpoint that triggered the trace, and follow that endpoint to the infrastructure host where it is running, all within a single session and a single tool. The LLM span is a first-class citizen of the Datadog trace hierarchy, not a separate entity appended to it (Datadog LLM Observability documentation, accessed Q2 2026).

#### The Span Taxonomy

The central organizing concept of LLM Observability is a structured classification of span kinds that maps the components of an AI pipeline onto a type hierarchy within the trace system.

**The Span Taxonomy.** Datadog's structured classification of LLM Observability spans by kind: `llm`, `tool`, `retrieval`, `embedding`, `workflow`, `agent`, and `task`, which maps the steps of an AI pipeline onto the trace hierarchy in a way that preserves correlation with standard APM spans. The taxonomy allows the trace backend to render an agentic execution as a structured tree with typed nodes, each carrying the attributes appropriate to its kind, rather than as an undifferentiated pile of generic spans. The taxonomy exists because the AI pipeline is not a single operation but a composition of distinct operation types, each with a different cost structure, failure mode, and diagnostic profile.

The seven span kinds correspond precisely to the layers of **the AI Application Surface**, introduced in Chapter 2 as the complete set of new observable components generative AI systems introduce. The `llm` span kind records a call to a language model provider. It captures the provider name, the model identifier as requested by the application and as returned by the provider, the input and output token counts, the latency from request dispatch to response receipt, the finish reason (stop, length, tool call, content filter, or error), and the estimated cost derived from the token counts and the provider's published pricing at the time of the call. The `embedding` span kind records an embedding API call, capturing the model identifier and the input text count. The `retrieval` span kind records a vector search operation, capturing the data source identifier, the number of documents returned, and any error type. The `tool` span kind records a tool invocation, capturing the tool name and the input and output payloads. The `workflow` span kind is a grouping span that contains the child spans representing a multi-step pipeline execution. The `agent` span kind represents an agent's primary execution span, containing whatever sub-spans the agent generated during one invocation. The `task` span kind represents a discrete sub-task dispatched by an agent to a worker or a sub-agent (Datadog LLM Observability documentation, accessed Q2 2026).

Together, these seven span kinds allow a trace for a multi-step agentic workflow to be rendered as a structured hierarchy. The root span is the `workflow` or `agent` span. Beneath it sit the `tool`, `retrieval`, `embedding`, and `llm` spans that the agent generated, each correctly positioned as children of the span that triggered them. The parent-child relationships in this hierarchy are enforced by the SDK's context propagation, not reconstructed by the backend from heuristics. When the SDK's `@workflow` decorator wraps the top-level function and the `@agent`, `@tool`, and `@llm` decorators wrap the component functions, the span hierarchy is built correctly at emission time, before any data reaches the Datadog backend.

This is the resolution that **the Span Topology Problem**, introduced in Chapter 4, describes as structurally difficult to achieve in agentic systems. The Span Taxonomy is Datadog's answer to that structural difficulty at the SDK level: by providing typed decorator annotations and corresponding auto-instrumentation hooks for supported frameworks, the SDK attempts to enforce a coherent hierarchy at the point of trace creation rather than relying on the trace backend to reconstruct one from orphaned spans. Where auto-instrumentation does not cover the framework in use, the Span Taxonomy still applies, but the application team must apply the decorators manually. The `LLMObs.annotate(input_data, output_data, metadata, metrics, tags)` method allows any span to be annotated with the typed attributes that make it queryable as a typed LLM Observability span (Datadog LLM Observability documentation, accessed Q2 2026).

The practical consequence of typed spans is query precision. A team investigating a cost spike can filter the Trace Explorer to spans of kind `llm` with a specific model provider and sort by estimated cost to find the most expensive calls. A team investigating a quality regression can filter to spans of kind `retrieval` and examine the similarity scores for the documents returned. A team investigating a scope creep event like those cataloged in **the Six Incident Categories** introduced in Chapter 6 can filter to `tool` spans and examine the input arguments to identify which tools the agent called and with what parameters. None of these queries are possible in a system where all spans are generic and carry no type information.

What the Span Taxonomy does not resolve is the cardinality problem that **the Cardinality Cliff**, introduced in Chapter 3, defines. A `tool` span with a `tool_name` tag whose value is drawn from a small, bounded set of tool names is low-cardinality and cheap to index. A `tool` span whose tags include any field derived from the tool's input arguments, which may be natural language, is potentially unbounded in cardinality. The Span Taxonomy types the spans; it does not constrain the tag values that teams choose to attach to them. That discipline is the territory of Chapter 9's treatment of Cardinality Control.

#### The 1 MB Payload Ceiling

Every LLM span can carry the prompt and response text as a payload. This is the feature that makes LLM Observability more than a latency and token-count recorder: it allows teams to inspect the actual conversation content associated with a trace, to investigate quality regressions by examining what the model was asked and what it said, and to populate evaluation datasets from production traffic.

The payload capacity is bounded.

**The 1 MB Payload Ceiling.** The hard per-span payload size limit enforced by Datadog LLM Observability, which truncates prompt and response payloads that exceed one megabyte, creating a structural tension between capturing full conversational context and staying within the trace storage budget. The ceiling applies to the combined input and output payload of a single span. A prompt that includes a 50,000-token retrieved context, a system prompt, and a conversation history can exceed one megabyte before the response is added (Gemini report on Datadog AI Era, Q2 2026; Datadog LLM Observability documentation, accessed Q2 2026).

The mechanics of the ceiling deserve precision. One megabyte of UTF-8 text is approximately 250,000 to 300,000 characters, depending on the character set. A 50,000-token context window in a modern LLM corresponds roughly to 175,000 to 200,000 words, which in turn corresponds to approximately 1.2 to 1.5 megabytes of UTF-8 encoded text. This means that any single LLM call consuming a context window larger than approximately 35,000 tokens is likely to exceed the ceiling. Long-context models now commonly support 128,000 token context windows, 200,000 token windows, and in some cases over one million tokens. The ceiling was set at a time when production LLM calls rarely consumed more than a few thousand tokens. The rapid expansion of context window sizes since the ceiling was established has made the constraint structurally tighter with each new model generation (Gemini report on Datadog AI Era, Q2 2026).

The implications of the 1 MB Payload Ceiling are not merely operational. They are architectural. A team building a long-context RAG system where the retrieval stage injects large document chunks into a context window faces a real choice: truncate the payload at the LLM Observability layer and lose the conversational evidence that would allow quality investigation, or restructure the trace so that retrieval content is stored separately from the generation span. Truncation produces incomplete traces that cannot support post-incident quality analysis. Separation requires custom instrumentation and means that the retrieval content is no longer co-located with the generation span in the same trace view, degrading the ergonomics of investigation without eliminating the problem.

Datadog's recommended mitigation is to log the full prompt and response to Flex Logs or a cold archive and store a content hash or a reference identifier in the LLM span's metadata fields. The hash allows the full content to be retrieved from the log store when an investigation requires it, while keeping the span itself within the billing and storage constraints. For systems where full payload capture is required at the span level, the 1 MB Payload Ceiling is a hard constraint that the Q2 2026 product version does not allow teams to configure above (Datadog LLM Observability documentation, accessed Q2 2026). The question of whether future product versions will allow the ceiling to be raised is one the Gemini report identifies as open as of Q2 2026, noting that the boundary creates increasing friction for long-context model applications.

#### The Bundled Sensitive Data Scanner

Prompt and response payloads present a data governance problem that the 1 MB Payload Ceiling does not address. A team that captures full conversation content for quality inspection also captures whatever sensitive information users included in their prompts: names, addresses, account numbers, health information, financial data. The **Sensitive Data Scanner** (a Datadog security product that predates the AI era) is now integrated natively with LLM Observability through a managed scanning group.

When the Sensitive Data Scanner is configured for LLM Observability, it runs before payload ingestion and redacts or hashes fields that match its configured detection rules. The rules cover standard PII patterns: credit card numbers, Social Security number formats, email addresses, phone numbers, and other patterns defined by the organization's compliance requirements. The scanner applies in the Datadog Agent, before the payload reaches the trace store, which means that a redacted payload never exists in the Datadog backend in unredacted form. The scanning configuration is managed through the Datadog UI, with a pre-configured managed scanning group specifically for LLM Observability that teams can enable without writing custom rules (Claude research report, "Datadog in the generative AI era," Q2 2026; Datadog LLM Observability documentation, accessed Q2 2026).

The limitation of this approach is the same limitation that Chapter 6's treatment of the PII leak incident category identified at the principle level: pattern-based PII detection does not catch indirect exfiltration or contextually sensitive content that does not match a standard format. The Sensitive Data Scanner reliably blocks credit card number patterns and Social Security number patterns. It does not block a response where the model summarizes a private document in natural language that contains no standard PII patterns, as occurred in the Slack AI and EchoLeak incidents documented in Chapter 6. That class of leakage requires behavioral monitoring on output structure and content. The scanner is a necessary component of LLM Observability's data governance posture; it is not sufficient by itself.

For organizations that require redaction to occur entirely within their own infrastructure, before any data leaves the environment, Datadog supports the Observability Pipelines Worker path: a processor deployment where the Sensitive Data Scanner runs in the organization's own environment and the redacted output is sent to Datadog rather than the original payload. This approach satisfies HIPAA-class requirements for organizations that cannot allow raw PHI to cross a cloud boundary. The Observability Pipelines Worker also provides a second function relevant to LLM Observability: Log Volume Control processors that filter out low-value AI payload logs before they reach Datadog's ingestion layer, reducing the log volume and associated cost for teams that log AI payloads at high rates (Claude research report, "Datadog in the generative AI era," Q2 2026).

#### Out-of-the-Box Evaluators

LLM Observability ships two categories of out-of-the-box automated evaluators: hallucination detectors and prompt injection detectors. Both run as LLM-based judges that score traces in near-real-time as they arrive in the platform, without requiring the team to configure a custom evaluation pipeline.

The hallucination evaluator uses a secondary LLM call to assess whether the model's response is grounded in the retrieved context. For RAG systems, it examines the retrieval span's output documents and the generation span's response, and produces a grounded or hallucinating assessment along with a reasoning string. The prompt injection evaluator assesses whether the model's input contains an attempt to override the system prompt or redirect the model to produce content inconsistent with its configured purpose. Both evaluators produce structured scores that land in the same Trace Explorer view as the latency and token-count metrics, making quality signals available in the same tool that the on-call engineer uses for operational investigation (Datadog LLM Observability documentation, accessed Q2 2026).

These evaluators address a part of the evaluation gap that **the Eval/Ops Split**, introduced in Chapter 5, defined. Datadog has made a pragmatic architectural choice: rather than waiting for the evaluation discipline to remain entirely outside the operational platform, it has shipped two high-frequency evaluation patterns as automated features within LLM Observability. The evaluators produce structured scores that sit alongside operational metrics in the same trace view.

The tradeoff is transparency and depth. The out-of-the-box evaluators are black-box classifiers whose rubrics are defined by Datadog and cannot be customized. For teams with domain-specific quality requirements, a response that scores as "hallucinating" under the default evaluator may be acceptable in their context, and a response that scores as "grounded" may contain domain-specific inaccuracies the generic evaluator cannot detect. The evaluators are a starting point for quality awareness in production, not a complete quality assurance system. The distinction is significant: a team whose quality requirements are satisfied by the generic hallucination evaluator can use LLM Observability as its sole evaluation surface. A team whose quality requirements are domain-specific, regulated, or require custom rubrics will find the out-of-the-box evaluators insufficient and will need the specialized evaluation tooling that Chapter 10 assesses.

Datadog supports custom evaluations through the LLM Observability SDK. Teams can submit evaluation scores from external judge systems, from human annotation workflows via Annotation Queues, and from LLM Experiments datasets to the same trace backend. The Automation Rules feature allows filter-based routing of traces to Annotation Queues and datasets at configurable sampling rates. A rule might route all traces where the hallucination evaluator scored "hallucinating" to an annotation queue at 5 percent sampling, populating a continuous review queue without manual trace selection. Annotation queues hold up to 1,000 records before the automation pauses, providing a natural throttle on the review workload (Datadog LLM Observability documentation, accessed Q2 2026).

#### The Cost View

One feature of LLM Observability that warrants explicit treatment at this chapter's level of analysis is the cost view. Datadog automatically calculates estimated LLM provider costs at the span level and aggregates them up through the trace hierarchy, so that a workflow span shows the total estimated cost of all LLM calls it contains, and an application's cost view shows the cost distribution across models, prompt templates, and user session categories.

The cost estimation uses the token counts captured in each `llm` span and applies the provider's published pricing for the model in question. The result is an estimate, not an exact bill: provider pricing changes, cached token pricing differs from standard input token pricing, batch API pricing differs from real-time API pricing, and some providers offer private pricing that differs from their published tiers. The cost view is calibrated for directional insight and comparative analysis, not for exact billing reconciliation (Datadog LLM Observability documentation, accessed Q2 2026; Claude research report, "Datadog in the generative AI era," Q2 2026).

The directional insight is nonetheless valuable. A team that can see that 23 percent of its total LLM cost is concentrated in 4 percent of its sessions, and that those sessions correlate with a specific entry point in the product, has the information it needs to decide whether to add a guardrail, a complexity classifier, or a session-level cost budget. Without the cost view, those sessions are invisible at the application level even if they are visible on the provider's monthly invoice. The cost view makes the **Runaway Cost Loop** pattern, introduced in Chapter 6, observable within the Datadog platform in near-real-time rather than at billing-cycle close.

### The Second Wave: The Agentic Platform Layer

The DASH 2025 announcements represent a structural shift in what Datadog is monitoring. LLM Observability monitors calls to models. The agentic platform layer monitors the systems that orchestrate those calls.

#### AI Agent Monitoring and the Distinction from AI Agents Console

The analytical distinction between instrumentation-depth and discovery-breadth separates two DASH 2025 announcements that are easy to conflate: AI Agent Monitoring (at GA) and AI Agents Console (at Preview). The distinction matters because the two products answer different questions and require different things from the teams that use them.

AI Agent Monitoring is the extension of LLM Observability's tracing capabilities to frameworks that orchestrate multi-step agent workflows. It uses the same Span Taxonomy described above, but adds specific support for the graph structures that frameworks like LangGraph, CrewAI, OpenAI Agents SDK, and Google ADK produce. For a LangGraph workflow, AI Agent Monitoring instruments the `CompiledGraph.invoke` and `Pregel.invoke` methods and renders the resulting trace as a graph view that reflects the DAG structure of the workflow definition. Nodes in the graph correspond to agent spans in the trace, edges correspond to parent-child relationships, and the graph view allows the engineer to follow execution across the branching paths the workflow took on any particular invocation. For a workflow that evaluated three paths before selecting one, all three paths are present in the trace as distinct sub-trees under the root agent span. For CrewAI, the monitoring instruments Crew kickoffs and traces the role-task-tool hierarchy as nested spans, so that the trace shows which crew member agent handled which task and which tools that agent called. For the OpenAI Agents SDK, the integration intercepts the SDK's built-in tracing and converts it to LLM Observability format via a trace processor rather than re-instrumenting the SDK from scratch, which means that OpenAI Agents SDK traces and LLM Observability spans share the same representation in the Datadog backend (Datadog LLM Observability documentation, accessed Q2 2026; DASH 2025 AI Agent Monitoring GA announcement, June 10, 2025).

The agentic graph view also exposes memory state for frameworks that maintain state across steps: CrewAI short-term and long-term memory appears in the agent span's metadata, as does LangGraph state. This matters for debugging agent behaviors that depend on accumulated context: a session where the agent's decisions shifted in the fifth step but the context that drove that shift was accumulated in steps two and three is debuggable from the trace when the state is recorded, and not debuggable without it.

AI Agents Console is a separate product at Preview status as of Q2 2026. Where AI Agent Monitoring focuses on traces produced by agent workflows that the team has built and instrumented, AI Agents Console focuses on inventory and governance for agents operating in the environment regardless of whether they were built by the team. It collects logs from agentic developer tools and surfaces them as a centralized catalog showing which agents are active, what actions they have taken, what they have cost, what their error rates are, and what their engagement metrics are. The inventory is built from log ingestion, not from SDK instrumentation, which means that agents whose source code the team does not control can appear in the Console if they produce logs that reach Datadog.

A documented use case from the December 2025 period involved monitoring Claude Code adoption within an engineering organization. The AI Agents Console logs showed which engineers were using Claude Code, what services it was interacting with, and what patterns of tool calls it was generating, without requiring the team to instrument Claude Code itself (Datadog blog, "Monitor Claude Code adoption in your organization with Datadog's AI Agents Console," December 2025). This is the governance use case: understanding what agents are doing in the environment without requiring consent or cooperation from the agent's developers.

The distinction between the two products, stated as precisely as the documentation supports: AI Agent Monitoring instruments code that the team controls and produces structured LLM Observability traces with the full Span Taxonomy. AI Agents Console collects logs from agents the team may not control and surfaces them as an inventory and governance layer. The first is instrumentation-depth, requiring setup and maintenance. The second is discovery-breadth, requiring only that agent activity produces logs that reach Datadog. Both are needed for a complete picture of agent behavior in a production environment, but they provide it through different mechanisms and with different coverage guarantees.

#### LLM Experiments

LLM Experiments, also announced at DASH 2025 in Preview, bridges the operational and evaluation domains in a way that the standard LLM Observability trace view does not. It allows teams to take a dataset of production traces, define a set of prompt variants or model configurations, run those variants against the dataset, and compare the outputs along evaluation dimensions the team defines. The results land in the same interface as the operational traces, allowing a team to compare a proposed prompt change against baseline production performance before deploying it.

The mechanics of LLM Experiments follow a dataset-first pattern. The team exports traces from the LLM Observability Trace Explorer, filtering by application, time range, evaluation outcome, or any other supported facet. Those traces become an experiment dataset. The team defines the variants, which may be alternative prompt templates, different model providers, different temperature settings, or any combination of parameters that the LLM call accepts. Datadog runs each trace's input through each variant and collects the outputs. The outputs are scored by the team's configured evaluators (the out-of-the-box hallucination and prompt injection evaluators, custom LLM judge evaluators, or human annotation queues). The comparison view shows the distribution of scores, latency, and estimated cost across variants, allowing the team to select the variant that best balances quality, cost, and performance before changing production configuration (Datadog blog, "Create and monitor LLM experiments with Datadog," December 2025; DASH 2025 announcements, June 10, 2025).

The gap that LLM Experiments does not close is prompt management lifecycle. The experiments product supports running experiments against dataset snapshots. It does not provide the version-controlled prompt registry with draft, review, and publish states that specialized tools like Langfuse Prompt Experiments and LangSmith prompt versioning offer. A team that needs to manage 50 prompt variants across 8 services, track which version is live in production, roll back a specific prompt on an incident signal, and maintain an approval workflow for production prompt changes will find Datadog's experiment interface less complete for that specific workflow than those alternatives. The Datadog tooling answers the question "how does this variant perform compared to the current prompt?" It does not answer the question "what is the current approved prompt for this service, and who approved it?" Chapter 10 treats this gap in the competitive landscape.

### The Third Wave: Bits AI and Autonomous Operations

The third wave of the Datadog AI Surface moves the platform from observing AI systems to acting on the operational signals those systems produce. The Bits AI product family sits at this layer. It is the most analytically interesting part of the Datadog AI Surface because it is the layer where the platform is no longer just an instrument but an agent in its own right, which creates the recursive question that Chapter 11 addresses directly: who monitors the monitor?

#### Bits AI SRE: GA at December 2, 2025

Bits AI SRE became Datadog's first generally available AI agent product on December 2, 2025. The product is a context-aware investigation agent that runs inside the Datadog platform and responds to triggered monitors by iteratively hypothesizing, querying telemetry, validating hypotheses against the evidence, and generating a root cause summary. It is not a dashboard or a report. It is an agent that takes actions within the Datadog environment: it queries spans, metrics, and logs; it correlates evidence across telemetry types; it reaches into runbooks and monitor configuration pages for context; and it produces a structured investigation output that the on-call engineer receives before they have opened the trace explorer themselves (Datadog press release, December 2, 2025; Datadog Bits AI SRE GA announcement video, December 2, 2025).

The investigation architecture of Bits AI SRE is worth describing in structural terms, because understanding what it does informs what governance it requires. When a monitor triggers and auto-investigation is enabled on that monitor, Bits AI SRE begins a hypothesis-testing loop. It generates an initial set of hypotheses about the root cause of the alert based on the monitor's configuration, the alert's context, and any runbook content associated with the monitor. It then queries the Datadog telemetry available to it (spans, metrics, logs, infrastructure data, past incident records) to validate or invalidate each hypothesis. Invalidated hypotheses are discarded. Validated hypotheses are drilled into with a second round of more specific queries. The loop continues until the agent has identified a root cause it can defend with evidence, or until it has exhausted its configured exploration budget without finding a definitive cause. The output is a structured summary presented in the Datadog Incident Management UI and, if Slack is connected via `/dd connect`, delivered to the incident Slack channel with the full analysis and a natural-language explanation of the evidence (Claude research report, "Datadog in the generative AI era," Q2 2026; Datadog Bits AI SRE documentation, accessed Q2 2026).

The entry points for Bits AI SRE at Q2 2026 span the alert lifecycle: monitor alerts at GA, APM latency graphs at GA, Watchdog stories at GA, Incident Management UI at GA, and Slack via the `@Datadog investigate` command at GA. Synthetic Browser and API test failures and freeform prompt entry remain at Preview with daily rate limits as of Q2 2026. Auto-investigation is configured per monitor in the Bits AI SRE Monitors page. One operational constraint worth noting for teams planning large-scale deployments: Terraform and API enablement of auto-investigations is not supported as of May 2026. The configuration is UI-only, which means that deploying auto-investigation across an organization's full monitor catalog requires manual configuration through the web interface rather than infrastructure-as-code (Claude research report, "Datadog in the generative AI era," Q2 2026, citing Datadog documentation). For organizations with dozens of monitors this is manageable. For organizations with thousands of monitors across many services, this constraint is a genuine deployment obstacle.

Beyond the investigation function, Bits AI SRE performs seven categories of triage actions: sending Slack or Teams messages, declaring incidents, paging responders, creating Case Management cases, creating Jira tickets, and triggering Workflow Automation actions. The Incident AI component posts incident summaries on Slack channel join, detects related incidents within a 20-minute window, supports the `/dd incident summary` command, and generates AI-enhanced notification templates. The Bits AI SRE to Dev Agent handoff, described in the following section, is in Preview as of Q2 2026.

The MTTR reduction claims associated with Bits AI SRE from documented customer cases, including iFood's reported 70 percent reduction and Energisa's root cause identification in under four minutes, are treated at length in Chapter 11 alongside the full governance analysis of what autonomous remediation means for trust verification and scope authority. The shorthand reference here is precise: those claims are from named customers cited in official Datadog materials, and they represent the class of outcome that motivates Bits AI SRE adoption. The governance question they raise is the core analytical territory of Chapter 11, not this one.

#### Bits AI Dev Agent: Partial GA

Bits AI Dev Agent had reached GA status in the Error Tracking context as of Q2 2026, with Preview status elsewhere. Its function is to take a root cause identified by an error or by Bits AI SRE and generate a code fix in the form of a pull request. The PR is generated in a per-repo sandbox, undergoes Datadog-run code scanning before surfacing, and never auto-merges under any configuration. Auto-push to the repository is an opt-in setting configurable per product context. The recommended configuration restricts auto-push to code-only workflows rather than telemetry-driven actions, to limit the attack surface for prompt injection via log content, a risk category that the PII and prompt injection incidents in Chapter 6 make operationally concrete (Claude research report, "Datadog in the generative AI era," Q2 2026, citing Datadog documentation).

The Bits AI Dev Agent workflow requires the Datadog GitHub App with the `bits_dev_write` permission and unified service and version telemetry tags. It runs in a per-repo sandbox and scans all generated code before surfacing the PR. The product's scope includes Error Tracking (generating fixes for detected errors at GA), Code Security SAST (generating fixes for static analysis findings, GA in that context), and general code assistance from APM or SRE findings (Preview). The never-auto-merge constraint applies universally across all entry points and contexts (Datadog Bits AI Dev Agent documentation, accessed Q2 2026).

The Bits AI SRE to Dev Agent handoff in Preview connects the investigation and remediation loops: when Bits AI SRE identifies a code-level root cause for an incident, it can hand that conclusion directly to Dev Agent for a fix PR generation without the engineer having to manually re-specify the problem context in a separate workflow. The chain from monitor alert to root cause summary to pull request operates entirely within the Datadog environment, gated by the engineer's review at the PR stage.

#### Bits AI Security Analyst: GA at April 2026

Bits AI Security Analyst reached general availability in April 2026. It operates within Cloud SIEM, autonomously triaging SIEM signals, classifying their severity, and generating investigation summaries with action recommendations. The initial scope covers AWS CloudTrail as its primary log source. When viewing Cloud SIEM signals, teams can filter to show only those that Bits AI Security Analyst has already investigated and classified, and can open any signal to see the full analysis and conclusion alongside the raw signal data (Forrester DASH analysis, Q2 2026; Datadog This Month in Datadog, April 2026).

Bits AI Security Analyst is distinct from Bits AI SRE in both scope and integration point. SRE operates on monitor alerts and APM signals across the full stack, focusing on reliability and performance. Security Analyst operates on SIEM security signals, focusing on threat detection and response. The two agents do not currently share a unified governance console, which means that an organization deploying both must configure and supervise them separately. The question of whether a unified Bits AI governance surface will emerge is one the current Q2 2026 product state does not answer.

### Watchdog: The Precursor Layer

Watchdog is the product that makes the Datadog AI Surface's trajectory legible as a coherent narrative. It predates all three waves described above and represents the platform's first systematic investment in machine-intelligence-driven telemetry analysis.

Watchdog is an always-on anomaly detection system that runs across Datadog's full telemetry intake. It analyzes infrastructure metrics, APM traces, log patterns, and RUM sessions without requiring manual threshold configuration, identifying statistical anomalies and performance regressions that it surfaces through two channels: Watchdog Alerts (severity-filtered notifications that can be consumed as monitors) and Watchdog Insights (embedded panels within the Log Explorer, Trace Explorer, and Kubernetes Explorer that highlight anomalous patterns in the context of whatever query the engineer is currently viewing). When a new deployment is detected, Watchdog's Automatic Faulty Deployment Detection compares the error rate distribution of the new version against the prior version's distribution and flags the deployment as faulty if the new version introduces a statistically significant increase in novel error types (Datadog Watchdog documentation, accessed Q2 2026).

Watchdog operates on ML models trained on Datadog's telemetry corpus. It does not operate on foundation models. The relationship between Watchdog and the Toto time-series foundation model research program is one of trajectory, not current integration: Toto's architecture, trained entirely on Datadog's internal telemetry, represents the research direction that the platform's engineering leadership intends to bring into production anomaly detection over a multi-year horizon. Toto is not a Watchdog feature as of Q2 2026 (Claude research report, "Datadog in the generative AI era," Q2 2026). Watchdog is the product that the platform's documentation recommends as the entry point for Bits AI SRE: Watchdog stories fire on anomalous patterns and become triggers for Bits AI SRE investigations. The two products are designed to work together at their interface, with Watchdog producing the anomaly signal and Bits AI SRE performing the structured causal investigation.

Watchdog's noise reduction is configurable through monitor scoping. Teams can scope Watchdog monitors to specific environments, alert categories, and team tags sourced from the Software Catalog. The recommended practice for AI workloads is to tag AI services with a dedicated environment value and a team tag that isolates them from the non-AI estate, so that the wide legitimate variance in AI workload metrics does not contaminate the Watchdog baseline for traditional services. There is no published AI-specific Watchdog tuning guide as of May 2026 (Claude research report, "Datadog in the generative AI era," Q2 2026). The absence of such a guide is a gap: teams instrumenting AI workloads for the first time have no documented starting point for Watchdog configuration tailored to the non-stationary distribution properties that **the Three-Pillar Strain**, introduced in Chapter 1, characterizes as structurally different from traditional APM workloads.

The trajectory from threshold-based alerting to ML-based anomaly detection (Watchdog) to foundation-model-based telemetry understanding (Toto and BOOM) is the subject of Chapter 11's analytical treatment of the Autonomous Monitoring Loop. The shorthand reference here is sufficient: Toto and BOOM are research assets released under the Apache 2.0 license on May 21, 2025, whose relevance to the production platform is prospective rather than current (Datadog press release, May 21, 2025).

### How the Three Waves Connect

Describing the three waves as sequential is accurate as a product history but misleading as an architectural description. In production, the three waves operate as a single integrated system whose components hand data and control to each other in structured ways.

The flow begins with instrumentation. A team deploys the ddtrace SDK with LLM Observability enabled. Auto-instrumentation fires for every supported framework call: each LLM call generates an `llm` span, each retrieval step generates a `retrieval` span, each tool invocation generates a `tool` span. The SDK organizes these into the Span Taxonomy hierarchy and ships the spans to the Datadog Agent, which forwards them to the backend. At the Agent, the Sensitive Data Scanner redacts any PII patterns before the payload leaves the environment. The spans land in the trace store, where they are correlated with the APM spans and infrastructure metrics that share their trace ID.

Watchdog receives the metrics from those same spans. It monitors the per-service latency distributions, error rates, and token-cost time series for anomalies. When a Watchdog alert fires on an AI-tagged service, it appears in the Watchdog Alerts panel and triggers auto-investigation if Bits AI SRE is configured on that monitor. Bits AI SRE receives the alert, accesses the same trace store that the LLM Observability SDK populated, queries the spans for the time window around the alert, and begins its hypothesis-testing loop. It may examine latency distributions across the `llm` spans, look for token-count anomalies in the `tool` spans, compare the current session's retrieval similarity scores against a rolling baseline, and cross-reference the findings with past incident records in Incident Management. When it identifies a root cause, it posts the summary to Slack and, if the root cause is code-level, it can hand off to Bits AI Dev Agent for a fix PR.

The LLM Experiments product draws on the same trace store at a different cadence. An engineer investigating a quality regression uses the Trace Explorer to filter to traces where the out-of-the-box hallucination evaluator scored "hallucinating," exports those traces as an experiment dataset, defines a set of prompt variants, and runs the experiment. The results come back to the same interface, scored by the same evaluators. The experiment data is annotated back to the production traces that seeded it, closing the loop between production observation and model improvement.

The AI Agents Console draws on a different data path: log ingestion from agentic developer tools that may not be instrumented with the LLM Observability SDK at all. Its inventory of agents running in the environment is built from whatever logs those agents produce, assembled into a governance catalog that sits alongside but separately from the instrumented trace hierarchy.

The architecture is layered but not monolithic. Each product in the surface can be deployed independently. A team that only wants LLM Observability tracing does not need Bits AI SRE. A team that only wants Watchdog does not need LLM Experiments. The value of deploying all three waves together is precisely the data sharing that the multi-model correlation example at this chapter's opening illustrates: when the LLM spans, the APM spans, and the Watchdog alerts all share the same underlying data model, the cross-signal investigations that would require context switching between four separate tools in a heterogeneous stack happen within a single interface.

The cost of that integration is the per-span billing unit that governs LLM Observability, the per-investigation billing unit that governs Bits AI SRE, the Datadog APM and infrastructure subscription that the full-stack correlation depends on, and the cardinality discipline that the Span Taxonomy requires to avoid the **Cardinality Cliff** at scale. The Datadog AI Surface is not free to operate at scale. It is a design choice to pay for integration rather than to pay for fragmentation. Chapter 9 and Chapter 10 together make that trade-off precise.

### The Evaluation Surface: What LLM Observability Owns and What It Does Not

The out-of-the-box evaluators and the LLM Experiments product together define what Datadog's evaluation surface looks like as of Q2 2026. It is worth being precise about what that surface owns and what it defers.

Datadog owns the plumbing: the trace store that accumulates production LLM interactions, the Trace Explorer that makes those interactions queryable, the Automation Rules that route them to annotation queues and experiment datasets, and the experiment runner that tests prompt variants against dataset samples. These are the infrastructure of an evaluation workflow. They allow a team to build the data pipeline from production traffic to labeled dataset to experiment to production without leaving Datadog.

Datadog does not own the rubric design, the domain expertise, or the human judgment that make evaluation meaningful. A hallucination evaluator that scores responses as "grounded" or "hallucinating" is using a rubric that Datadog defined based on a general conception of groundedness. A financial services team evaluating whether a wealth advisory response cites the correct fund product for the user's risk profile needs a rubric defined by its compliance team, calibrated against its domain experts, and validated against its regulatory requirements. None of that rubric design is something the Datadog platform provides. The platform provides the infrastructure to apply whatever rubric the team defines. The rubric itself is the team's responsibility.

This is why the Thoughtworks Technology Radar, in its assessment of Datadog LLM Observability, frames the product as the natural choice for organizations already on Datadog but not necessarily the first choice for AI-native teams needing advanced evaluation workflows (Perplexity report, "Datadog, Generative AI, and Observability in 2026," Q2 2026, citing Thoughtworks Technology Radar). The framing is accurate. Datadog's evaluation surface is infrastructure-complete and rubric-thin. Specialized evaluation tools like Arize Phoenix and Langfuse are infrastructure-lighter and rubric-richer. The choice between them is a choice about where to invest in depth. Chapter 10 organizes that choice into the Decision Framework that production AI teams can apply to their specific circumstances.

What the Q2 2026 evaluation surface does well is continuity. The Automation Rules that route production traces to annotation queues and experiment datasets operate continuously, without manual trace selection. The experiment results reference the production traces that seeded them. The annotation labels applied by human reviewers flow back into the same trace store that the Trace Explorer queries. A team that invests in building the evaluation workflow within Datadog gets a system that is always drawing on fresh production data rather than on a static validation set that may have drifted from the distribution the model encounters in practice. That continuity is the specific advantage of owning the evaluation surface within the operational monitoring platform, and it is an advantage that no standalone evaluation tool can fully replicate without a deeply integrated data pipeline from the operational backend.

### What the Surface Covers and Where It Stops

With all three waves and the supporting capabilities mapped, the shape of the Datadog AI Surface becomes precise enough to assess at its edges. What it covers at Q2 2026: end-to-end tracing of LLM calls and agentic workflows correlated with APM and infrastructure telemetry in a single trace store; a structured span vocabulary with seven typed span kinds for every component of the AI pipeline; automated PII redaction at ingestion via the bundled Sensitive Data Scanner, with an in-environment alternative through the Observability Pipelines Worker; out-of-the-box hallucination and prompt-injection evaluators scoring production traces in near-real-time; agentic framework auto-instrumentation for LangChain, LangGraph, CrewAI, OpenAI Agents SDK, Google ADK, Google GenAI, Bedrock, Bedrock Agents, Anthropic, Vertex AI, LiteLLM, and others at various GA and Preview states; AI Agents Console for third-party agent inventory and governance at Preview; LLM Experiments for production-grounded quality comparison at Preview; Bits AI SRE for autonomous incident investigation at GA; Bits AI Dev Agent for code-fix generation at partial GA; Bits AI Security Analyst for SIEM triage at GA; Watchdog for ML-based anomaly detection across the full stack at GA; and a research capability in Toto and BOOM that signals the platform's next-generation anomaly detection direction.

What it stops short of at Q2 2026: prompt management lifecycle with version control, staged rollout, and approval workflows at the depth that specialized tools provide; deep evaluation pipelines with fully customizable rubrics and human annotation workflows for teams with domain-specific quality requirements; ground-truth comparison datasets and systematic offline evaluation workflows for teams running safety-critical applications; governance controls for third-party agents operating in the environment with explicit scope-limiting authority boundaries; a Terraform-configurable auto-investigation enablement mechanism; and AI-specific Watchdog tuning guidance for the non-stationary distribution properties of AI workload metrics.

The boundary between what the surface covers and what it does not is not static. Every product in the third wave has a Preview-to-GA trajectory. The AI Agents Console, LLM Experiments, and the Bits AI agent handoff workflows are each at different points in that trajectory as of Q2 2026. The coverage gaps visible today reflect the maturity curve of a platform that moved from LLM Observability preview in August 2023 to a full autonomous-operations layer in less than three years, not the limits of the platform's intended scope.

The surface is large. The edges are real. The edges are also moving.

### Key Takeaways

The Datadog AI Surface at Q2 2026 spans three successive waves built in less than three years: LLM Observability (GA June 2024), the agentic platform layer including AI Agent Monitoring (GA DASH 2025) and AI Agents Console (Preview DASH 2025), and the autonomous-operations layer anchored by Bits AI SRE (GA December 2025). The platform's structural advantage is cross-signal correlation: LLM spans share a trace ID with the surrounding APM traffic, which shares a host with the infrastructure metrics, producing joins at the collection layer that no specialized LLM observability tool can replicate without explicit OTel propagation configuration.

- The **Span Taxonomy** organizes LLM Observability traces into seven typed span kinds (llm, tool, retrieval, embedding, workflow, agent, task), each mapping to a distinct layer of the AI Application Surface and enabling correlation with standard APM spans rather than requiring a separate trace store.
- The **1 MB Payload Ceiling** is a structural constraint, not a configuration option: spans exceeding one megabyte are truncated at ingestion, creating a tension between capturing full conversational context for debugging and staying within the trace storage budget for long-context or multi-turn applications.
- The Bits AI SRE GA marks the transition from passive observability to active autonomous investigation: the agent monitors the full signal stream continuously, initiates investigations before an on-call engineer has opened the alert, and can propose remediation PRs, but cannot merge without human approval.
- The evaluation surface at Q2 2026 includes out-of-the-box hallucination, prompt-injection, and topic-relevance evaluators alongside LLM Experiments for offline evaluation; it does not include pre-deployment blocking gates, fully customizable human annotation workflows, or prompt management lifecycle tooling at the depth specialized tools provide.
- Datadog company history (founding 2010 by Olivier Pomel and Alexis Lê-Quôc, IPO 2019) is context for the platform's infrastructure-first orientation, which is the architectural reason cross-signal correlation works by default: LLM spans are first-class citizens in the same trace store as the APM and infrastructure telemetry, not a separate product with a separate data model.
- AI-native customers represented approximately 12 percent of Datadog's revenue by Q2 2026, with 19 customers above $1 million ARR in the AI-native segment, indicating the surface is mature enough for enterprise production deployment while remaining in active development across its second and third waves.
