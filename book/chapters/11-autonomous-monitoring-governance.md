# Chapter 11: Autonomous Monitoring and the Governance Problem

*In this chapter, the book examines what happens when the system that detects production anomalies is itself an AI system whose behavior requires monitoring. Bits AI SRE, Toto, and Watchdog represent a trajectory from threshold alerting to foundation-model-based detection. The performance claims are documented: 70 percent MTTR reduction at iFood, root-cause isolation in under four minutes at Energisa. The governance infrastructure that would let a team reason about when to trust the autonomous agent, and what to do when it is wrong, does not yet match the product investment. That gap is the subject.*

On December 2, 2025, Datadog made Bits AI SRE generally available.<sup>[1]</sup> The launch described the agent as "always-on" in a way that carried a specific technical meaning: the agent does not wait to be invoked. It monitors the full Datadog signal stream continuously, and when a monitor fires, it begins its investigation before any human on-call engineer has opened the alert.

The launch documentation described the workflow plainly. Bits AI SRE receives an alert, traverses the platform's metrics, traces, and logs in parallel, generates hypotheses about the root cause, tests those hypotheses against available evidence, and produces a structured incident summary with a confidence-ranked root-cause assessment. At iFood, the agent delivered a 70 percent reduction in Mean Time to Resolution following activation. At Energisa, it isolated root causes in under four minutes.<sup>[2]</sup> Thomson Reuters and Fanatics used it to maintain triage efficiency during high-traffic events when alert volumes would otherwise overwhelm on-call rotations.

What the launch documentation did not describe in detail was what happens when Bits AI SRE is wrong. The agent generates code patches. It populates Jira tickets. It integrates with GitHub to propose remediation pull requests. The Bits AI Dev Agent component opens actual GitHub PRs in sandboxed environments; the documentation is explicit that the Dev Agent never auto-merges, and human review is required before any code reaches production.<sup>[3]</sup> But the trust-verification step, the moment at which the human on-call engineer reads the agent's root-cause assessment and decides whether to act on it, is not itself observable. The confidence score the agent assigns to its own conclusion is not a Datadog trace. The human's decision to approve or reject the agent's recommendation is not a span. Whether the agent's diagnosis was correct lives in an incident ticket and a Slack thread, not in the same telemetry system that detected the original anomaly.

That asymmetry is the subject of this chapter.

## The Condition This Chapter Names

What makes the Bits AI SRE launch relevant is not the capability it demonstrated. The capability was real: reducing MTTR by 70 percent at a production logistics platform is an outcome any on-call team would value, and the root-cause identification speed at Energisa represents a qualitative shift in how quickly an incident can be characterized. What the launch revealed, structurally, is the endpoint of the AI-in-observability trajectory. The system doing anomaly detection and remediation is itself an AI system, and that AI system's behavior requires monitoring. The observer is now the observed. The monitoring platform must monitor its own monitor.

**Autonomous Monitoring Governance.** The set of formal constraints, scope-of-authority boundaries, trust-verification steps, and audit mechanisms required before an autonomous AI agent is permitted to take remediation actions in production on behalf of a human on-call engineer. The governance problem is not whether autonomous agents can improve incident response. The documented evidence says they can. The problem is whether the trust-verification step and the scope-of-authority boundary have been formally defined, and whether the observability infrastructure was designed to observe that class of event with the same rigor it applies to any other production service.

## From Threshold Alerting to Foundation Models

To understand where autonomous monitoring governance sits in production observability, it helps to trace the path from which it arrived. Datadog's Watchdog service makes that path visible in a single product line.

Watchdog began as a statistical anomaly detection layer built on top of the metrics, traces, and logs Datadog collected. The fundamental operation was comparison: a metric time series for a service was compared against a learned baseline, and significant deviations triggered a Watchdog alert. The baseline was not hand-tuned. Watchdog computed it from the service's own history, adjusting for weekly seasonality, deployment events, and traffic volume changes. This was threshold-based alerting generalized: instead of "alert when error rate exceeds 5 percent," the rule was "alert when error rate deviates from its expected distribution by more than a statistically significant amount." The approach removed the need for teams to set and maintain static thresholds across a growing service topology.

Over time, Watchdog expanded beyond metric anomalies to correlate anomalies across signal types. A latency spike in a service, accompanied by a deployment event in the same service and an error-rate anomaly in a downstream dependency, is more likely to represent a real incident than any of the three signals in isolation. **Watchdog Insights** developed the capacity to surface correlated anomaly clusters, ranked by likely operational significance. The ranking was statistical: it estimated the probability that a given cluster represented a genuine degradation rather than coincidental variation. The engineering team did not need to configure the correlation logic. Watchdog assembled it from the relationships already implicit in shared tags and trace IDs.

The **Watchdog Explains** feature extended this further: when Watchdog identified an anomaly, it provided a natural-language explanation of the likely cause, drawn from correlated evidence. An engineer reading a Watchdog alert would see not only the anomalous metric but a structured account of the deployment, the upstream dependency, the geographic region, or the customer segment that Watchdog assessed as the most likely contributing factor.<sup>[4]</sup> The faulty deployment detection capability, which compares the performance of new code versions against previous versions and surfaces degradations within minutes of a deployment event, extended this interpretability into the deployment lifecycle.<sup>[5]</sup>

Each of these extensions moved Watchdog in the same direction: from detection to characterization, from observation to interpretation. The anomaly detection engine was acquiring the capacity to reason about what it saw. Not only to report it.

Toto and BOOM represent the next step in that trajectory, and they originate from Datadog's AI Research division rather than from the product organization. The provenance matters.

## Toto: Architecture and What It Changes

**Toto** is a time series forecasting foundation model with 151 million parameters, developed by the Datadog AI Research team (led by Ben Cohen and Emaad Khwaja among others) and open-sourced on May 21, 2025, under the Apache 2.0 license.<sup>[6]</sup> The abbreviation stands for Time Series Optimized Transformer for Observability. The model uses a decoder-only transformer architecture with several novel components designed for the specific properties of production observability time series.

The first is **patch-based causal instance normalization**. Observability time series are highly nonstationary: a CPU utilization series during a traffic spike looks nothing like the same series during off-peak hours. A traditional instance normalization approach computing statistics from the entire series would leak information about future values into the normalization step, violating causality. Toto normalizes each patch (a contiguous window of 64 time steps) using only the statistics computed from that patch and all preceding patches. The causal constraint preserves the model's ability to forecast the next patch without having seen any of it.<sup>[7]</sup>

The second is **proportional factorized attention**. Observability metrics are typically high-cardinality multivariate series: a single service health assessment might involve dozens of correlated metrics simultaneously. Full cross-attention between all variates at all time steps is computationally prohibitive at production monitoring scale. Toto uses a factorized attention mechanism that alternates between time-wise and variate-wise attention blocks in an 11:1 ratio, processing temporal relationships and cross-variate relationships in separate passes rather than jointly.<sup>[8]</sup>

The third is a **Student-T mixture model output head**. Observability time series exhibit heavy-tailed, skewed, and often multimodal distributions: request latency has a long right tail; error rate has extreme values during brief outages interleaved with near-zero values during normal operation. Standard Gaussian output assumptions produce poor calibration. Toto uses a mixture of K Student-T distributions to model each time step's predictive distribution, capturing both central tendency and extreme tail behavior.<sup>[9]</sup>

What distinguishes Toto from prior foundation models is its training corpus. Toto was trained on approximately 2.36 trillion time series points, of which 1.59 trillion are non-repeated and non-synthetic. Forty-three percent of the total training corpus consists of anonymized observability metrics from the Datadog platform itself: real production metrics stripped of customer-identifying metadata. This is 4 to 10 times larger than the training corpora of other leading time series foundation models such as TimesFM, Time-MoE, Moirai, and Chronos.<sup>[10]</sup> The depth of that training corpus in observability-specific data gives Toto a distributional knowledge of what production infrastructure metrics look like that general-purpose forecasting models cannot match from their training on financial, IoT, or retail datasets.

The consequence is that Toto can be applied to observability time series in zero-shot mode, without fine-tuning on the target dataset. A new service, a newly deployed feature, or a newly instrumented endpoint can be assessed against Toto's pre-trained priors from its first data point. Zero-shot anomaly detection with a foundation model removes a constraint that has historically limited ML-based anomaly detection at scale: the requirement to collect sufficient history from each individual metric series before the model can reason about it.

**BOOM** (Benchmark of Observability Metrics) was open-sourced alongside Toto on the same date. BOOM consists of 350 million observations across 2,807 distinct multivariate time series, drawn from Datadog's staging environment to prevent contamination with Toto's training data. The evaluation data originates from a separate staging environment, isolated at the platform level from the production data used for training.<sup>[11]</sup>

The dataset spans four domain categories: Application Usage (request rates, API call patterns, user activity counts), Infrastructure (CPU utilization, memory consumption, Kubernetes cluster metrics), Database (query latency, cache efficiency, command execution counts), and Networking (throughput, authentication traffic, intrusion detection signals). The median is 60 variates per series, compared to one variate per series in GIFT-Eval and seven in the older Long Sequence Forecasting benchmark. That high-cardinality multivariate structure makes BOOM substantially more representative of the data production monitoring systems actually encounter.<sup>[12]</sup>

The benchmark results are concrete. On BOOM, Toto achieves a 12 percent improvement in Continuous Ranked Probability Score over the next-best method, Moirai (mean CRPS of 0.375 versus 0.447). On GIFT-Eval, Toto holds the top position among all reported models with an average ranking score of 5.495 as of May 2025. On the Long Sequence Forecasting benchmark, Toto attains best performance on 8 of 12 reported metrics in zero-shot evaluation.<sup>[13]</sup>

The relationship between Toto and Watchdog at Q2 2026 is research proximity, not specified product integration (per Datadog documentation, accessed Q2 2026). Datadog has stated that Toto and BOOM are being integrated into the platform without specifying which production features they already power. The open-weights release (model weights at huggingface.co/Datadog/Toto-Open-Base-1.0 under Apache 2.0) positions Toto as both a research contribution and a foundation for future product development.<sup>[14]</sup>

The trajectory is legible: Watchdog began with statistical baselines, developed through ML-based anomaly correlation, acquired natural-language explanation capabilities, and is moving toward foundation-model-based time series understanding. Toto is the research artifact that marks the far end of that trajectory. It matters because it describes the path from a monitor that flags anomalies to a monitor that understands them. Autonomous monitoring governance becomes more consequential, not less, as the underlying detection layer grows more capable.

## What Bits AI SRE Actually Does

The Bits AI SRE architecture, as documented at Q2 2026, is worth describing with structural precision. The agent operates continuously within the Datadog platform's event stream. When a monitor fires, the agent receives the alert and begins its investigation workflow immediately.

The investigation proceeds in layers. The first layer is signal assembly: the agent pulls the full set of metrics, traces, and logs correlated with the alerting service over a configurable lookback window, typically the past hour and the past day. The correlation is structural, not assembled through a query the team must write. Because Datadog's telemetry shares tag keys across signal types (specifically the service, env, version, and team tags applied through **Unified Service Tagging**), the agent can construct a cross-signal view of the service's behavior at the moment of the alert without pre-configured join configuration. The second layer is hypothesis generation: the agent proposes candidate root causes, ordered by confidence, drawing on assembled evidence and patterns learned from thousands of historical incidents. The third layer is hypothesis testing: the agent checks each hypothesis against available evidence, attempting to confirm or disqualify it through the data. A hypothesis that a latency spike was caused by a downstream dependency timeout is tested against the trace data for that dependency. A hypothesis that an error rate increase was caused by a deployment 20 minutes ago is tested against the deployment event timeline.<sup>[15]</sup>

The output is a structured incident summary with a natural-language root-cause assessment, a confidence score, a list of affected services and their contributing indicators, and a proposed remediation action. The remediation may be a runbook link, a Slack message to the relevant team, a Jira ticket created automatically, or (where the Bits AI Dev Agent is enabled and the service has a GitHub repository mapped) a draft pull request with a code-level fix.

The MTTR claims are the primary evidence base for operational value. The 70 percent MTTR reduction at iFood and root-cause identification in under four minutes at Energisa are vendor-controlled case studies; the corpus does not include independent replications.<sup>[16]</sup> The directional evidence is real. The magnitudes carry vendor-controlled sourcing.

***The performance claims are documented; they have not been independently replicated, and the failure mode taxonomy for the agent has not been published.***

## RBAC, Retention, and the Configuration Gap

Bits AI SRE requires specific RBAC configuration before it can take actions beyond passive investigation. Enabling the agent requires setting `bits_*` permissions in the Datadog RBAC model, connecting a Slack workspace via the `/dd connect` command, and enabling the Incident AI features toggle. The auto-investigation capability must be enabled per-monitor through the Datadog UI; as of May 2026, this configuration is not available through Terraform or any infrastructure-as-code mechanism (per Datadog documentation, accessed Q2 2026).<sup>[17]</sup>

That constraint is not incidental. It means the scope of the agent's automated authority is set by UI actions taken by a human administrator, not by version-controlled configuration auditable in the same way as application deployment infrastructure. A concrete artifact makes this visible:

| Configuration Element | IaC Available (Q2 2026) | Consequence |
| --- | --- | --- |
| RBAC `bits_*` permissions | Yes (Terraform) | Auditable, version-controlled |
| Slack workspace connection | No (UI `/dd connect` only) | Not version-controlled |
| Per-monitor auto-investigation | No (UI toggle only) | Not version-controlled |
| Dev Agent repository mapping | No (UI only) | Not version-controlled |

Three of the four configuration elements that define the agent's scope of authority exist only as UI state. No infrastructure-as-code drift detection applies.

The **zero data retention** model constrains the agent's ability to learn from its own history. Bits AI SRE enforces zero retention with external LLM providers: the reasoning the agent performs does not persist outside Datadog's infrastructure (per Datadog documentation, accessed Q2 2026). This design protects customer data from exposure to external model training. The operational consequence is that the agent cannot accumulate knowledge of a specific customer's infrastructure idiosyncrasies across investigations. If a service behaves in a way that is unusual by industry-wide standards but normal for that specific customer, the agent will not learn that distinction over time. It will re-examine the evidence each time, potentially reaching the same incorrect conclusion repeatedly until a human adjusts the alert configuration or annotates the relevant runbook.<sup>[18]</sup>

Bits AI Security Analyst, generally available as of April 2026 (per the April 29, 2026 release announcement), extends the autonomous investigation model to the security domain.<sup>[19]</sup> The Security Analyst integrates into Datadog's Cloud SIEM pipeline and autonomously investigates security alerts using the same layered evidence-assembly approach, with the addition of SOC investigative playbook logic. The documentation states SIEM triage time reduction of up to 98 percent (per Datadog documentation, accessed Q2 2026).<sup>[20]</sup> The governance questions are structurally identical to those for Bits AI SRE, with an additional layer of sensitivity because security investigation often involves logs containing personally identifiable information, authentication tokens, and other regulated data categories.

The configuration relationship between the two agents is worth describing explicitly. A production environment that enables both against the same telemetry substrate has two autonomous reasoning systems simultaneously traversing the same metrics, traces, and logs, constructing hypotheses, and proposing actions. The operational alert from a failing service might trigger Bits AI SRE's incident investigation. If that failure has characteristics matching a security detection rule, the same event might trigger the Security Analyst's SIEM investigation. The two investigations are independent; neither agent is aware of the other's in-progress analysis. The on-call engineer may receive an incident summary and a SIEM alert for the same underlying event, each framed differently, each carrying a different confidence score and a different proposed action. The coordination of those two outputs is the engineer's responsibility, not the platform's.

That coordination gap, the absence of a meta-layer that reasons across the outputs of multiple autonomous agents operating on the same telemetry, is the next structural problem in the trajectory.

## The Reasoning Trace Gap

The governance gap that autonomous monitoring introduces has a specific shape. It is not primarily a safety problem in the sense that the agent will take obviously harmful actions: the guardrails on direct production changes are explicit, and the human-review requirement for code merges is real. The problem is subtler. It concerns the relationship between the agent's authority model and the observability model that surrounds it.

The shadow agent problem names this gap precisely. An autonomous remediation agent's actions (a Jira ticket, a Slack message, a draft GitHub PR) generate records. The agent does not act invisibly. The shadow is that the reasoning chain connecting the anomaly to the proposed fix is not captured in the telemetry substrate that the rest of the production system uses to justify and audit decisions. The action is traceable. The reasoning behind the action is not.

Consider what a Datadog trace of a Bits AI SRE investigation looks like at Q2 2026. The original anomaly is a monitor event: a structured datum in the Datadog event stream with a timestamp, a service tag, and a trigger condition. The investigation is a series of API calls to the Datadog query layer, retrieving metrics, trace summaries, and log aggregations. The agent's output arrives as a structured message in Slack and as fields in a Datadog Incident Management record. The draft GitHub PR, if one was generated, arrives in the repository with a commit message and a description.

None of this is missing from the audit trail in the sense that the information does not exist. What is missing is the connection. The Incident Management record does not carry a trace ID linking it back to the original monitor event. The Slack message does not carry a span ID linking it to the API calls the agent made. The draft GitHub PR does not carry metadata linking it to the Datadog incident in a form the trace backend can query. The end-to-end investigation, from anomaly detection to remediation proposal, cannot be reconstructed as a single trace in the same tool that would investigate any other production incident.<sup>[21]</sup> Reconstructing it requires consulting the Incident Management record, the Slack thread, and the GitHub PR history, assembling the timeline manually from records whose timestamps are in different systems and whose identifiers do not cross-reference each other.

An on-call engineer whose remediation action was wrong would be visible in the incident timeline: Slack messages, commands run, PRs opened, rollbacks triggered, all attributable to a named human actor whose reasoning could be reconstructed from the record. A Bits AI SRE remediation action that was wrong is visible in the same channels, but the agent's reasoning exists only in the natural-language summary it produced. If the summary was incorrect or the reasoning was flawed in a way the summary did not expose, reconstructing the agent's decision path requires examining intermediate outputs not currently structured in a queryable form.

The PocketOS database destruction incident of April 2026 is instructive as shorthand: an autonomous agent with overly broad authorization executed an irreversible action the operator did not intend, because the scope of delegated authority was misconfigured. Bits AI SRE operates under narrower authority by design, but the underlying governance problem is structurally identical: an agent acting under delegated authority makes decisions whose rationale is difficult to inspect and whose scope, if misconfigured, is difficult to bound after the fact.

## The Recursive Monitoring Problem

Autonomous monitoring governance generates a specific observability question that has no clean answer at Q2 2026: how does a team monitor Bits AI SRE?

The question sounds self-referential to the point of paradox, but it resolves cleanly when framed operationally. What a team running Bits AI SRE needs to monitor is a specific set of behavioral properties: the agent's investigation success rate, the accuracy of its root-cause assessments, the rate at which its remediation proposals are accepted versus rejected by on-call engineers, the latency of its investigation workflow, and the cases where the agent produced a confident but incorrect assessment.

These are evaluation-layer metrics in the sense that **the Eval/Ops Split** (the principled separation between production health questions and model quality questions, established in Chapter 5) defines that boundary. They are not operational health metrics of the agent as a running service; they are quality metrics of the agent as a reasoning system. Datadog's LLM Observability product, applied to the agent's own LLM calls, would capture the token costs of Bits AI SRE's investigations, the latency of the LLM calls the agent makes during reasoning, and the error rates on those calls. It would not capture the accuracy of the conclusions the agent reached.

Accuracy requires a ground-truth comparison: what the agent concluded versus what the actual root cause turned out to be, as determined by the incident postmortem. That comparison is an evaluation-layer operation. The postmortem establishing ground truth exists only after the incident is resolved, and in most production environments it is written as a document rather than structured in a form that any automated system reads back against the agent's conclusions. The tooling loop is not closed.

Consider what closing that loop would require. The postmortem would need a structured field for actual root cause using the same vocabulary as Bits AI SRE's hypothesis output. The incident record would need to link the postmortem's conclusion back to the specific investigation. The agent's investigation output would need to persist in a queryable form after the incident closes, despite the zero data retention model. A system comparing postmortem conclusions against agent hypotheses would need to consume both streams and produce calibration metrics over time. None of these components exist as standard products at Q2 2026.<sup>[22]</sup>

The practical consequence is that production teams running Bits AI SRE are operating on instruments whose calibration has not yet been established through sufficient published operational history. The agent's outputs are real; its MTTR improvements at iFood are documented; its root-cause identification speed at Energisa is measured. What is not yet documented in published form is the failure mode taxonomy: the characterization of when and under what conditions the agent's reasoning becomes unreliable. Datadog's training corpus for the agent spans thousands of customer environments in pre-GA testing.<sup>[23]</sup> The systematic characterization of error modes has not been published.

This is not a criticism specific to Datadog. Every first-generation autonomous incident remediation agent faces the same problem. The calibration data establishing a reliability boundary requires production deployment at scale over time. That data accumulates through exactly the operations teams are running now. The teams running Bits AI SRE in 2025 and 2026 are generating the calibration corpus that would, in principle, enable a future version to characterize its own failure modes. It is a maturity boundary that any technology reaches only through deployment at scale.

## Scope of Authority

The shadow agent problem resolves to a specific governance question: how much authority does the agent have, over what resources, and how is that authority bounded and audited?

The Bits AI SRE answer at Q2 2026 is partially specified. The agent can read from the full Datadog telemetry substrate, with access governed by RBAC permissions granted at configuration time. It can write to Slack, Jira, and Incident Management records. Through the Dev Agent component, it can propose code changes in sandboxed GitHub environments, but it cannot merge those changes without human approval. It cannot execute arbitrary shell commands. It cannot modify production infrastructure directly. The kill-switch design, where authority is bounded by explicitly permitted output channels and remediation actions require human approval for code changes, is a reasonable first-generation governance model.<sup>[24]</sup>

What the current model does not provide is a mechanism for reviewing the agent's authority boundary as the services it monitors change scope. The RBAC configuration governing read access to Datadog telemetry is set at deployment time. If a service that was low-risk when the agent was configured becomes high-risk because of a change in the data it processes, the agent's read access to that service's LLM traces (which may now contain personally identifiable content or commercially sensitive material) does not automatically narrow. The scope-of-authority problem is, in part, a configuration drift problem: the agent's authorization was scoped at a point in time, but the observability surface it reads from evolves continuously.

The EU AI Act provides a relevant frame in shorthand. An autonomous incident remediation agent that can propose production code changes operates in territory the Act's high-risk classification criteria are designed to address. The documentation requirements, including logging of the agent's reasoning outputs, specification of human oversight mechanisms, and evidence of accuracy claims, are substantially more demanding than what a standard alerting rule requires. The gap is precise: an alerting rule has a documented trigger condition readable from a configuration file; an autonomous reasoning agent's decision to open a remediation PR has no equivalently readable trigger condition, because the trigger was the agent's assessment of assembled evidence, expressed in natural language, scored with a confidence value, generated by an LLM call whose prompt and completion are subject to the 1 MB per-span payload limit and whose full reasoning chain may not be stored after the investigation closes.

The Datadog MCP Server creates an adjacent governance question. When an engineer uses Claude Code with the MCP Server to investigate a production issue, the queries Claude Code issues against the Datadog platform are structurally analogous to those Bits AI SRE issues during an investigation.<sup>[25]</sup> Both represent an AI reasoning engine with access to live production telemetry, proposing actions based on assembled evidence. The difference is that the Claude Code session is initiated and supervised by a human in real time, while Bits AI SRE is initiated automatically by a monitor firing. The same architectural pattern carries different governance requirements depending on whether a human is present at initiation.

## Interpretability at Scale

The parallel trajectory of Watchdog and Toto illuminates both what is coming and why governance challenges will deepen rather than narrow as the underlying technology matures.

Watchdog, as it existed at Q2 2026, is a statistical pattern-matching system operating on features extracted from pre-defined metric types. An alert from Watchdog can be traced back, in principle, to the statistical baseline comparison and the deviation threshold that triggered it. The interpretation is available to an engineer who understands the detection methodology. A Watchdog alert for a latency anomaly is grounded in a comparison between a current percentile value and a learned seasonal baseline. That comparison can be expressed in terms an engineer can verify.

Toto operates differently. As a foundation model trained on 2.36 trillion time series points, it develops internal representations of what production time series look like and how they relate. Its forecasts are probabilistic: for a given input context, it predicts a distribution of likely future values expressed as a mixture of Student-T distributions with learned mixture weights, means, and degrees of freedom. When Toto flags an anomaly, the basis is the discrepancy between its probabilistic forecast and the observed value, measured by Continuous Ranked Probability Score. CRPS is a proper scoring rule that penalizes both miscalibration and sharpness, but it is not an explanation in the same sense that "the P99 latency deviated from its two-week seasonal baseline by 3.2 standard deviations" is an explanation.<sup>[26]</sup> The foundation model's output is a probability distribution. The anomaly detection decision requires a threshold on a summary statistic of that distribution, and the threshold's relationship to the model's internal representations is not directly readable by an engineer without access to the model's weights and architecture.

This is not a limitation unique to Toto. It is a property of all sufficiently complex learned models. The trade-off between model performance and model interpretability is well-established, and it applies to observability models the same way it applies to any other domain. Toto achieves better zero-shot anomaly detection than Watchdog's statistical baselines precisely because it has learned representations more complex than those a seasonal decomposition can express. That complexity is the source of the performance gain and the source of the interpretability cost.

The governance implication is direct: as the detection engine underlying Bits AI SRE moves from Watchdog's statistical comparisons toward Toto's foundation model representations, the character of the explanation changes. An alert grounded in a statistical anomaly becomes a probabilistic signal grounded in Toto's learned prior over observability time series. The on-call engineer reading Bits AI SRE's summary in 2027 or 2028, if the trajectory continues, will see a root-cause assessment that is better calibrated and more accurate than the one available in 2025, but also more difficult to independently verify. The trust-verification step becomes both more consequential and harder to perform with confidence.

***This is not a reason to halt the trajectory; the performance gains are documented. It is a reason to invest in the evaluation infrastructure, calibration datasets, and governance frameworks that would allow teams to reason about when to trust the loop and when to interrupt it.***

## Operating the Loop in Practice

For production teams deploying Bits AI SRE at Q2 2026, autonomous monitoring governance is an operational posture that must be defined before the agent goes live. Three governance boundaries determine its shape.

The first boundary is which monitors to enable for auto-investigation. Enabling auto-investigation on every supported monitor maximizes coverage but also maximizes the volume of AI-generated incident summaries that on-call engineers must evaluate. A team that enables auto-investigation on 200 monitors and experiences 40 alert firings per week will generate 40 Bits AI SRE summaries per week. If the on-call engineer reads each summary in three minutes and acts on the agent's recommendation without independent verification, the engineer is effectively delegating the diagnostic decision 40 times per week. Whether that delegation is appropriate depends on the agent's reliability for each class of alert, which returns to the calibration problem: the team cannot know reliability per alert class until they have accumulated enough investigations to establish an empirical baseline.

The second boundary is which services to grant Dev Agent access to. The Dev Agent's ability to propose code changes is the most consequential capability autonomous monitoring governance must address. Granting access to all services in a GitHub organization maximizes remediation capability. Limiting access to a specific set of low-risk services, whose code changes have bounded blast radius and whose repositories have active reviewers, creates a tighter scope of authority. The decision requires knowing which services are low-risk, which requires the same kind of service catalog discipline that **The AI Application Surface** (the observable component set introduced in Chapter 2) requires across the broader telemetry estate. The catalog must exist before the authority model can be designed.

The third boundary is the feedback loop: the mechanism by which the team learns whether the agent's assessments were correct. At Q2 2026, this feedback loop is informal in every production deployment. It exists in postmortem documents, in the tracking that on-call engineers maintain when they accept or reject recommendations, and in periodic review of agent-generated Jira tickets. None of this feedback is structured in a form Bits AI SRE can consume to improve its accuracy. The zero data retention model ensures this: the agent cannot learn from past mistakes because it does not retain context across investigations. Calibration happens entirely in the human layer above the agent.

Building a structured feedback loop requires three components that production teams must assemble from general-purpose tooling:

| Component | Purpose | Native to Datadog at Q2 2026 |
| --- | --- | --- |
| Structured verdict field | Records actual root cause in controlled vocabulary aligned with agent hypothesis vocabulary | No |
| Linkage record | Durable reference from postmortem back to specific investigation record | No |
| Aggregation layer | Computes accuracy over time per alert type and per service; surfaces calibration data | No |

Without the structured verdict field, comparing agent output to postmortem conclusions requires natural language parsing rather than a simple match. Without the linkage record, comparisons must be performed manually. Without the aggregation layer, calibration data does not exist as a metric the platform team can monitor. Teams that want closed-loop calibration must build these from Incident Management API exports, custom metrics, and postmortem tooling.<sup>[27]</sup> Most teams will defer this investment until the value of the closed loop is demonstrated by visible calibration failures.

The cardinality discipline established in earlier chapters applies to the agent's own telemetry. Bits AI SRE's investigations generate events: API calls, Incident Management records, Slack messages, GitHub PR metadata. A team that indexes every investigation event with full metadata, across services, alert types, investigation outcomes, and remediation proposals, is creating an observability substrate for the agent's behavior. **The Cardinality Cliff** applies to agent telemetry as surely as it applies to application telemetry. The discipline of deciding what to observe about the observer, and at what storage and query cost, is the same discipline that applies to any other production system.

## The Trust Boundary

Autonomous monitoring governance creates a trust boundary in production engineering that has not existed before: the boundary between what the agent is authorized to observe, what the team believes it is doing, and what it is actually doing.

The authorization layer is specified in RBAC configuration and the list of monitors enabled for auto-investigation. The belief layer is the on-call engineer's confidence in the agent's assessments, accumulated through experience over time. The gap between the two is the trust boundary. When the gap is small, the engineer reads the summary, agrees with the diagnosis, and acts with high confidence. When the gap is large, the engineer finds the summary plausible but uncertain and spends additional time independently verifying. The agent's MTTR benefit is proportional to how often the gap is small enough that the engineer can act without independent verification.

The trust boundary is not static. It evolves as the team accumulates experience across different incident classes, and as the agent's underlying reasoning logic evolves through Datadog's product updates. An update to Bits AI SRE that changes hypothesis generation or evidence weighting logic may shift the boundary without the team's immediate awareness. The team cannot run a diff between the current investigation logic and the logic in place during the last incident. They can observe the agent's outputs and infer from those outputs whether the logic has changed, but that inference requires sufficient investigation volume to be statistically meaningful. The trust boundary is calibrated by experience, and the experience accumulation rate depends on how frequently relevant incidents occur.

Pomel and Lê-Quôc, Datadog's co-founders, have described observability's core value as reducing the friction of agreeing on what a system is doing during an incident.<sup>[28]</sup> Bits AI SRE extends that mission: reducing the friction of the investigation itself, not just the data assembly that precedes it. The extension is real. The friction reduction at iFood and Energisa is documented. The new friction the extension introduces, the governance work required to bound, audit, and calibrate the agent's authority, is not yet fully characterized in production experience or product documentation.

Alexis Lê-Quôc, in the March 2026 fireside chat with Mike Yorwerth of Kraken Technologies, described the core principle of safe autonomous operation: *"In order to go fast you have to be a safety is paramount particularly when you go fast. If you can't assure some sense of safety then what's going to happen is you at some point going to crash."*<sup>[29]</sup> The sentence, verbatim from the transcript, is grammatically rough but operationally precise. Speed and safety are not opposites in autonomous agent deployment. They are co-requirements. The agent that investigates faster than a human must be bounded more carefully than a human, because speed makes the consequences of miscalibration arrive faster.

The engineering community that built distributed tracing to observe microservices, sampling strategies to observe distributed tracing, and anomaly detection to observe metric streams now faces the task of building governance frameworks to observe the anomaly detectors. That task does not have a product catalog to draw from yet. Datadog AI Research's publication of Toto and BOOM, and the Watchdog team's extension toward foundation model capabilities, signal that the technical substrate for more capable autonomous detection is being built. The governance layer that would tell a team when to trust that detection, and what to do when it is wrong, remains ahead of the product.

**The Span Topology Problem** applies directly to the autonomous agent's own telemetry: the trace for a Bits AI SRE investigation cannot be represented as a coherent tree because the investigation itself is an agentic workflow. The same non-determinism problem that distributed tracing faces in agentic workflows applies equally to the governance layer. A human on-call engineer, given the same alert and evidence twice, will reach the same conclusion with high probability because the engineer's reasoning is consistent and her knowledge of the specific system accumulates. An autonomous reasoning agent, given the same alert and evidence at two different times, may reach different conclusions if the underlying LLM's sampling temperature introduces stochastic variation, or if model weights have been updated between investigations. The non-determinism is bounded in practice by the structured output format and the hypothesis-testing layer, but it is not eliminated. A team that runs two investigations of identical alerts and compares the outputs to validate consistency is doing evaluation work that no current product automates.

The system that detects anomalies is now the thing that produces anomalies worth detecting.

## Key Points

- **Autonomous Monitoring Governance** must be defined before deploying a remediation agent; applying it retroactively after deployment is structurally more expensive.
- The reasoning trace gap: actions appear in audit logs but the diagnostic chain from alert to proposed fix is not captured in queryable telemetry.
- Three of four Bits AI SRE configuration elements exist only as UI state at Q2 2026, outside infrastructure-as-code drift detection.
- Zero data retention protects customer data from external model training but prevents the agent from accumulating environment-specific knowledge across investigations.
- Closing the calibration loop requires a structured verdict field, a linkage record, and an aggregation layer no current product provides natively.
- As the detection engine moves from statistical baselines to foundation models, alert quality improves but independent verifiability of each alert decreases.
