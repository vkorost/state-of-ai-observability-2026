## Chapter 2. The AI Application Stack as a New Observability Surface

The engineers at a mid-size financial services company shipped their first production RAG system in the autumn of 2024 and considered themselves well-prepared. They had instrumented the LLM calls. They had latency dashboards. They had error rate monitors tied to HTTP status codes. They had configured custom metrics on token counts. By every measure their monitoring team recognized, the system was observable.

What they had not built was any visibility into the retrieval layer, and by December 2024 that gap had begun producing consequences they could not explain.

Customer support queries that the system had handled accurately in October started returning confident but incorrect answers. The LLM spans looked healthy. Token counts were within the expected range. Error rates were flat. Latency was within the p95 envelope. The retrieval spans were not instrumented at all, which meant that when the quality regression surfaced, the investigation started in the wrong place and stayed there. The team spent three weeks tuning the model and adjusting the system prompt before someone suggested examining the vector database directly.

What they found was embedding drift. The third-party embedding API provider had silently updated the model behind the same endpoint in November 2024. The provider's release notes documented the change, but the team had not subscribed to that channel. The new model generated vectors in a coordinate space that was mathematically incompatible with the pre-computed index built using the previous version. Similarity scores were returning plausible-looking numbers, but those numbers no longer measured semantic proximity. Documents that ranked in the top three results were frequently irrelevant. The LLM processed them faithfully and produced confident, grounded-sounding, wrong answers.

The incident did not throw a single exception. No service entered an error state. The monitoring stack's operational health indicators stayed green throughout. The problem was invisible because the team had instrumented the layers they recognized and left completely uninstrumented the layer that failed (as documented in the Gemini report on architectural observability of vector databases and retrieval layers in production RAG pipelines, 2026).

The investigation that followed revealed something more significant than a misconfigured system. It revealed that the team's entire mental model of what was observable about their AI system was borrowed from a prior generation of technology. They had APM intuitions applied to an AI Application Surface that those intuitions could not fully address.

---

**The AI Application Surface.** The complete set of new components that generative AI systems introduce as observable entities, including LLM calls, RAG pipelines, vector stores, agentic orchestration loops, and multi-model chains, each emitting telemetry primitives that did not exist in the pre-AI APM taxonomy. The AI Application Surface is not an extension of the existing observability surface. It is a second surface, with different components, different failure modes, different cost drivers, and different signal types, which happens to share infrastructure with the systems that traditional APM already monitors.

---

What makes that financial services team's story relevant to this book is that they were not negligent. They were thorough, by the standards available to them. The **Observability Gap**, as established in Chapter 1, is partly a gap of tooling and partly a gap of vocabulary. Teams cannot alert on things they have not named. The AI Application Surface demands a new naming pass: a component-by-component accounting of what each layer of the generative AI stack emits, what each layer can fail at without producing an exception, and why the pre-AI telemetry taxonomy has no slot for those failure modes.

This chapter performs that mapping. It proceeds through the five principal layers of the AI Application Surface, then turns to the specific failure modes at the retrieval layer that the source corpus documents most thoroughly, then addresses where the industry's standardization effort stands as of Q2 2026 and what remains absent from the specification.

### The Five Layers of the AI Application Surface

A production generative AI system built to current architectural norms is not a single service that calls a model API and returns a response. It is a pipeline. In many production deployments, it is a pipeline of pipelines, where the output of one stage becomes the input configuration of the next. The canonical stack has five layers. Each layer has a distinct observability profile. Understanding those profiles requires tracing the data flow through the system, naming what can fail at each stage, and comparing those failure modes against what the pre-AI APM vocabulary can express.

**Layer One: The LLM Call.** An application sends a prompt to a foundation model provider, via a direct API call or a managed inference endpoint, and receives a completion. The components in this layer are the client application code, the network path to the provider, the provider's serving infrastructure, and the response buffer that streams tokens back to the client.

The telemetry primitives this layer introduces have been well-named by the OpenTelemetry GenAI semantic conventions. A client span records the provider identifier, the requested and actual model names, the operation type (chat, generate content, text completion), the temperature and other sampling parameters, the request and response token counts separated into input and output buckets, extended token counts for reasoning and cached tokens where supported, the finish reason, and the time to first token alongside time per output token as latency histograms (OpenTelemetry GenAI semantic conventions v1.37, Q2 2026). Seven standardized metrics cover token usage as a counter and operation duration as a histogram, with dimensions for provider, model name, and error type. The coverage here is, in the vocabulary of this chapter, settled territory.

What the LLM call span does not and cannot capture is the semantic quality of the completion. A finish reason of "stop" records that the model reached a natural stopping point. It does not record whether the answer was factually grounded. A p95 latency within the configured SLO records that the model responded within the expected time envelope. It does not record whether the response was coherent. Token count within a configured budget records that the model did not exhaust the context window. It does not record whether the response was relevant to the user's intent. These are evaluation signals, not operational signals. They require a different toolchain, a different question, and a different team. That distinction runs through the entire architecture of the AI Application Surface and receives its full treatment in Chapter 5.

**Layer Two: The Embedding Pipeline.** Many production AI applications convert text to dense vector representations before storing them in a retrieval index, and convert user queries to vectors before searching that index. The embedding pipeline is the component that performs these conversions, typically by calling an external embedding API or running a locally hosted embedding model.

The embedding operation has a standardized span structure in the OpenTelemetry GenAI conventions. Every call to an external service to translate text strings into vector coordinates uses the operation name "embeddings" and the naming convention `{gen_ai.operation.name} {gen_ai.request.model}`. The required attributes capture the exact model identifier requested by the application in `gen_ai.request.model`, the actual model that fulfilled the request on the provider side in `gen_ai.response.model`, the number of output dimensions in `gen_ai.embeddings.dimension.count`, and the encoding format of the returned arrays in `gen_ai.request.encoding_formats` (OpenTelemetry GenAI semantic conventions v1.37, Q2 2026). This is sufficient to record that an embedding operation happened, which model handled it, and what parameters governed the output.

What the embedding span does not record is the geometric identity of the vector space produced. Two embedding spans with identical `gen_ai.request.model` values are not necessarily operating in the same mathematical coordinate space. The attribute captures the name the application requested. It does not capture the model's actual weights, its version hash, or its output geometry. When an embedding API provider updates the weights behind a stable model name identifier, the span continues to record the same model name as before. The downstream retrieval system continues to execute similarity searches. The similarity scores continue to fall within a plausible numeric range. The semantic connection between the query vector and the index vectors, however, has broken.

This is the gap that the financial services team encountered in December 2024. It is not a gap that any operational span attribute can close, because the information that would close it, the version-level identity of the embedding model, exists at the provider's infrastructure boundary and is not surfaced through the standard API contract.

**Layer Three: The Vector Store and Retrieval Operation.** The vector store holds the pre-computed embeddings for the document corpus, organized into an index that supports approximate nearest-neighbor search. When a user query arrives, the application converts it to a query embedding, sends that embedding to the vector store, and receives back the k nearest neighbors, expressed as document identifiers with similarity scores. The retrieved document chunks are extracted from storage, assembled into a context payload, and injected into the prompt that will be sent to the LLM.

This layer is where the most concentrated set of AI-specific failure modes lives. None of them produce exceptions. All of them are operationally silent at the standard telemetry level.

The OpenTelemetry GenAI conventions have partially standardized the retrieval span. The working group did not define a standalone span type for retrieval; instead, retrieval operations are represented as client spans with `gen_ai.operation.name = "retrieval"`. The required and recommended attributes cover the provider identifier in `gen_ai.provider.name`, the target index or collection identifier in `gen_ai.data_source.id`, standard error type in `error.type`, and the server address and port (OpenTelemetry GenAI semantic conventions v1.37, Q2 2026). These attributes tell the observability platform that a retrieval happened, against which index, and whether it produced an error code.

They do not tell the observability platform how relevant the retrieved documents were to the query, how many documents were injected into the prompt context, what the actual similarity scores were for each retrieved chunk, or how old the documents in the index are relative to the upstream data source. Those four unrecorded properties are exactly the signals that distinguish a healthy retrieval layer from a degraded one.

**The Retrieval Layer.** The subsystem within a RAG pipeline that fetches document chunks from a vector store and injects them into the prompt context, introducing its own class of failure modes: embedding drift, index staleness, retrieval relevance degradation, context window poisoning, and query distribution shift. These failure modes share a structural property that makes them distinctively difficult to monitor: they do not produce exceptions. They produce silence in the operational error channel and confident-sounding wrong outputs in the generation channel.

The data flow through the retrieval layer, described precisely so the reader can reconstruct its topology without a diagram: the application layer generates a query embedding by calling the embedding API with the user's text. The query embedding, which is a dense floating-point vector of several hundred to several thousand dimensions, is sent as the primary input to the vector store's search interface. The vector store executes an approximate nearest-neighbor search across its index, computing similarity scores between the query vector and the pre-computed document vectors, and returns the top-k results as a ranked list of document identifiers paired with similarity scores. The application extracts the document text for each returned identifier from the underlying document store, assembles those chunks into a context block, and includes that context block in the prompt that is sent to the LLM. The LLM's response is grounded in whatever those retrieved chunks contained. If the chunks were relevant, the response tends to be accurate. If the chunks were not relevant, the response tends to be confidently wrong.

The vector store sits between the embedding pipeline and the LLM call. Its instrumentation requirements span both sides of that boundary: it needs to record the input query vector's provenance (which embedding model produced it, at what version), the output document set's properties (similarity scores, document age, document count), and the aggregate effect on the downstream LLM call (input token budget consumed by retrieved context). Traditional APM has no primitives for any of these.

**Layer Four: The Agentic Orchestration Loop.** The three layers described so far, LLM calls, embedding, and retrieval, form the telemetry surface of a passive pipeline: a system where each user request follows a fixed execution plan. Retrieve, embed, inject, generate, return. The plan may be complex, but it is predictable in structure.

An agentic system adds a fourth layer: a decision-making component that determines the execution plan at runtime. The orchestrator receives a goal from the user, decides which tools to invoke and in what order, processes the results of those tool invocations, decides on the next step, and continues until it judges the goal to be achieved or reaches a configured limit. The key distinction from a traditional pipeline is that the number of steps, the tools invoked, the order of invocation, and the total token consumption are all determined by the agent's reasoning at runtime, not by the application's code at design time.

The observability challenge this creates is fundamental. Traditional distributed tracing models execution as a tree: a root span representing the request, child spans representing downstream service calls, grandchild spans representing their downstream calls, and so on. The tree closes when the root span closes, which happens when the response is returned. The depth and branching of the tree are bounded by the application's code paths. An agentic loop violates all of these properties. The tree depth is unbounded in the general case. The number of children under any node is determined at runtime. The loop may revisit the same tool multiple times with different parameters. The execution may branch based on tool results that were unpredictable when the request arrived.

The OpenTelemetry GenAI conventions name operations for this layer: `invoke_agent` for the internal execution steps and decision loops within an active agent, and `invoke_workflow` as a grouping container span that wraps multiple interconnected agent components (OpenTelemetry GenAI semantic conventions v1.37, Q2 2026). The `execute_tool` operation name captures client-side or tool-side lifecycle boundaries where external systems fulfill the agent's calculated actions. These three operation types provide the vocabulary for basic agentic tracing.

But the vocabulary has gaps. The working group has identified, in meta-issue 2664, that autonomous, collaborative, and multi-agent environments require a broader taxonomy. The issue outlines six relational structures for agent interaction, ranging from single-agent tool-use loops to hierarchical multi-agent teams with supervisor agents coordinating specialist agents. Modeling these structures inside the mathematical constraints of distributed tracing, specifically the requirement that trace topology forms a tree or directed acyclic graph rather than a cycle, remains contested as of Q2 2026. The strategy planning gap is the most operationally significant: there is no standardized convention for the phase where an agent decides its execution sequence, the moment between receiving a goal and executing the first tool call, where the agent's reasoning determines everything that follows and where the highest-value diagnostic information about why the agent made the choices it made would live (Gemini OTel semantic conventions analysis, Q2 2026).

Cost attribution across an agent loop introduces a telemetry primitive with no equivalent in traditional APM. A web service call costs microseconds of compute. An agentic session that invokes an LLM forty times, each call consuming several thousand tokens, costs real money at each step and significant money in aggregate. The standard `gen_ai.client.token.usage` counter records token consumption per operation. Attributing cumulative cost across a multi-step loop with branching tool calls, where some branches are expensive and others cheap, and where the total cost depends on decisions made at runtime, is a cross-span aggregation problem that a single span or metric cannot represent. Industry ranges for the number of LLM spans generated per user interaction run from 3 to 10 for a simple RAG pipeline to hundreds or thousands for a complex multi-agent session (Claude Datadog in the generative AI era report, Q2 2026). A telemetry system that records each span in isolation but has no mechanism for presenting their aggregate cost as a single observable quantity has left the most operationally important number untracked.

**Layer Five: The Multi-Model Chain.** Some production systems route requests through multiple models in sequence. A small, fast classification model determines intent. A large reasoning model handles complex questions. A specialized code model generates implementation snippets. A reranker model scores retrieved document candidates before they are injected into the prompt. A moderation model checks the final response before it is returned to the user.

Each model call in this chain is a separate LLM span. The chain that connects them is the structure that multi-model observability must represent. For operational health, the chain requires tracking the handoff between models: when the output of model A becomes the input to model B, is the transformation at that handoff producing the right format, the right length, the right content type? A classification model that categorizes a request as "complex" routes it to the expensive reasoning model. If the classification is wrong, the routing is wrong, and the cost and latency implications of that misrouting compound across every affected request.

Traditional distributed tracing handles sequential service calls well. The parent-child span hierarchy represents the sequence accurately when each step calls the next and waits for a response. But the quality signals that matter for multi-model chains are relational: they require comparing the characteristics of what model A produced against the characteristics of what model B expected. Those relational signals live at the junction between spans, not inside any individual span. They require a cross-span analysis layer that APM was not built to provide.

### The New Telemetry Vocabulary

Naming the layers of the AI Application Surface is necessary but not sufficient. Each layer demands telemetry primitives that the pre-AI APM vocabulary cannot express. Enumerating those primitives concretely is the step from mapping to instrumentation.

**Token counts and cost per inference.** These are the most basic new primitives and the most well-standardized. Input and output token counts appear in every LLM span. The extended fields cover reasoning tokens, which some reasoning-optimized models consume internally before producing output and which are charged at the full output token rate, and cache tokens, which are charged at a discount when providers implement prompt caching. The financial implications are direct: token counts are the primary driver of per-call cost, and per-call cost is the primary driver of per-session cost, and per-session cost is the primary driver of AI infrastructure expense at scale. APM tracked CPU milliseconds and network bytes because those were the units that cost money. AI observability must track tokens for the same reason.

**Retrieval relevance scores.** The operational proxy for the quality of what the retrieval layer returned. A relevance score records how semantically close the retrieved documents are to the query, using cosine or dot-product similarity from the vector search plus, in some implementations, a secondary evaluation from a reranker model or an LLM-as-a-Judge. The OpenTelemetry retrieval span does not mandate this attribute. Arize Phoenix provides contextual relevance evaluators that run asynchronously against production spans, scoring each retrieval result against the query and writing the label back to the span as a persistent annotation (Gemini RAG observability report, 2026). Langfuse provides operation-level live evaluators that target the retriever observation type specifically, allowing engineers to run custom LLM-as-a-Judge scoring on isolated retrieval inputs and outputs without ingesting the full trace, on a configurable sampling rate such as 5 percent of production traffic (Gemini RAG observability report, 2026). Datadog provides built-in trace-level evaluators including a faithfulness template that validates whether LLM completions are grounded in the retrieved context (Datadog LLM Observability documentation, accessed Q2 2026). The primitive exists; the standardization does not.

**Embedding drift metrics.** The statistical signals that would have caught the financial services team's incident before it propagated to user-facing answers. Detecting drift requires computing rolling statistics over the distribution of embedding vectors generated at query time and comparing them against the distribution used to build the index. Three specific signals are load-bearing for production alerting: the class-based mean similarity between live probe queries and known-good document chunks within a targeted domain, the standard deviation of similarity scores across that probe set, and the noise floor maximum similarity computed by matching probe queries against completely unrelated corpus chunks. When the proximity gap between the class-based mean and the noise floor collapses, the index has lost its discriminative power (Gemini RAG observability report, 2026).

A production drift detection setup enforces a 14-day warmup gate at system initialization, during which the rolling baseline is established and false positives are suppressed. After warmup, a three-sigma deviation from the rolling proximity gap triggers an automated alert. Production platforms with continuous drift analysis capabilities, specifically the commercial Arize AX platform, ingest live traces on a 5-minute cadence and route alerts through PagerDuty, Slack, or OpsGenie when any vector class drifts past its historical boundary. Open-source implementations and standard APM platforms require custom background workers to compute these statistics, because neither the OpenTelemetry standard nor Datadog's LLM Observability product includes native continuous vector drift detection as of Q2 2026 (Gemini RAG observability report, 2026).

**Context window utilization.** The capacity signal that tracks how much of the LLM's input token budget is consumed by the retrieval layer specifically. A retriever configured with a generous `top_k` value and large document chunks injects a large token payload into the prompt, consuming budget that would otherwise be available for the system prompt, conversation history, and the user's current query. The observable signal is the correlation between input token count and document injection count across retrieval spans. Spikes in input token consumption without a corresponding increase in user query length indicate that the retriever is operating with excessive chunk sizes or an over-extended `top_k` setting (Gemini RAG observability report, 2026). Datadog surfaces this through its LLM Observability retrieval span, which records the document text via `.meta.output.documents[*].text` and enables correlation with the downstream LLM span's input token count (Datadog LLM Observability documentation, accessed Q2 2026).

**Hallucination indicators and groundedness scores.** The quality signals that are most contested in terms of where they belong in the observability taxonomy. These are not operational metrics in the traditional sense: they cannot be read off a health check, they require a secondary model to evaluate, and they answer questions about output quality rather than system health. The OpenTelemetry working group has proposals under active development for `gen_ai.response.quality` style metrics and a standardized `gen_ai.evaluation.result` event type, but the debate about whether these belong on operational spans, as separate events, or outside the tracing system entirely has not resolved as of Q2 2026 (Gemini OTel semantic conventions analysis, Q2 2026). Datadog provides out-of-the-box hallucination and prompt injection evaluators within LLM Observability as trace-level LLM-as-a-Judge pipelines (Datadog LLM Observability documentation, accessed Q2 2026). These are the most direct bridge the platform currently offers between operational telemetry and evaluation signal, and the question of where they sit in the architecture is the subject of Chapter 5.

**Session-level cost attribution.** The aggregate financial primitive that requires cross-span composition. A single session in an agentic system may span dozens of LLM calls, each with its own token counts and its own cost per call. Attributing the total cost of a session to the session's root span, and then attributing that session's cost to the user or feature or team that drove it, requires trace-level aggregation that standard APM metrics pipelines were not designed to perform. The cardinality implications of session-level cost attribution, specifically the fact that session IDs are high-cardinality identifiers that will rapidly exceed standard custom metric cardinality budgets if used naively as metric dimensions, are the subject of Chapter 3.

### The Standardization Landscape as of Q2 2026

The OpenTelemetry GenAI semantic conventions provide the most credible attempt to build a shared vocabulary across the AI Application Surface. Understanding what that vocabulary covers and what it leaves to vendor implementations or manual instrumentation is a practical requirement for any platform team deploying LLM-backed systems in production.

The settled territory covers a substantial fraction of the LLM call layer. Provider identifiers, model names, operation types, token counts, latency histograms, and finish reasons are all standardized and stable as of v1.37. The specification uses an explicit opt-in mechanism, the environment variable `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental`, to allow instrumentation libraries to emit the updated convention set while suppressing the pre-v1.36.0 layout, providing a controlled migration path for teams on older SDK versions (OpenTelemetry GenAI semantic conventions v1.37, Q2 2026). The covered providers include OpenAI, Anthropic, AWS Bedrock, Azure AI Inference, Azure AI OpenAI, Vertex AI, Gemini, DeepSeek, Perplexity, Cohere, and xAI, as well as any provider accessible via the OpenAI-compatible API surface.

The embedding layer is fully standardized for standard embedding operations: model name, dimension count, encoding format, and the required naming convention for embedding spans. This is the layer where the financial services team's incident originated, and the standard gives platforms everything they need to record that an embedding operation happened with a specific model. The gap, as described above, is that it cannot record the geometric identity of the space produced.

The retrieval span is partially standardized. The provider identifier, data source identifier, and error type are specified. The similarity scores, document injection count, and document freshness are not. The working group's attention on retrieval is oriented toward the infrastructure end: the ongoing proposals in issues 3602 and 3603 focus on adding `gen_ai.workflow.name` and `gen_ai.agent.name` as low-cardinality binding attributes across child retrieval, embedding, and inference spans, enabling observability backends to group operations by logical pipeline and agent role without traversing the full trace tree at query time (Gemini OTel semantic conventions analysis, Q2 2026). These are important infrastructure primitives. They do not address retrieval quality.

The agentic topology layer is the most actively contested area of the specification. The meta-issue 2664 taxonomy of six agent relational structures represents the problem statement, not the solution. Three sub-issues are in active development: the strategy planning gap (how to represent the agent's reasoning about execution sequence), the session-versus-conversation boundary (how to represent a user session that spans multiple multi-agent execution tracks), and the user perspective entry span (how to represent performance from the end user's point of view when the internal trace structure uses framework-internal operation names that do not map to user-visible concepts). Each of these has competing proposals and is unresolved as of Q2 2026 (Gemini OTel semantic conventions analysis, Q2 2026).

The evaluation layer, specifically hallucination indicators, answer correctness, and groundedness scoring, is at the proposal stage. A `gen_ai.evaluation.result` event type exists in experimental status. The broader debate about whether evaluation metrics belong on spans, as separate events, or in a distinct evaluation data model that sits alongside the trace but is not part of it remains open.

The payload handling debate deserves specific mention because it has direct consequences for every team building RAG or agentic systems. The specification currently isolates prompt and completion text into separate OpenTelemetry Event objects rather than storing them as span attributes. The rationale is that large text payloads should not inflate the primary tracing span with data that most observability backends are not optimized to store or query. Several major observability backend vendors have pushed back against this design in issue 2010, arguing that separating payloads from spans removes the temporal and spatial locality that makes trace-based debugging efficient: the engineer debugging a latency spike on a span should not have to cross-reference a separate event log to see what was in the prompt (Gemini OTel semantic conventions analysis, Q2 2026). This debate has not resolved, and in the meantime different vendors handle payload storage differently: Datadog's ddtrace SDK stores prompt and completion content directly on the span, within the 1 MB per-span payload limit that is the subject of its own architectural constraint treated in Chapter 7.

### The Fragmentation Problem in Vector Store Instrumentation

The section above describes what the OpenTelemetry standard specifies for the retrieval layer. The larger picture is that the vector store instrumentation landscape is fragmented across approaches that are not interoperable.

Datadog captures retrieval semantics through high-level orchestration interception. Its integrations for LangChain, LlamaIndex, and native provider SDKs intercept the application-level calls that drive the retrieval operation and extract the query text and retrieved document content at that boundary. The infrastructure layer, meaning the vector database's own performance metrics such as index query time, memory pressure, and node utilization, is captured through the standard Datadog Agent infrastructure monitoring. But the semantic content of the retrieval, the query text and the retrieved chunks, is captured at the application level, above the database. The vector database itself, whether Pinecone, Weaviate, Qdrant, pgvector, or Chroma, is treated as an opaque store rather than as a deeply instrumented component (Datadog LLM Observability documentation, accessed Q2 2026; Gemini RAG observability report, 2026).

Arize Phoenix takes a different approach. It instruments retrieval via the OpenInference specification, an open-source semantic convention layer built on top of base OpenTelemetry protocols. The `arize-otel` convenience package auto-instruments LlamaIndex, LangChain, DSPy, and the Vercel AI SDK, capturing a retrieval span with the query text in `retrieval.query.text` and the retrieved documents as a structured JSON array in `retrieval.documents`, including the document content and the similarity score for each result. This approach makes the similarity scores a first-class span attribute available for filtering and analysis (Gemini RAG observability report, 2026). The open-source Phoenix framework is a local visualizer and static evaluation library; continuous production drift analysis with rolling z-score alerting requires the commercial Arize AX platform.

Langfuse implements a first-class `retriever` observation type in its native schema. A retrieval operation recorded in Langfuse is not buried as a generic span but represented as a distinct observation with explicit input and output fields. Operation-level evaluators can be targeted specifically at retriever observations, running custom LLM-as-a-Judge scoring on isolated retrieval inputs and outputs without processing the entire trace (Gemini RAG observability report, 2026). This granularity allows teams to build evaluation pipelines that run on every retrieval operation, rather than on every complete user interaction, reducing the token cost and latency overhead of live evaluation.

The practical consequence of this fragmentation is that teams building production RAG systems face a choice that the standardization landscape does not make easy. Using the standard OpenTelemetry retrieval span gives portability but sacrifices retrieval quality signals. Using a specialized framework's instrumentation, whether OpenInference for Arize Phoenix or the Langfuse SDK, gives quality signals but introduces a non-standard observability layer that requires bridging back to the primary APM platform for operational correlation. No single tool gives both operational completeness and retrieval quality depth at Q2 2026.

### The Query Distribution Shift Problem

A failure mode that the standard telemetry vocabulary has no primitive for, and that the financial services incident did not encounter but that production RAG systems routinely do, is query distribution shift. The vector index is built on an assumption about what queries will look like. Chunk boundaries, embedding model selection, metadata schemas, and document selection criteria are all optimized for the query distribution present at build time.

Production query distributions change. Users change how they phrase requests as they learn the system's capabilities. Business contexts change as the product evolves. Seasonal patterns shift query volume and content. When the runtime query distribution diverges significantly from the distribution the index was built for, retrieval accuracy decays even though nothing in the system has changed. The vectors are correct. The index is fresh. The embedding model is consistent. The mismatch is between the system as built and the users as they actually behave.

The signal for query distribution shift is in the statistical properties of the incoming query vectors before they reach the index. A rising variance in the similarity scores of top-1 results, a drop in average top-1 matching confidence, and an increase in downstream user frustration signals such as immediate query reformulation and explicit negative feedback all indicate that the runtime query distribution has moved outside the bounds the index was designed for (Gemini RAG observability report, 2026). None of these signals appears in a standard operational span. Each requires computing statistical moments over a stream of incoming vectors, which is a monitoring discipline specific to the retrieval layer.

The detection approach requires continuous monitoring of the standard deviation and topological clustering of incoming query vectors as they arrive, before they reach the index. This is a proactive measurement, not a reactive one. It requires infrastructure that the standard APM telemetry pipeline was not designed to support.

### What the Coverage Gap Looks Like in Practice

The table below summarizes where standard observability coverage exists, where it is partial, and where it is absent across the five layers of the AI Application Surface. It organizes the coverage by observability dimension, making explicit the distinction between what the standard vocabulary can express and what it cannot.

| Layer | Operational Coverage | Quality Coverage |
|---|---|---|
| LLM call | Stable: token counts, latency, finish reason, error type | Absent from standard; vendor-specific evaluators exist |
| Embedding pipeline | Stable: model name, dimension count, latency | Absent: coordinate-space identity, version tracking |
| Vector store and retrieval | Partial: provider, data source, error type | Absent: similarity scores, document age, relevance |
| Agentic orchestration | Partial: agent and workflow operation names | Contested: planning gap, session boundary, loop depth |
| Multi-model chain | Basic: individual spans per model call | Absent: cross-model quality signals, misrouting signals |

The pattern in this table is not that AI systems are difficult to monitor. It is that the existing monitoring vocabulary was built for a specific class of system, the synchronous web service with bounded call graphs and exception-based failure modes, and a new class of system, the generative AI pipeline, has arrived with failure modes that are neither bounded nor exception-based.

### Sub-patterns in the Retrieval Layer

The **Retrieval Layer** as named in this chapter has several sub-patterns that production teams encounter as distinct operational problems. These sub-patterns are worth naming individually because each has a different monitoring strategy and a different remediation path.

**The Embedding Drift Problem.** The silent update of an embedding model's weights behind a stable API endpoint breaks the coordinate-space compatibility between a runtime query embedding and a pre-computed vector index. The standard span records the model name as a string. The name is not sufficient provenance. Two spans with identical `gen_ai.request.model` values may be operating in incompatible vector spaces.

Detecting this proactively requires monitoring three statistical signals: the class-based mean similarity between live probe queries and known-good document chunks in a targeted domain class; the standard deviation of similarity scores across that probe set, where increasing variance reveals that some vector groups have experienced spatial displacement; and the noise floor maximum similarity, the maximum similarity score when probe queries are matched against completely unrelated corpus chunks, where a rising noise floor signals that the coordinate space has compressed or rotated in a way that brings unrelated documents too close to the query boundary (Gemini RAG observability report, 2026). The 14-day warmup gate prevents false positives during the period when the rolling baseline is being established. After warmup, a three-sigma deviation triggers an alert.

The practical implication for teams running embedding-dependent systems: the model name attribute on the span is necessary but not sufficient for provenance. The index should store the version hash or the build timestamp of the embedding model used to construct it as an explicit metadata field. Queries should carry the same identifier. When they diverge, the mismatch should be detectable before the quality regression reaches users.

**The Index Staleness Problem.** The vector index is a point-in-time snapshot of the document corpus. The upstream corpus changes continuously. When internal knowledge bases are updated, product documentation is revised, or pricing information changes, the index continues to serve the old content with high confidence. Queries that would retrieve the updated content correctly if the index were fresh are routed to stale documents, and the LLM produces confident, plausibly-formatted, outdated answers.

Monitoring staleness requires tracking a freshness SLA on the index alongside a rolling distribution of the metadata timestamps embedded in retrieved document chunks. The primary signal is the median age of retrieved chunks trending upward while the upstream source documents have been recently modified. Automated data governance pipelines can connect to circuit breakers that flag indices for re-indexing when upstream source tables have undergone modifications without a corresponding index refresh (Gemini RAG observability report, 2026). This is a data governance monitoring requirement. It has no equivalent in the APM taxonomy.

**The Context Window Pressure Problem.** A retriever configured to return a large number of document chunks with large chunk sizes injects a large token payload into the prompt. This payload competes for space in the model's context window with the system prompt, the conversation history, and the user's current query. When the retrieved context is large relative to the available window, the model may not have access to the full system prompt or may lose earlier turns of the conversation history. The result is a model operating with degraded instruction context, producing responses that do not follow the system prompt correctly or that do not maintain conversational coherence.

The observable signal is the correlation between input token count and document injection count across retrieval spans. An input token count that spikes without a corresponding increase in user query length indicates that the retriever is returning excessive context. The relationship between input token count and the number of retrieved documents, computed as a ratio, provides a per-request signal for whether retrieval configuration is operating within its designed bounds (Gemini RAG observability report, 2026; Datadog LLM Observability documentation, accessed Q2 2026).

**The Sensitive Data Injection Problem.** Document repositories in enterprise RAG deployments routinely contain unredacted personally identifiable information, proprietary internal documentation, and confidential business data. When a retriever extracts chunks containing this information and injects them into prompts sent to an external LLM API, the PII or proprietary content crosses a provider boundary that the enterprise did not intend to cross. The telemetry implication is that the retrieved document text stored in observability systems must also be scanned and redacted before storage, because the prompt payloads that land in trace storage carry the same sensitive content. Datadog's LLM Observability integrates with its Sensitive Data Scanner, which scans text streams inside incoming prompts, final completions, and retrieved documents at the agent level before permanent storage, applying regex patterns and classification models to intercept, mask, or redact PII and proprietary tokens in the telemetry pipeline itself (Datadog LLM Observability documentation, accessed Q2 2026; Gemini RAG observability report, 2026). This is a telemetry architecture requirement, not just a compliance requirement: the monitoring system that is supposed to provide visibility cannot store what it monitors in unredacted form.

### The Agentic Loop as a New Span Topology

The fourth layer of the AI Application Surface, the agentic orchestration loop, requires a deeper treatment of its trace topology because it is the layer where the gap between traditional APM's conceptual model and AI system architecture is widest.

The standard distributed tracing model represents execution as a tree of spans. The root span is the incoming request. Each service call the request drives is a child span. Each downstream call those services make is a grandchild span. The tree terminates when all spans close. The tree's depth is bounded by the application's code paths. A typical microservices trace might be three to five levels deep.

An agentic system executing a multi-step task produces a different structure. The root span is the workflow invocation: the user's goal is received, and the orchestrator begins execution. The orchestrator makes an initial LLM call to determine the first action. That LLM call is a child of the workflow span. The LLM returns a tool call specification. The orchestrator executes the tool, producing a child span for the tool execution. The tool returns its result. The orchestrator makes another LLM call, passing the tool result in the conversation history, to determine the next action. This second LLM call is a sibling of the first, not a child of it. The pattern repeats until the agent reaches a stopping condition.

The trace that results from this pattern is a workflow span with N sibling LLM child spans and N sibling tool child spans, interleaved in execution order. The number N is determined at runtime. For a simple task, N might be 3 or 5. For a complex research task, N might be 40 or 60. For a runaway loop that has not reached a stopping condition, N grows without bound until an external constraint terminates execution.

The OpenTelemetry specification assigns `invoke_agent` to the internal execution steps, `invoke_workflow` to the grouping container, and `execute_tool` to the tool execution spans. The topology traced in the preceding paragraphs is representable in this vocabulary. The diagnostic value of the resulting trace, however, depends on what additional context each span carries. An LLM span that records "the model was called with this prompt and returned this completion" is useful. An LLM span that also records "this was the fourth call in the workflow, triggered by the result of tool call 3, and the model chose to call tool 5 next because the prior results did not satisfy the goal" is substantially more useful. The information that distinguishes these two spans is the agent's intermediate reasoning, and there is no standardized convention for recording that reasoning as of Q2 2026 (Gemini OTel semantic conventions analysis, Q2 2026).

The session boundary problem compounds this. A user may interact with an agent over multiple turns, each turn potentially triggering a separate agent execution. The `gen_ai.conversation.id` attribute links turns within a conversational thread. But a single conversational turn may itself trigger multiple parallel agent executions if the orchestrator fans out to multiple specialized agents. The relationship between the session, the conversation, and the individual execution traces requires a three-tier semantic hierarchy, as the OpenTelemetry SIG has proposed in issue 2883: a `session.id` attribute sitting above `gen_ai.conversation.id`, which itself sits above the individual span hierarchy. This three-tier structure is in active development and not yet stable.

### The Observability Surface Has Changed. The Tools Have Not Kept Up.

The **Three-Pillar Strain**, introduced in Chapter 1 as the structural tension between AI workloads and the metrics-logs-traces model, manifests at the AI Application Surface as a specific mismatch: the primitives that AI systems emit most importantly are neither clean metrics, nor structured logs, nor bounded trace trees. They are semantic quality signals on unstructured text payloads, statistical distributions over embedding spaces, and cost attributions across non-deterministic execution graphs.

None of these fit the three-pillar model cleanly. Retrieval relevance is not a metric, because computing it requires a secondary evaluation step rather than a direct measurement. Embedding drift is not a log, because it is a statistical property of a distribution rather than a discrete event. Agentic execution cost is not a trace span attribute, because it is an aggregate computed across a variable-depth tree whose depth is not known at span creation time.

The OpenTelemetry GenAI semantic conventions are the most systematic attempt to build a shared vocabulary for the AI Application Surface, and they have made genuine progress on the LLM call layer. The retrieval layer has partial coverage. The agentic loop layer has foundational operation names but lacks the semantic depth that makes agentic traces diagnostically useful. The quality and evaluation layer has experimental proposals under active debate.

Teams building production AI systems today are not waiting for the standard to stabilize. They are building their instrumentation against whichever combination of standard attributes and vendor-specific extensions gives them the signals they need now, with the knowledge that some of those extensions will become standard and others will not.

There is a further practical complexity at the boundary between the AI Application Surface and the existing observability infrastructure: the privacy and access control requirements are different. Standard APM traces contain service names, user IDs, timing data, and error messages. These are sensitive in aggregate but rarely require field-level redaction before storage. AI traces contain the full text of prompts and completions, which may include medical records, financial data, personal communications, and proprietary business documents depending on the application. The retrieval layer amplifies this because the injected document chunks carry whatever content the document repository holds. An observability pipeline that treats AI traces the same way it treats standard APM traces and stores the prompt payload verbatim is creating a data liability at the same time it is creating operational visibility.

Datadog addresses this at the ingestion boundary with the Sensitive Data Scanner integration, which scans incoming spans for recognized PII patterns before storage. But the design choice to store prompt content in the span rather than in a separate event, while operationally convenient for debugging, creates a mandatory redaction step before the data is safe to store or share across teams. When an OpenTelemetry GenAI span reaches Datadog via native OTLP ingestion without the ddtrace agent, and the `gen_ai.input.messages` attribute contains prompt text, that attribute flows into the general APM Trace Explorer database by default. Preventing unauthorized access to that data across teams requires manually configuring a separate Restricted Dataset policy in the APM interface to align access permissions with the LLM Observability product's own RBAC controls (OpenTelemetry GenAI semantic conventions v1.37, Q2 2026; Datadog LLM Observability documentation, accessed Q2 2026). This is a configuration step that teams migrating from ddtrace to native OTLP ingestion frequently miss.

The cardinality implications of the AI Application Surface's telemetry primitives are significant enough to warrant an early flag here, though full treatment appears in Chapter 3. Token counts, model names, and provider identifiers are low-cardinality dimensions that APM metric pipelines handle without strain. Session IDs, user IDs, and especially prompt text fragments are high-cardinality identifiers that compound storage cost rapidly when placed in metric dimensions. Retrieval relevance scores, when recorded as span attributes rather than as computed metric values, can accumulate into a corpus of float-precision data that grows proportionally to the number of retrieval operations rather than to the number of distinct queries. A platform team that instruments the AI Application Surface naively, recording everything the components emit into indexed dimensions, will encounter the **Cardinality Cliff** introduced in Chapter 3 before they have had time to verify that the instrumentation is producing useful signal.

The financial services team from this chapter's opening case eventually built their own monitoring layer for the vector store. They added custom span attributes for the similarity scores of each retrieved chunk, the version identifier of the embedding model used to build the index, and the timestamp of the most recently modified document in each retrieved result set. They built a background worker to compute rolling embedding drift metrics against a probe query set. They configured alerts on the proximity gap falling below a threshold. Their rebuild took six weeks. Their production retrieval quality issues were detectable within three seconds after the rebuild was deployed.

The six-week rebuild was a vocabulary problem solved by engineering effort. The vocabulary now exists in the corpus of solutions teams have built. The question is how much of it becomes standard infrastructure and how much remains custom work for each team.

There is a useful way to frame what this chapter has established in terms that will carry forward. The AI Application Surface introduces telemetry primitives at three levels of standardization maturity. The first level is settled and portable: token counts, model names, operation types, latency histograms, and finish reasons are in the OpenTelemetry standard and supported by every major observability platform. Teams should instrument these unconditionally. The second level is partial and contested: retrieval span attributes beyond the basics, agentic workflow operation names, and the embryonic evaluation signal conventions. Teams should instrument what exists and plan for the vocabulary to shift. The third level is absent from any standard and requires custom engineering: embedding drift metrics, index staleness tracking, cross-span cost attribution for agent sessions, and retrieval quality scoring at the level of individual span-attached similarity scores. Teams that need these signals today will build them; teams that do not yet need them should understand that the gap exists before a quality regression surfaces unexpectedly.

The financial services team needed all three levels. Their gap was entirely in the third tier: signals that no standard specified, no vendor provided out of the box, and no monitoring discipline had previously required. The rebuild that closed the gap was six weeks of precision instrumentation work. For each team that ships a production RAG system without closing that gap, there is a version of December 2024 waiting somewhere in the future.

The components are named. The primitives are cataloged. The gaps in the standard are documented. Naming the surface is the prerequisite. Knowing what it costs to observe the surface completely is the next discipline.

### Key Takeaways

The AI Application Surface introduces five distinct component classes, each emitting telemetry primitives that did not exist in the pre-AI APM taxonomy: LLM call spans (tokens, cost, finish reason), RAG pipeline spans (retrieval relevance, document metadata), vector store spans (index staleness, embedding model version), agentic orchestration spans (workflow depth, tool invocation sequences), and multi-model routing spans (provider selection, cross-model correlation). The most dangerous failure modes on this surface are silent: embedding drift, index staleness, and retrieval quality degradation produce no operational error signal and are invisible to any monitoring that does not instrument the retrieval layer explicitly.

- Embedding drift is the canonical silent failure of retrieval systems: a third-party embedding API can update its model behind a stable endpoint, breaking coordinate-space compatibility between the runtime query embedding and the pre-computed vector index, with no operational error signal and plausible-looking similarity scores.
- The retrieval layer requires its own instrumentation discipline: similarity scores, embedding model version identifiers, and index update timestamps must be captured explicitly as span attributes; they are not emitted automatically by any general-purpose APM framework.
- The three-tier standardization maturity of the AI Application Surface determines instrumentation strategy: the settled tier (token counts, model names, latency) should be instrumented unconditionally; the partial tier (retrieval attributes, agentic operation names) should be instrumented where available with a plan for vocabulary shifts; the absent tier (embedding drift metrics, cross-span session cost attribution) requires custom engineering.
- Prompt and completion content stored in spans creates a mandatory data classification and redaction step; observability pipelines that treat AI spans the same as standard APM spans risk creating data liability at the moment they create operational visibility.
- The OTel GenAI semantic conventions at v1.37 have settled the LLM call layer and partially covered retrieval; the agentic workflow layer and the evaluation signal layer remain contested or absent, meaning teams building agentic systems today are instrumenting against a vocabulary they know will shift.
