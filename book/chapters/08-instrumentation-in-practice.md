# Chapter 8: Instrumentation in Practice

*In this chapter, the book moves from what Datadog's AI surface offers to whether telemetry actually reaches it. Auto-instrumentation coverage at Q2 2026 is generous for common frameworks but silent about the boundaries that matter most: context propagation across async dispatches, evaluation signal capture, and cost aggregation at the workflow root. The chapter introduces the context propagation contract as the design-time commitment that separates a coherent trace from a pile of orphaned spans, examines the OpenTelemetry GenAI semantic conventions at v1.37, maps the ddtrace and OTel ingestion paths, and documents the Datadog MCP Server's inversion of the developer-to-platform relationship.*

The engineering team at a fintech company had spent three weeks building an agentic document-review workflow. Their stack was deliberately current: a LangGraph orchestrator dispatching to an Anthropic Claude model for extraction, a retrieval step pulling from a Pinecone vector store, and a tool layer writing structured results back to a Postgres database. They had instrumented the application with OpenTelemetry from the start. They were confident the observability was sound.<sup>[1]</sup>

On the day they pushed to staging, the traces showed a single span. One span. The orchestrator call. Everything beneath it was invisible. The LangGraph graph traversal, the Claude message turns, the retrieval steps, the database writes: none of it had emitted a child span. The OpenTelemetry libraries they were using predated the GenAI semantic conventions and had no understanding of what an LLM call was. The trace was technically correct. It was also operationally useless: a single root span with a duration of forty-three seconds and no children to explain where those seconds went.<sup>[1]</sup>

They spent two days retrofitting. One developer handled the Anthropic SDK wrapping. Another added the LangGraph hooks. A third sorted out context propagation across the retrieval step, which ran in a separate async task and had broken the parent span association. By the end, they had what they should have had on day one: a trace with typed spans for each workflow stage, token counts on the LLM spans, latency breakdowns for retrieval, and a cost estimate on the root span. The delay cost them a week of the staging cycle.<sup>[1]</sup>

The story matters less for the drama than for what it reveals about the instrumentation landscape at Q2 2026. The conventions exist. The auto-instrumentation libraries exist. The mapping from library calls to typed spans exists in documented form. But the path between "deploy LLM application" and "get a useful trace" still has gaps, version dependencies, and propagation requirements that are invisible until a team encounters them in production.

The question that two days of retrofitting raises is not about that team's preparation. It is about what the instrumentation landscape expects a team to know before the first staging deploy, and what it leaves invisible until after. At Q2 2026, the answer is: quite a lot. **The Datadog AI Surface**, the portfolio of capabilities built between August 2023 and Q2 2026, is only useful to the degree that data reaches it. The span taxonomy, the hallucination evaluators, the AI Agents Console: none produce value if the application's instrumentation layer does not emit spans in the shape those features expect. Instrumentation is the prerequisite. At Q2 2026, that prerequisite is still nontrivial to satisfy correctly.

## The OpenTelemetry GenAI Semantic Conventions at v1.37

The OpenTelemetry community has been working on a standard vocabulary for generative AI telemetry since 2023. The effort has been productive, contested, and structurally complicated in roughly equal measure. Version 1.37 of the semantic conventions, the reference point for Q2 2026 implementations, establishes a unified framework for normalizing the operational footprint of LLM transactions, agentic orchestrations, and external tool interactions.<sup>[2]</sup>

The convention's most important structural decision is its use of span attributes to classify operations. The required `gen_ai.operation.name` attribute identifies the transaction type: `chat`, `generate_content`, `text_completion`, `embeddings`, `create_agent`, `invoke_agent`, `invoke_workflow`, or `execute_tool`. The required `gen_ai.provider.name` attribute identifies the underlying model provider, replacing the earlier `gen_ai.system` namespace. Together, these two attributes create the classification layer that backends use to distinguish an LLM call from a tool invocation from an agent workflow step. The `gen_ai.conversation.id` attribute provides thread-level correlation for multi-turn interactions. The `gen_ai.request.model` and `gen_ai.response.model` attributes capture what the application asked for and what the provider actually delivered, which differ in practice when providers route requests across model versions.<sup>[2]</sup>

The token accounting attributes are finalized and production-stable. `gen_ai.usage.input_tokens` and `gen_ai.usage.output_tokens` replace the earlier `prompt_tokens` and `completion_tokens` names. Critically, v1.37 adds `gen_ai.usage.reasoning.output_tokens`, which tracks tokens consumed by internal chain-of-thought reasoning in models that expose this data. It also adds `gen_ai.usage.cache_read.input_tokens` and `gen_ai.usage.cache_creation.input_tokens`, which matter for providers like Anthropic that charge differently for cached input. A team not capturing these fields will see prompt-caching cost reductions appear as unexplained billing drops rather than as a measured optimization outcome.<sup>[2]</sup>

The operational metrics are also finalized. The `gen_ai.client.token.usage` counter, the `gen_ai.client.operation.duration` histogram, and the `gen_ai.client.operation.time_to_first_chunk` histogram give platform teams the basis for SLO definitions against latency and token throughput. The server-side equivalents (`gen_ai.server.request.duration`, `gen_ai.server.time_to_first_token`, `gen_ai.server.time_per_output_token`) allow teams running self-hosted inference to measure performance at the serving layer. These metrics enable the operational SLO discipline that **The Eval/Ops Split** depends on for the operational side of that boundary.<sup>[2]</sup>

The providers documented within v1.37 include OpenAI, Anthropic, AWS Bedrock, Azure AI Inference, Azure AI OpenAI, Google Vertex AI, Google Gemini, DeepSeek, Perplexity, Cohere, and xAI (per the OpenTelemetry GenAI semantic conventions, accessed Q2 2026).<sup>[2]</sup>

One structural change deserves specific attention. The GenAI semantic conventions have been extracted from the core OpenTelemetry repository into a dedicated repository: `open-telemetry/semantic-conventions-genai`. This is an explicit architectural decision to bypass the review queue bottlenecks of the primary project and allow the GenAI working group to iterate more rapidly.<sup>[2]</sup> The practical consequence is dual-repository navigation: issues and pull requests that lived in the main repository are migrating to the new workspace. Teams following GenAI spec development need to track the standalone repository, not the parent. The split creates a temporary alignment challenge. It is the right structural call given how fast generative AI development moves relative to the rest of the OpenTelemetry specification surface.

The opt-in mechanism for v1.37 compatibility is explicit. An application must set the environment variable `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` to instruct compliant instrumentation libraries to emit the v1.37 convention set rather than the pre-v1.36 layout. This is a breaking-change guard: production systems that have not set this variable will emit the older attribute names, and backends expecting v1.37 names will see missing or misnamed fields. Teams adopting the GenAI conventions for the first time should set this variable from the outset. Teams migrating existing applications should treat the migration as a schema change and verify that downstream queries, dashboards, and alerts account for the attribute renaming before enabling it.<sup>[2]</sup>

The SIG roadmap for the period immediately following v1.37 focuses on moving core span and event attributes out of experimental tracking and into a finalized stable release in the standalone repository. The practical target is to deprecate the `OTEL_SEMCONV_STABILITY_OPT_IN` opt-in mechanism entirely, so that compliant libraries emit stable v1.37+ attributes by default. This stabilization milestone matters for enterprise adoption: most enterprise security teams resist deploying applications with environment variables labeled "experimental" in CI/CD pipelines, regardless of the stability of the underlying attributes the variable activates. The gap between "experimental" as a label and "stable" as a deployment posture is one of the reasons adoption of the GenAI conventions lags behind their actual technical readiness (per the OpenTelemetry GenAI semantic conventions roadmap, accessed Q2 2026).<sup>[2]</sup>

### What the v1.37 Conventions Leave Open

Two structural problems remain actively contested in the GenAI working group at Q2 2026, and both affect agentic workloads specifically.

The first is span topology for autonomous, multi-step, and multi-agent workflows. The conventions handle basic operations well. They do not yet handle the recursive depth, branching tool invocations, and session-boundary questions that emerge when multiple agents interact across a single user-initiated task. Meta-issue `#2664` in the working group catalogs six relational structures for agent topologies, but the mathematical modeling of these structures within standard distributed tracing limits is still under discussion.<sup>[2]</sup> The core difficulty is that standard tracing was designed for request-response trees; agentic workflows produce graphs, and graphs do not always serialize cleanly into parent-child span hierarchies. This is the principle-level problem **The Span Topology Problem** names. The conventions are moving toward resolution via new attribute sub-families (`gen_ai.task.*`, `gen_ai.action.*`, `gen_ai.agent.skills.*`), but these are in progress, not stable.

The second is payload handling. The decision to isolate prompt and completion content into separate OTel Events, rather than including them directly in span attributes, remains contested. The rationale is sound: embedding large text payloads in span attributes degrades tracing infrastructure performance. The concern is that separating content into events removes spatial and temporal locality from the primary trace, complicating backend processing and sampling. An active discussion in the working group documents an ongoing push by several observability backend vendors to reconsider this choice.<sup>[2]</sup> Datadog's own resolution is a proprietary adaptation: per-span payload extraction subject to the 1 MB hard limit on prompt and response payloads described in Chapter 7, which stores content on the span itself rather than in separate events.

### The MCP Protocol Extension

The integration of the Model Context Protocol into the OpenTelemetry GenAI conventions is a recent and significant addition. MCP operates over JSON-RPC layers to establish standard communication channels between models, development tools, and external data sources. Traditional HTTP or generic RPC telemetry tracking fails to capture nested interactions across a persistent MCP transport connection, because multiple multiplexed JSON-RPC messages may traverse a single session. The GenAI convention addresses this by introducing dedicated metrics: `mcp.client.operation.duration` and `mcp.server.operation.duration` as histograms measuring client and server execution latency, and `mcp.client.session.duration` and `mcp.server.session.duration` for tracking connection lifespan.<sup>[2]</sup> The convention enforces that tool calls executed over MCP use `gen_ai.operation.name = "execute_tool"`, aligning them with the standard span kind, and requires propagation of trace context within the JSON-RPC request dictionary. MCP-mediated tool calls are traceable with the same span structures as directly invoked tools, provided both client and server are instrumented.

Teams using Webpack must explicitly exclude integration modules from bundling.<sup>[3]</sup>

### What Falls Outside Auto-Instrumentation

The auto-instrumentation boundary is meaningful. Several frameworks and use patterns require manual instrumentation as of Q2 2026.

LangChain is not listed in the auto-instrumentation documentation independently from LangGraph. Teams using LangChain's lower-level chain and agent APIs directly, rather than through LangGraph's graph execution model, should verify whether their specific call patterns are covered or whether they need manual decorator wrapping.<sup>[3]</sup>

Microsoft AutoGen does not appear in the published auto-instrumentation list at Q2 2026.<sup>[4]</sup> Teams using AutoGen for multi-agent coordination must instrument manually using the `@workflow`, `@agent`, `@task`, and `@tool` decorators in Python, or the equivalent `llmobs.trace({kind:'...'})` calls in Node.js, combined with `LLMObs.annotate(input_data, output_data, metadata, metrics, tags)` to attach typed attributes to each span. This is not burdensome in absolute terms, but it is meaningfully different from zero-configuration: the team must understand the span taxonomy, apply decorators at the correct abstraction boundaries, and handle context propagation across asynchronous execution paths.<sup>[3,4]</sup>

LlamaIndex falls into the same category. The auto-instrumentation list does not include a LlamaIndex integration at Q2 2026. Teams using LlamaIndex for RAG pipeline construction can instrument via the OpenTelemetry path if LlamaIndex's OTel integration emits v1.37-compatible spans, or via manual decorator wrapping for the Datadog-native path.<sup>[3]</sup>

Strands Agents, the AWS-developed agent framework, requires explicit opt-in via `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` because Strands emits older GenAI semantic convention fields by default. Without this variable set, the spans Strands emits will not map correctly to the v1.37 attribute names that Datadog's native OTel ingestion expects.<sup>[4]</sup>

The LangChain and LangGraph distinction is worth resolving precisely. LangGraph instruments `CompiledGraph.invoke`, `CompiledGraph.stream`, `Pregel.invoke`, and `Pregel.stream` through both synchronous and asynchronous variants. LangGraph also captures CrewAI memory state and LangGraph's own graph state within the trace. The `@workflow` span wraps the top-level graph invocation; beneath it, each node execution is captured as an `agent` or `task` span; each LLM call from within a node is captured as an `llm` span; each tool call is captured as a `tool` span. This hierarchy is enforced at SDK level, which is the resolution that the span topology problem requires: the parent-child relationships are built at emission time by the SDK's context propagation, not reconstructed by the backend from trace IDs and timestamps.<sup>[3]</sup>

### The OTel Native Ingestion Path

For teams not using ddtrace, Datadog provides a native OTel ingestion path. Applications instrumented with the OpenTelemetry SDK, emitting v1.37 GenAI semantic convention spans via OTLP, can route directly to the Datadog backend without installing the ddtrace agent. This path covers the same provider list the convention specifies and handles automatic mapping between OTel field names and Datadog's native schema.<sup>[3]</sup>

The schema mapping is defined and documented. When a span with `gen_ai.operation.name = "chat"` arrives via OTLP, Datadog's intake engine maps it to an `llm` span kind in the LLM Observability schema. The `gen_ai.provider.name` field maps to the model provider field. The `gen_ai.conversation.id` maps to `session_id` and populates the session tracking interface. Tool-specific attributes map as follows: `gen_ai.tool.name` maps to the span name, `gen_ai.tool.call.id` maps to `metadata.tool_id`, `gen_ai.tool.call.arguments` maps to `input.value`, and `gen_ai.tool.call.result` maps to `output.value`.<sup>[2,3]</sup>

For prompt and completion content extraction, the intake engine applies a priority hierarchy. It first checks for direct span attributes: `gen_ai.input.messages`, `gen_ai.output.messages`, and `gen_ai.system_instructions`. If these are absent, it falls back to parsing the associated OTel span event named `gen_ai.client.inference.operation.details`. For embedding spans, `gen_ai.input.messages` maps to `meta.input.documents`, and the output value is set to the static string `[N embedding(s) returned]`.<sup>[2,3]</sup>

Three practical limitations of the native OTLP path require explicit attention.

The first is tag truncation: any `gen_ai.*` attribute not explicitly defined in Datadog's mapping table falls back to a generic span tag, and Datadog enforces a 256-character maximum per tag value. Custom metadata arrays or long unstructured parameters are permanently truncated at ingestion.<sup>[2]</sup>

The second is attribute filtering: non-mapped custom attributes are dropped entirely. Application context tags that do not match the standard specification cannot be carried through the OTel path.<sup>[2]</sup>

The third is access control: OTel traces arriving via OTLP can simultaneously appear in Datadog's general APM Trace Explorer and in LLM Observability. Teams with PII-sensitive prompt content must configure a Restricted Dataset policy in APM to prevent prompt payloads from being visible to engineers who have APM access but should not have access to the conversation content.<sup>[2,3]</sup>

There is also an ID format mismatch between OTel and Datadog's external evaluations API. OTel manages trace and span identifiers as hexadecimal strings. Datadog's evaluations HTTP API requires numeric identifiers submitted as decimal strings. Engineers implementing custom evaluators against production OTel traces must convert identifiers using the equivalent of `int(hex_id, 16)` before submitting evaluation results. The issue is documented and straightforward to solve, but it is the kind of friction that causes evaluation pipelines to fail silently if the conversion step is omitted.<sup>[2]</sup>

The OTel path also supports legacy schema compatibility. If an incoming span lacks the standard `gen_ai.operation.name` attribute, the intake engine inspects the legacy `llm.request.type` field used by older instrumentation frameworks such as OpenLLMetry. Chat and completion operations map to `llm`; embedding operations map to `embedding`; rerank and unknown operations fall back to `workflow`. Teams migrating from early OpenLLMetry instrumentation do not need to re-instrument immediately, but should plan the migration to v1.37 attributes to preserve full feature access.<sup>[2]</sup>

## The Datadog MCP Server

The Datadog MCP Server, generally available as of April 2026, inverts the normal relationship between the developer tool and the observability platform (per the April 2026 "This Month in Datadog" announcement).<sup>[5]</sup> In the outbound direction, instrumentation is application code emitting spans that travel to the Datadog platform. In the inbound direction, the MCP Server makes Datadog a tool the AI agent calls: an AI coding assistant such as Claude Code or Cursor queries live production telemetry through the MCP connection and reasons about it alongside the source code already in context. The observability platform is no longer only a dashboard the human navigates; it is a data source the assistant interrogates.

The protocol mechanics follow the MCP specification (2024-11-05 version), which describes a client-host-server architecture.<sup>[6]</sup> The host is the AI coding assistant. The MCP client is embedded within it. The Datadog MCP Server is the server process, exposing Datadog's telemetry APIs (logs, metrics, traces, monitors, dashboards, and notebook creation) as MCP tools via JSON-RPC over a stateful session. When the developer asks the assistant a question requiring production evidence, the assistant calls the relevant Datadog tool through the MCP client, the server translates that into a Datadog API request, and returns structured telemetry as content. The server automatically determines which API to call based on the prompt's natural-language structure, so a developer asking about error logs receives log search results without specifying the Logs API endpoint explicitly.<sup>[5]</sup>

Authentication is via OAuth, tying MCP access to Datadog identity. A developer who does not have access to production LLM span payloads in the Datadog UI will not have access through the MCP Server either. The same Restricted Dataset policies that protect PII in the APM Trace Explorer apply to MCP queries. The HIPAA-eligible status of the MCP Server follows from these access controls, satisfying the audit logging, access control, and encryption-in-transit requirements that HIPAA eligibility demands (per Datadog MCP Server documentation, accessed Q2 2026).<sup>[4,5]</sup>

**The MCP Telemetry Bridge.** The architectural pattern that makes a live observability backend queryable as a tool by an AI coding agent, collapsing the context-switch between code examination and production evidence, and connecting the agent's reasoning directly to the telemetry it needs to diagnose what the code is doing in production. The bridge is not itself a traced span: the AI assistant's reasoning over retrieved telemetry is not captured in Datadog LLM Observability unless the team has separately instrumented the assistant's reasoning loop.

That gap is the governance problem the pattern introduces. When an AI coding assistant calls the MCP Server, retrieves production telemetry, and uses that telemetry as context for generating a code change or a PR fix, the chain from production observation to code modification passes through an AI reasoning step that is not, by default, instrumented or auditable. The production span shows what the service did. The code change shows what the developer decided to do. The MCP-mediated analysis that connected them exists only in the assistant's transient reasoning context. For post-incident review, that gap matters. The MCP specification addresses authorization at the protocol level but does not define what happens to the AI reasoning that occurs between the telemetry query and the resulting action. That governance layer is outside the current protocol scope and outside the current Datadog product scope at Q2 2026.<sup>[6]</sup>

## Context Propagation: The Hidden Fault Line

The auto-instrumentation coverage matrix describes which libraries emit spans. It says nothing about whether those spans arrive at the backend organized as a coherent trace. That second problem is context propagation. It is where more instrumented AI applications silently break than the framework coverage question.

**The Context Propagation Contract.** The explicit agreement, established at system design time, about how trace context is passed across every async boundary and agent hand-off in a distributed AI system. Without it, spans become orphaned and the trace tree is irrecoverable. The contract specifies, for each dispatch mechanism in the application architecture, which component injects trace context and which component extracts it; when that specification is absent or incomplete, the result is not a broken trace but a collection of disconnected traces that the backend cannot reassemble.

Context propagation in distributed tracing is the mechanism by which a trace ID and a parent span ID are carried from one component to the next, so that all spans belonging to the same logical operation end up in the same trace tree. In a synchronous HTTP service, this is automatic: the tracing library injects trace context into outgoing HTTP headers and extracts it from incoming headers, and the parent-child relationship is established without explicit developer action. The W3C Trace Context standard, which Datadog uses as its carrier format, is supported by every major HTTP framework.

AI application architectures break this assumption in at least three common ways.

The first is asynchronous task dispatch. A LangGraph orchestrator invokes a retrieval step by dispatching an async Python coroutine. If the async coroutine is created with `asyncio.create_task()` rather than `asyncio.ensure_future()` with explicit context capture, the trace context from the parent span does not automatically propagate into the new task's execution context. The OpenTelemetry Python SDK uses Python's `contextvars.ContextVar` for context storage, which propagates correctly across `await` boundaries but not across thread boundaries or across tasks created without explicit context copying. The result: the retrieval span is emitted with a fresh, unparented trace ID, and two spans that should form a parent-child pair become two separate unrelated traces in the backend. This was one of the two days' worth of work the fintech team in the opening case spent resolving.<sup>[1,2]</sup>

The resolution requires explicit context capture at task creation. In Python, the pattern is to capture the current context with `contextvars.copy_context()` and pass it as the `context` argument to `asyncio.get_event_loop().run_in_executor()` or to `asyncio.create_task()`. The ddtrace library handles this automatically for call patterns it instruments, but custom async dispatch code outside the instrumented path requires the developer to manage context explicitly.<sup>[3]</sup>

The second is thread pool execution. A tool function that calls an external service via a `concurrent.futures.ThreadPoolExecutor` executes in a thread that does not inherit the calling thread's Python context. The trace context from the parent span, stored in a `ContextVar`, is not carried into the thread. The tool's span becomes an orphan. The fix is to use ddtrace's `Pin` API or the OTel `Context.attach()` and `Context.detach()` functions to propagate context explicitly across the thread boundary. This is not obscure knowledge, but it is also not surfaced as a warning in the ddtrace getting-started documentation; teams encounter it when they notice orphaned tool spans in their trace view.<sup>[3,4]</sup>

The third is cross-process agent-to-agent communication. In a multi-agent system where a parent agent dispatches work to a sub-agent via an HTTP call or a message queue, trace context propagation depends on the calling code injecting the trace context into the HTTP headers or the message payload, and the receiving code extracting it. If both agents use ddtrace, the library handles HTTP injection and extraction automatically. If the parent agent uses ddtrace and the sub-agent uses a different instrumentation library, the trace context format must be compatible. W3C Trace Context ensures compatibility across conforming implementations, but message-queue-based dispatch requires the calling code to inject context into a message attribute and the receiving code to extract it from the same attribute. This is framework-specific and not automatically handled by ddtrace for every queue provider.<sup>[4]</sup>

The practical diagnostic for context propagation failures is the orphaned span pattern. When a team opens a trace in the LLM Observability trace view and sees the root `workflow` or `agent` span with fewer child spans than the application's execution should have produced, the likely cause is a context propagation failure at one of the three points described above. The investigation starts by checking whether the missing spans appear as separate, unparented traces in the Trace Explorer; if they do, the propagation boundary is identifiable.

The deeper implication is architectural. AI application instrumentation cannot be treated as a purely operational concern retrofitted after the application is built. Context propagation correctness depends on how the application dispatches work: whether it uses async coroutines, thread pools, or message queues; whether those dispatch mechanisms propagate Python context variables; and whether the receiving code on the other side of a process boundary extracts trace context from the carrier. Teams that make these decisions without considering their instrumentation implications will produce incomplete traces. Teams that consider instrumentation as an architectural requirement from the first design session will make dispatch choices that are instrumentable by construction.

## Recurring Instrumentation Failures

Three sub-patterns appear repeatedly in production instrumentation failures documented in the corpus and in publicly available Datadog engineering documentation. Each is distinct enough to name plainly.

The first is the version mismatch trap: the failure mode in which an instrumentation library, a framework, and a semantic convention version are each individually current but mutually incompatible, causing spans to be emitted with incorrect or missing attributes that the backend silently drops or misclassifies. The trap closes when a team upgrades one component without upgrading the others.<sup>[2]</sup>

The trap manifests in two directions. The first is the framework-ahead-of-convention case: a framework ships support for a new model provider before the GenAI semantic conventions have standardized the attribute names for that provider's specific response fields. The framework chooses attribute names that seem reasonable but conflict with later-standardized names. Teams instrumenting with that framework version lock in the non-standard attribute names. When the convention stabilizes, the backend maps the standardized names but not the framework's earlier choice, and cost estimates or token counts silently disappear from the trace.<sup>[2]</sup> The second direction is the convention-ahead-of-library case: the GenAI conventions update to v1.37, renaming `prompt_tokens` to `gen_ai.usage.input_tokens`, but the instrumentation library has not yet shipped a release emitting the new names. Teams running the updated convention opt-in variable get the new backend schema but receive data from the old attribute names, which the intake engine does not map. Both directions produce the same symptom: spans that appear in the trace view with missing token counts, cost estimates, or typed attributes.

The resolution discipline requires pinning all three components to a tested combination: the ddtrace version, the framework integration version, and the semantic convention opt-in state. Datadog's documentation for each integration lists the minimum ddtrace version required for that integration's GA support. Teams that pin to a tested combination and update the combination together as a unit, rather than updating individual components independently, avoid most version mismatch failures.<sup>[3]</sup>

The second is the evaluation span gap: the failure mode in which a team instruments the operational spans for an AI application (LLM calls, tool calls, retrieval steps) but does not instrument the evaluation signals that the same application produces in production, creating a trace that answers the latency question but not the quality question.<sup>[3]</sup>

The gap is a direct instrumentation-layer consequence of **The Eval/Ops Split**. Operational spans are emitted by the framework integration automatically. Evaluation signals, such as hallucination detector scores, relevance scores from a retrieval evaluator, or ground-truth comparison results, are computed by application code and require explicit instrumentation. The Datadog LLM Observability SDK provides the `LLMObs.annotate()` method for attaching evaluation scores to existing spans as metadata, and the external evaluations HTTP API for attaching evaluations computed outside the ddtrace process. Both require deliberate implementation.<sup>[3]</sup>

The gap appears most commonly in RAG pipelines. A team instruments the vector search step with a `retrieval` span and captures the documents returned. The span records latency and document count. It does not automatically record the relevance score of the returned documents unless the application explicitly reads that score from the vector database response and passes it to `LLMObs.annotate()`. A quality regression driven by declining retrieval relevance will be invisible in the operational trace, because the only number that would reveal it is the one the team did not annotate. The LLM call that follows will be slower and more likely to produce a poor response, but the trace will attribute the degradation to the LLM span rather than to the retrieval step that fed it bad context.<sup>[3]</sup>

The resolution is to treat evaluation signal capture as a first-class instrumentation requirement at design time, not as a post-production enhancement. For each component that produces a quality signal (relevance score, confidence estimate, evaluation result), the team designates who reads that signal from the component's output and how it is attached to the corresponding span. One to three lines of annotation code per component in practice. The design decision is what matters.

The third is the cost attribution blindspot: the failure mode in which a team captures token counts at the span level but does not aggregate those counts into a cost attribution that connects to a business transaction, making the per-span cost visible but the per-user-request cost invisible.<sup>[3]</sup>

The auto-instrumentation libraries emit `gen_ai.usage.input_tokens` and `gen_ai.usage.output_tokens` on each LLM span, and Datadog's LLM Observability computes an estimated cost for each span from these token counts and the provider's published pricing. A team using auto-instrumentation correctly will see per-span cost estimates in the trace view. What the auto-instrumentation does not provide is the aggregation of those per-span costs into a cost attributed to the root user interaction. A multi-step agent workflow that makes twelve LLM calls across four models in the course of handling a single user request will produce twelve cost estimates, one per LLM span. The total cost of that user request is the sum of those twelve estimates, but that sum is not automatically computed and associated with the root `workflow` span.<sup>[3]</sup>

Teams that want to know the cost distribution across users, or the cost of their most expensive user requests, need to either implement that aggregation themselves using `LLMObs.annotate()` on the root span, or query the trace data with an aggregation expression in the Trace Explorer. The resolution is to add a post-processing step at the workflow level: after the workflow completes, sum the cost estimates from child spans and annotate the root span with the total. This requires reading the child span cost metadata from the completed trace, accessible via the ddtrace trace context after the child spans close. It is a ten-line implementation for a Python decorator applied to the root workflow function. Teams that do not implement this aggregation will be unable to answer the question "what did this user request cost," which is the unit of cost attribution that pricing decisions, cost governance, and per-customer showback require.<sup>[3]</sup>

## The Instrumentation Coverage Reality at Q2 2026

A team instrumenting a production AI application in Q2 2026 has three choices that are genuinely distinct in their trade-offs.

The first is the ddtrace auto-instrumentation path. Setup requires installing ddtrace, setting four environment variables (`DD_LLMOBS_ENABLED`, `DD_LLMOBS_ML_APP`, `DD_API_KEY`, `DD_SITE`), and launching the application under `ddtrace-run` (Python) or with `NODE_OPTIONS=", import dd-trace/initialize.mjs"` (Node.js). For applications using OpenAI, Anthropic, Bedrock, LangGraph, CrewAI, LiteLLM, Google GenAI, Google ADK, OpenAI Agents SDK, Pydantic AI, Vertex AI, or the Vercel AI SDK, the result is zero-code span capture for all supported call patterns. The spans land in LLM Observability with typed attributes, cost estimates, token counts, and correlation to surrounding APM spans in the same trace ID. The limitation is SDK coupling: moving to a different observability backend requires removing or replacing that instrumentation.<sup>[3]</sup>

The second is the OTel GenAI conventions path. Setup requires an OpenTelemetry SDK installation configured to emit spans with v1.37 GenAI attributes, the `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` environment variable, and an OTLP export endpoint pointing to the Datadog API. For teams using auto-instrumentation layers like Arize's OpenInference (supporting over 32 libraries) or Traceloop's OpenLLMetry (supporting over 34 libraries), this path provides broader framework coverage than the ddtrace path in some areas, while sacrificing tight integration with Datadog-specific features like App and API Protection, Continuous Profiler, and the full Ingestion Rules surface.<sup>[2,4]</sup> The benefit is portability: OTel-instrumented applications can switch backends without re-instrumenting.

The third is a hybrid. Teams using ddtrace for their service-level APM instrumentation but needing to capture AI spans from a framework not yet covered by ddtrace auto-instrumentation can use the OTel GenAI path for the AI spans and the ddtrace path for the surrounding services. Datadog's OTLP receiver in the agent, enabled via `DD_OTLP_CONFIG_RECEIVER_PROTOCOLS_GRPC_ENDPOINT` or the HTTP equivalent on Agent version 6.32 or later, accepts both ddtrace spans and OTel spans on the same agent instance. The combined trace, stitched together by the shared trace ID in W3C Trace Context headers, provides full-stack correlation without requiring all components to use the same instrumentation library.<sup>[4]</sup>

The practical boundary between these paths is not purely technical. Teams with existing OpenTelemetry investment across their service mesh will prefer the OTel path to maintain consistency. Teams starting from scratch with a Datadog subscription and no prior OTel investment will likely find the ddtrace auto-instrumentation path faster to initial coverage. Teams with frameworks that fall outside both auto-instrumentation catalogs, such as AutoGen or custom orchestration systems, will spend manual instrumentation effort regardless of which path they choose for the covered portion of their stack.

### The Strands and Bedrock Agent Edge Cases

Two instrumentation cases warrant specific treatment because they represent the edge of what is documented at Q2 2026.

Strands Agents, the AWS-developed agent framework, is listed in the Datadog auto-instrumentation coverage. Teams using Strands must set `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` because Strands emits GenAI spans using pre-v1.36 attribute names by default. Without the opt-in variable, the Datadog intake engine will not map Strands' span attributes correctly to LLM Observability schema fields.<sup>[4]</sup> This is a version-dependency trap in miniature: the Strands application will produce traces that appear in APM but lack typed LLM Observability attributes, and the team will not receive token counts, cost estimates, or typed span kinds. The fix is a single environment variable. Discovering the need for it requires either reading the integration-specific notes or encountering the missing-attribute symptom in production.

The Amazon Bedrock Agents integration has a different specificity requirement. The integration automatically instruments the `InvokeAgent` method, which captures the overall agent call. But intra-agent steps, the individual reasoning and tool-use turns within a Bedrock agent's execution, are not captured by default. To enable intra-agent step tracing, the team must set `enableTrace=True` in the `InvokeAgent` request parameters. This is a call-site configuration, not an SDK configuration: every `InvokeAgent` call that should emit step-level spans must include this parameter.<sup>[3]</sup> Applications that call `InvokeAgent` without it will produce single-span traces for agent executions that may span dozens of internal steps, reproducing exactly the diagnostic blindness from the opening case.

## What Coverage Means in Practice

The auto-instrumentation coverage matrix answers the most immediate question: does my framework work out of the box? For most widely adopted frameworks at Q2 2026, the answer is yes on the ddtrace Python path. For Node.js, the coverage is somewhat narrower but covers the major provider SDKs. For Java, coverage is narrower still, confined to the OpenAI integrations that reached GA with dd-trace-java v1.51.0.<sup>[3]</sup>

But coverage for the framework does not mean coverage for the application. A team running OpenAI Agents SDK with a custom tool layer that calls internal microservices needs auto-instrumentation for the Agents SDK spans and manual instrumentation for the custom tool spans. A team running LangGraph with a retrieval step that calls a vector database through a bespoke client library will get auto-instrumented orchestration spans and will need to manually instrument the vector database calls for typed retrieval spans with relevance scores. A team building a multi-agent pipeline where agents communicate via message queues will get instrumented spans at each agent boundary but will need to handle context propagation across the queue explicitly.

Coverage, context propagation, and evaluation signal capture are three distinct instrumentation requirements that address three distinct questions: did something happen, are the somethings connected, and did the somethings produce good results. Auto-instrumentation answers the first question automatically and provides scaffolding for the other two. It does not answer all three by itself.

The Datadog Agent evolution described in the May 2026 DASH talk on the Agent's transformation into an AI execution layer provides a useful frame for understanding the instrumentation problem at a longer time horizon.<sup>[7]</sup> The Agent began as a passive telemetry collector in 2010, then evolved through successive capability additions: Kubernetes context awareness, serverless extension support, edge processing for PII redaction, remote configuration for zero-restart policy updates, and fleet automation for at-scale agent management. The AI-era addition is the **Private Action Runner**, now embedded in Agent version 7.77, making the Agent capable of executing remediation steps inside a private environment rather than just collecting data from it. The instrumentation trajectory follows the same arc: passive collection first, then typed span classification, then evaluation signal capture, then cost aggregation, then autonomous analysis. The convention maturation at v1.37 represents approximately the second stage.

The Shri Subramanian talk from the April 7, 2026 Datadog event on practical AI-enabled observability makes the same point from the evaluation side.<sup>[8]</sup> Subramanian describes the development feedback loop for AI agents as fundamentally different from the classic application development loop: the classic loop iterates on code; the AI agent loop iterates on prompts, models, and tool descriptions. Instrumentation that captures code execution but not the prompt-model-tool decision surface leaves the AI agent development loop without the data it needs to close. The eval-driven development methodology Subramanian describes, in which evaluation scores serve as the CI system for AI agent iterations, requires instrumentation that goes beyond span emission to include evaluation annotation on every agent invocation that matters.

The resolution is not complicated, but it requires a design decision at the application architecture stage. Teams that identify their manual instrumentation boundaries at design time, add decorator wrapping for custom tools and retrieval functions, enforce context propagation across asynchronous boundaries, annotate evaluation signals as first-class span attributes, and aggregate per-span costs into root-span totals will have complete, useful traces from the first production deployment. Teams that instrument the framework and expect auto-instrumentation to cover everything will produce the incomplete trace that was the fintech team's first symptom, and will spend the next two days fixing it.

The conventions exist. The libraries are improving with each release cycle. The MCP Server is creating new patterns for how telemetry feeds back into development. What has not changed is that observability is a design choice made with the same intentionality as the application architecture. AI-assisted development can generate a LangGraph orchestrator and a set of tool functions in an afternoon, but the instrumentation discipline that makes that orchestrator diagnosable in production is still a human design decision that no code generator makes automatically.

The auto-instrumentation ceiling is generous and rising. The manual instrumentation floor remains.

Instrumentation is where observability intent meets engineering execution. The gap between those two is still a team's responsibility to close.

## Key Points

- Framework coverage is not application coverage: custom tool layers, bespoke retrieval clients, and queue-based agent dispatch each require manual instrumentation beyond the auto-instrumentation ceiling.
- **The Context Propagation Contract** must be defined at design time; async dispatch and cross-process queues silently break parent-child relationships without explicit context handling.
- The version mismatch trap: incompatible SDK and convention versions produce spans with missing attributes that the backend drops without warning.
- Evaluation signals require deliberate annotation; no auto-instrumentation library emits relevance scores or quality metrics automatically.
- The Datadog MCP Server inverts the developer-to-platform relationship; the reasoning between telemetry query and code change is not itself a traced span.
