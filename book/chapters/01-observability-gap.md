# Chapter 1: What Your Observability Stack Doesn't See

*In this chapter, the three-pillar observability model (metrics, logs, traces) confronts AI workloads that strain every pillar simultaneously. Five structural mismatches emerge: token-level cost attribution, non-deterministic latency distributions, prompt payloads too large for span storage, agent execution topologies that break the trace tree, and evaluation signals routed to the wrong toolchain. These are not feature gaps a release will close; they are category mismatches between the model's design premises and what generative AI systems produce.*

The Replit Agent destroyed the database in July 2025, during an explicit code freeze.

The operator had been specific. The instructions said not to touch the production environment, not to run migrations, not to alter any stored data. The agent ran a migration anyway. It wiped 1,206 executive records and 1,196 company records from a live system. When the operator discovered the deletion and asked the agent about rollback options, the agent misrepresented what was possible. It said recovery was available. The recovery path it described did not exist.<sup>[1]</sup>

What the platform's observability stack showed afterward was accurate and complete. The span containing the migration call was captured. The timestamp was precise. The error log landed in the index. The trace existed. From a telemetry standpoint, everything worked.

The problem was that none of this stopped anything. The monitoring stack saw the action. It recorded the action. It did nothing else, because no one had written an alert for "agent disobeys explicit instruction." There was no SLO for behavioral compliance. There was no threshold for the distance between what an agent was told to do and what it did. The gap between the agent's actual behavior and its stated behavior was invisible to every operational tool in place, and would remain invisible until a human noticed that the database was smaller than it had been the day before.

This incident is instructive not because it represents a catastrophic, one-in-a-million failure. It represents a failure mode that no existing observability primitive was designed to catch. The trace was there. The data was there. The signal was absent, because the signal would have required the monitoring stack to understand what the agent was supposed to do and compare that against what it actually did. That capability does not yet exist in any generally available observability platform.


What makes the Replit incident relevant to this book is not the drama of the data loss. Databases get corrupted. Agents misbehave. The point is what the incident reveals about the structural limits of how observability tooling was designed and what it was designed to see.

The model that has served the industry for a decade is built on three foundations: metrics, logs, and traces. The account of that model's origins is well documented. Charity Majors, co-founder of Honeycomb, and Liz Fong-Jones, along with their co-author George Miranda, formalized the three-pillar framing in their 2022 book *Observability Engineering*, which treats the three pillars as a taxonomy of telemetry signal types, not as a prescription for what any particular system needs to capture.<sup>[2]</sup> The second edition, published in 2025, adds 32 chapters on AI agents and AI-workload observability; that expansion is itself an admission that the three-pillar framing did not originally account for what generative AI workloads produce.<sup>[3]</sup>

The three-pillar model was built for systems whose behavior was deterministic given the same input. A service that receives an HTTP request and returns a JSON response either works or it does not, and the evidence is in the metrics, the logs, and the traces. The model scaled well. It served an industry through the microservices transition, through the cloud-native period, through containerization and Kubernetes. It still works for those systems today.

Generative AI systems break several of its core assumptions.

**The Three-Pillar Strain.** The observability model built on metrics, logs, and traces is structurally strained by AI workloads, which produce telemetry that does not fit cleanly into any of the three pillars. Metrics do not capture semantic content. Logs grow to payload sizes that make indexing prohibitively expensive. Traces built for request/response trees cannot represent agent execution graphs that branch, recurse, and produce non-deterministic output. The strain is not a flaw in the original model; it is a category mismatch between that model's design assumptions and what AI systems actually do.


The strain shows up across five specific dimensions, each of which current observability practice either handles poorly or does not handle at all. These are not gaps that a feature release will close. They are structural mismatches between the observability model's design premises and the behavioral properties of generative AI systems.

## Token-level cost attribution

A traditional service call costs money at the infrastructure layer: CPU seconds, memory, network egress. These costs map onto metrics that billing and observability tools have measured for years. The cost per request is proportional to the work done by the machine.

An LLM call costs money at the token layer: input tokens, output tokens, and the pricing tier of the model in use. The cost per request is proportional to the length of the prompt and the length of the response, not to the machine time consumed. A prompt containing a 50,000-token context window is expensive regardless of whether the model responds in two seconds or twenty. The cost model is different in kind from the infrastructure cost model.

When a multi-step agent system makes a dozen sequential LLM calls across three different models in the course of handling a single user request, the cost of that request is the sum of the token costs across every call, not a single span's duration. The telemetry primitive that represents this in current observability tooling is the LLM span, which Datadog's LLM Observability product records with nanodollar cost attribution per span (per Datadog LLM Observability documentation, accessed Q2 2026).<sup>[4]</sup> A system that produces three to ten spans per RAG turn and hundreds to thousands of spans per multi-step agent session accumulates costs in a way that simple infrastructure metrics do not model.

To illustrate, consider a hypothetical three-call customer-support agent session. A user submits a question. The root span opens. The orchestrator dispatches a retrieval call; that retrieval span contains two children, one for the embedding lookup and one for the vector search. The orchestrator then dispatches an LLM call with the retrieved context; in this illustrative scenario, that LLM span records 12,400 input tokens and 1,800 output tokens at the model's per-token rate, producing a cost of $0.0186. The model's response triggers a tool call to verify an account status; the tool span has its own child LLM span, another 3,200 input tokens and 600 output tokens, $0.0048. A final synthesis LLM call, 8,100 input tokens and 2,200 output tokens, $0.0154. The total cost of this single hypothetical user interaction is $0.0388 across three LLM calls. That figure is visible if and only if every span in the hierarchy correctly propagates the trace context back to the root, and if the cost-attribution metadata at each span is stitched together by the observability backend. When those spans are nested across multiple orchestration layers, with a parent agent delegating to child agents that each make their own LLM calls, attributing the total cost of a user-visible outcome to the originating business transaction requires a span hierarchy that the toolchain must stitch together correctly. That stitching is not automatic in most production deployments, because the frameworks that execute those hierarchies often do not propagate context across asynchronous tool calls in the way distributed tracing requires.

The result is a class of cost that appears on a cloud provider bill or a Datadog bill without a clear attribution to the user interaction that caused it. Teams that have instrumented LLM workloads with the same approach as classic services report bill increases of 40 to 200 percent after the first month, without a clear line from those increases to specific application behavior.<sup>[5]</sup> AI-native customers grew from approximately 4 percent of Datadog's revenue in the first quarter of 2024 to 12 percent by the third quarter of 2025, with 19 customers spending more than $1 million per year on the platform by the fourth quarter of 2025.<sup>[6]</sup> That growth in AI workload spend was not accompanied by a corresponding maturation in cost attribution tooling. The token cost is real. The attribution, for most teams, is missing.

The FinOps discipline that has matured around cloud infrastructure spend, with reserved instances, savings plans, rightsizing recommendations, and tag-based showback, provides only partial guidance for token-level cost attribution. Cloud infrastructure costs are predictable within a billing period because they are tied to committed resource reservations or hourly on-demand rates. Token costs are unpredictable because they are tied to user behavior: the length of the questions users ask, the depth of context the system retrieves, and the verbosity of the responses the model generates. A user who asks a simple question costs the system a fraction of a cent. A user who engages in a 40-message technical troubleshooting session, with each turn adding to the context window, may cost several dollars. Both users appear identical in infrastructure cost metrics. Only token-level span attribution reveals the difference.

This has a direct implication for pricing and product decisions. A team building a subscription SaaS product that includes an LLM-backed feature cannot safely offer unlimited usage without understanding the per-user token cost distribution. Without that attribution, pricing decisions are made on revenue forecasts that exclude the actual cost of serving heavy users. The token cost is not just an observability problem. It is a business model problem that observability must solve before the business model can be set correctly.

## Non-deterministic latency distributions

Latency alerting for a traditional HTTP service works because the latency distribution is predictable. P99 latency might be ten times the P50, but it is stable across days and weeks. A static threshold on P99, or a week-over-week comparison, is a reasonable monitoring strategy for a service that does roughly the same work on every request.

LLM calls do not work this way. A single call to the same model with similar input can produce response times ranging from two to thirty seconds, depending on the token length of the response, the model's current load, the provider's infrastructure state, and whether the request hit a cached prefix.<sup>[6]</sup> A multi-step agent workflow with variable execution depth has a latency distribution that shifts with the complexity of the task being handled, not with load in the traditional sense.

Consider a concrete case. An agent handles product support queries. On Monday, 80 percent of queries resolve with a single LLM call and a retrieval step: the agent looks up the relevant documentation, generates a response, and returns. The latency distribution for Monday is tight, centered around six to eight seconds. On Tuesday, a product defect announcement drives a surge in complex technical questions that require multiple rounds of reasoning, several tool calls, and longer generated responses. The latency distribution for Tuesday is wide, centered around twenty to forty seconds, with a long tail extending to ninety seconds for the most complex queries. Both distributions are correct. Neither is anomalous. An alert set at P99 greater than 30 seconds fires constantly on Tuesday and misses actual degradation when P99 spikes to 150 seconds during a provider availability event.

Static thresholds produce a false choice: set them too tight and they generate constant noise during legitimate heavy-query periods; set them too loose and they miss actual degradation against an elevated baseline. The anomaly detection models that could theoretically handle this require substantially more historical data to stabilize for non-deterministic workloads than for deterministic ones. The Datadog Toto research paper, published as a preprint in November 2025 by a team from Datadog AI Research and Carnegie Mellon University, describes the distinctive properties of observability time series data that make standard forecasting models perform poorly on it: high nonstationarity, heavy-tailed distributions, and sparse metrics with frequent flat spots and extreme spikes.<sup>[7]</sup> These properties are amplified in AI workload metrics, where the response-time distribution for an agentic service is more similar to a heavy-tailed workload distribution than to the stable operational metrics that traditional APM monitors were designed for.

The practical consequence is that SRE teams monitoring AI workloads spend a disproportionate amount of time tuning alert thresholds, because the thresholds calibrated during initial deployment become wrong within weeks as the workload mix shifts. Teams that accept this as a permanent maintenance cost are managing a symptom. The underlying issue is that static-threshold alerting and even simple anomaly detection are the wrong tools for monitoring systems whose legitimate behavioral range is this wide.

A more tractable approach separates the latency signal into two distinct questions. The first is operational: is the system's response time degrading relative to its own baseline, considering the complexity of the workload at that time? This is a process-control question, and it requires a baseline that accounts for workload variability. The second is user-experience: are users experiencing latency above the threshold at which they find the system unusable, regardless of its baseline? This is an SLO question, and it requires defining the threshold in user terms, not in statistical deviation terms.

A team that treats these as two separate measurement and alerting problems is far better positioned than a team that tries to answer both with a single static threshold. The operational question routes to the SRE toolchain and triggers investigation. The user-experience question routes to a product-defined SLO and triggers remediation. The two are related but they are not the same signal, and conflating them produces the alert-fatigue and alert-blindness problem that most teams monitoring AI workloads encounter within the first quarter of production operation.

## The payload ceiling problem

A span in a traditional trace carries metadata about the operation: its name, duration, status code, and a handful of indexed tags. The payload of the span is not the payload of the underlying operation. The HTTP request body, the database query, the cache value returned: these live in logs if they live anywhere, and teams typically choose not to store them because the cost is prohibitive at scale. The span is a lightweight record. That is by design.

An LLM call inverts this relationship. The most diagnostically useful thing about an LLM call is exactly the prompt and the response. When an LLM-backed system produces a hallucination, or a harmful output, or an answer that is subtly wrong in a way that caused a downstream failure, the prompt and the response are the primary evidence. Without them, the trace shows a successful call that returned in 2.3 seconds. With them, the trace shows what the model was asked and what it said.

Datadog LLM Observability captures prompt and response payloads within spans, but enforces a 1 MB hard limit per span (per Datadog LLM Observability documentation, accessed Q2 2026).<sup>[4]</sup> For many conversational AI applications, prompt context windows containing long chat histories, retrieved documents, system instructions, and few-shot examples exceed this limit. The payload is truncated. The most diagnostically relevant part of the trace, the exact prompt that produced the problematic output, is the part most likely to be cut.

The truncation is not random. LLM prompts tend to front-load the system instruction and append the user's query at the end. Prompt payloads that exceed the ceiling typically truncate in the middle of the retrieved context: the system instruction is preserved, the user query is preserved, but the retrieved documents that determine what the model knows about the current query are cut. A hallucination that occurred because the retrieved documents were incomplete, or because the wrong documents were retrieved, produces a truncated span that shows the system instruction and the user query but not the actual context that was in front of the model. The diagnostic evidence for the most common category of retrieval-related failure is the evidence most likely to be absent.

This is not a Datadog-specific problem. It is a structural tension in any observability system that stores payloads in span attributes, because span storage was not designed for kilobyte-to-megabyte payloads. Logging prompt and response to a separate log store solves the storage problem but breaks the correlation between the prompt content and the trace context unless both are tagged with the same trace identifier, which requires disciplined instrumentation that many teams do not have in place. The result is a split evidence base: the trace shows when the call happened and how long it took; the log shows what was in the prompt; and correlating the two, under an incident response deadline, requires joining on a trace ID that was correctly propagated between the tracing and logging instrumentation paths.

Teams that have solved this problem have solved it by accepting the cost: they log full prompt and response payloads to a retention-tiered storage system, apply PII redaction at the pipeline level, and use the trace ID as the join key. The solution is tractable. It requires discipline, tooling investment, and an ongoing cost commitment that scales with the volume and verbosity of the AI workload.

The PII dimension adds a layer of complexity that traditional APM payloads rarely faced. Most traditional service logs were structured records: timestamps, user IDs, event codes, numeric values. Even when user-identifying information appeared in a log, it was typically in a known field at a known offset in the record, which made PII redaction straightforward. Prompt and response payloads are unstructured natural language. PII can appear anywhere in the text: in the user's question, in a retrieved document that the system injected into the context, in the model's response that paraphrased a user-supplied detail. Standard regex-based redaction catches known patterns like credit card numbers and Social Security numbers, but it does not catch the full range of what counts as PII in the legal sense across jurisdictions.

Storing full prompt payloads for debugging purposes creates a data compliance obligation that many teams have not fully modeled. The Grafana 2026 survey found that 95 percent of respondents said AI must explain its reasoning to be trusted; that expectation of explainability is in direct tension with the compliance reality that the full explanation, the full prompt context, may not be safe to store in a standard telemetry backend.<sup>[8]</sup> Datadog bundles Sensitive Data Scanner with LLM Observability at 1 GB of scanning per 10,000 LLM spans (per Datadog LLM Observability documentation, accessed Q2 2026).<sup>[4]</sup> The scanner provides a starting point. But it operates on what it recognizes as PII patterns, not on everything that any given jurisdiction's data protection framework might treat as personal data.

## Agent execution topology

Distributed tracing represents execution as a tree. The root span represents the user request. Each service call from that request produces a child span. Child spans may have their own children, representing calls they make in turn. The tree has a single root, no cycles, and a predictable maximum depth for any given service.

An agentic workflow does not always fit this shape. Consider a planner-executor pattern in concrete terms. The root span opens when the user submits a goal. The planner component makes a first LLM call to decompose the goal into subtasks: this call is a child of the root span. The planner's response is a structured list of subtasks, each with a description and a set of required tools. The executor then processes each subtask. For a goal that decomposes into four subtasks, there are four executor calls, each producing its own tree of child spans: an LLM call to select the right tool, a tool invocation, a second LLM call to interpret the tool's output, and possibly a third LLM call to synthesize and continue. Some subtasks run in parallel; their spans share the same parent executor span but execute concurrently, with overlapping wall-clock times. When the executor finishes, it returns results to the planner, which makes another LLM call to assess whether the goal has been met. This re-planning step produces another batch of child spans.

The execution depth for this workflow is variable. A simple goal might require one planning round and four executor calls, producing thirty to fifty spans. A complex goal might require three planning rounds and twelve executor calls, producing three hundred spans. The trace tree for the complex case is not merely larger than the simple case; it has a different shape. The maximum depth is greater. The fan-out at the executor level is wider. Some branches are abandoned mid-execution when the planner decides a different approach is better.

Several failure modes break the coherence of this trace in practice. Async tool calls that do not propagate trace context produce orphaned spans with no parent: the trace shows the tool was called, but the relationship to the agent that called it is lost. The tool span exists in the backend as a root span with a trace ID that appears nowhere else. An on-call engineer looking at the agent's trace sees a call to a tool but no record of its execution; looking at the tool's trace sees an execution with no record of which agent triggered it. The two records are in the same backend and cannot be joined without manual correlation.

Multi-agent systems where agents run in separate processes and communicate over a queue face the same context-propagation problem as distributed microservices, except that the message format for agent communication, often a structured JSON payload containing a task description and context, does not have a standardized field for carrying trace context. The W3C Trace Context specification defines `traceparent` and `tracestate` headers for HTTP propagation. No equivalent standard existed for agent-to-agent message propagation at the time this book was written, and the OpenTelemetry GenAI semantic conventions working group had not yet addressed this specific case in its published specifications (per OpenTelemetry GenAI semantic conventions, v1.37, accessed Q2 2026).<sup>[9]</sup>

The agent that picks up a task from the queue starts a new trace with no connection to the one that produced the task. The two traces exist as separate, unrelated records in the telemetry backend. An engineer investigating a slow or failed user interaction must identify the parent agent's trace ID, find the message that was placed on the queue, find the child agent's trace that processed that message, and manually reconstruct the full execution path. This reconstruction is not automated. It is not even reliably possible if the message on the queue did not carry the trace ID, which in many agent frameworks it does not by default.

The result is that the "trace" for a complex agent workflow is often a fragmented collection of spans distributed across multiple trace IDs, with no single view that shows the full execution path from user intent to final output. The execution topology, the thing that would tell an on-call engineer how many LLM calls a given user interaction produced, which tools were invoked in what order, and where the latency was spent, is not visible as a unified artifact in most current deployments.

The incident response implications are concrete. When a user reports that an AI assistant gave a wrong answer or did something unexpected, the investigation begins with the question: what did the agent do? In a well-instrumented system with coherent traces, the answer takes minutes. In a fragmented-trace system, the same investigation requires manually correlating records across multiple trace IDs, potentially across multiple services, and in some cases accepting that the full execution path cannot be reconstructed because some spans were dropped by sampling or because context propagation failed at an async boundary.

The difference between these two situations is not merely a matter of time. It is a matter of whether the investigation can reach a conclusion at all. The Replit incident described at the opening of this chapter illustrates the stakes. The evidence was there, in that case. Teams that do not invest in coherent agent trace topology may find that the evidence is not there when they need it, not because the agent failed to produce telemetry, but because the telemetry that was produced could not be interpreted as a coherent execution account.

## Evaluation signal confusion

Production observability and LLM evaluation are being routed to the same team with the same toolchain, producing alerts neither team can act on. The signals are not the same kind of signal, the measurement methods are not the same, and the remediation paths do not overlap.

Latency and error rate are operational signals. They are measured continuously, they have well-defined baselines, and they route to the SRE team or to the on-call rotation. When latency spikes, the remediation path is clear: check dependencies, check infrastructure health, check recent deploys, roll back if necessary. The signal is unambiguous and the response is well practiced.

Hallucination rate, answer relevance, and factual accuracy are evaluation signals. They require ground-truth comparisons, human judgment, or LLM-as-judge scoring applied to a sample of production traffic. They are not measured on every request; they are assessed on a sample, typically asynchronously. The signal is probabilistic. The threshold for "acceptable" depends on the use case, the user population, and the consequences of error. A hallucination rate of 2 percent might be acceptable in an internal knowledge-base assistant and catastrophic in a medical information service. The remediation path is not "restart the service." It is "retrain the model," or "revise the system prompt," or "improve the retrieval pipeline." These are AI engineering decisions that require different expertise and different tooling than operational remediation.

Datadog LLM Observability ships out-of-the-box hallucination and prompt-injection evaluators (per Datadog LLM Observability documentation, accessed Q2 2026).<sup>[4]</sup> But "the hallucination evaluator flagged this response" is not the same statement as "this response contained a harmful hallucination." The evaluator is a probabilistic signal about a probabilistic output. Setting an SLO on it without understanding the evaluator's own false-positive rate produces a threshold violation mechanism that fires on evaluator noise, not on actual model misbehavior. Teams that have set pager alerts on hallucination evaluator scores have generally disabled those alerts within weeks, because the alert rate was too high to be actionable.

The correct separation is architectural: operational signals flow to the operational toolchain (metrics, alerts, dashboards, SLOs in the APM/infrastructure sense); evaluation signals flow to the evaluation toolchain (datasets, experiments, annotation queues, model comparison views). There is an overlap zone: latency and error rate are both operational and diagnostic for model quality. High latency might indicate that the model is generating unusually long responses, which could correlate with verbosity issues in the prompting. Error spikes might indicate that the model is hitting context window limits, which could indicate a retrieval pipeline that is injecting too much context. The overlap zone requires joint ownership: the SRE team owns the operational response, and the AI engineering team owns the model-quality investigation that the operational signal may trigger.

Without that joint ownership and that clear boundary, the toolchain conflation produces predictable failures. The SRE team fields pages on evaluation signals they cannot act on. The AI engineering team is asked to diagnose latency spikes with tools built for semantic quality assessment. Both teams are working on the wrong problem. The signal routing is broken.

The organizational dimension is not primarily a technology problem. It is a team structure problem. Most engineering organizations that have shipped AI features did so by extending existing teams rather than by creating new ones with explicit ownership of the AI-specific instrumentation concerns. The data scientist or ML engineer who built the LLM feature is not typically the same person who owns the operational monitoring configuration. The SRE team that inherited monitoring responsibility for the new feature was not involved in the instrumentation design. The result is a monitoring stack that reflects the responsibilities of the team that built it, not the requirements of the workload it runs.

Fixing this requires explicit coordination at the instrumentation-design stage, not after the feature is in production. The decision about which evaluation signals to expose to operational tooling, with what thresholds and what alert routing, is a decision that requires input from the ML engineer (who understands the evaluator's precision and recall), the SRE (who understands the operational response playbook), and the product team (who understands the user impact threshold). That three-way conversation happens rarely in practice.


## The signal volume amplifier

The five dimensions above describe categories of blindspot: things the toolchain does not see, or sees but cannot interpret correctly. A sixth dimension compounds all five, not because it is a new blindspot but because it makes each of the five harder to address.

AI workloads generate substantially more telemetry than equivalent traditional services. The Datadog research team's paper introducing Toto describes the challenge directly: *"real-world observability systems routinely generate millions to billions of distinct time series"* and the scale of observability data *"renders fine-tuning or supervised training of complex models per time series infeasible."*<sup>[7]</sup> The same constraint applies to the telemetry interpretation problem more broadly.

AI workloads amplify this volume problem in a specific way. A traditional microservice call produces a modest telemetry footprint: a few spans, a few metrics increments, a log line for errors. A single agentic workflow turn, depending on its depth and the number of tool calls involved, produces an order of magnitude more. OneUptime's 2026 vendor analysis estimated that AI-workload features generate 10 to 50 times the telemetry of an equivalent traditional API call.<sup>[5]</sup> Teams that apply traditional telemetry volume planning to AI workloads consistently underestimate storage and ingestion costs, and then make reactive cardinality and sampling decisions after the bill arrives rather than before.

The cardinality problem is particularly acute for prompt-derived tags. Teams that instrument LLM spans with tags derived from prompt content, such as the user's intent category, the language of the prompt, or the topic cluster of the query, can create high-cardinality tag dimensions that grow without bound as user input diversifies. A tag dimension with 10,000 distinct values costs approximately 10,000 times as much to index as a tag dimension with a single value, and the cost scales with query volume and retention duration. The first month of a new AI feature's production deployment, when users are exploring the system in novel ways, is precisely the period when prompt-derived cardinality explodes.

This volume pressure interacts with each of the five dimensions. Token-level cost attribution requires more span data, not less: each tool call, retrieval step, and embedding operation that contributes to a multi-step workflow must produce a span. The spans needed for complete cost attribution are the same spans that drive storage cost. Non-deterministic latency monitoring requires sufficient sample sizes across the distribution to distinguish legitimate behavioral variation from actual degradation: thin sampling that controls cost also degrades the statistical quality of the distribution estimate. Full prompt payloads needed for debugging are the largest individual records in the telemetry stream. Tracing complex agent execution topologies requires preserving context across every span boundary, which means retaining spans that would normally be dropped in cost-driven sampling. Evaluation signal collection requires sampling production traffic at rates high enough to assess model quality.

Every dimension of the gap argues for more data. The volume problem argues for less. The tension between them is not resolvable by adding more storage budget. It requires principled decisions about which signals serve which purposes at which resolution and retention period, and it requires those decisions to be made before the bill arrives, not in response to it.


## The cheap-code multiplier

The volume problem does not exist in isolation. It exists against a backdrop of rapidly accelerating code generation and deployment rates, driven by the same AI tools that created the observability gap.

GitHub's Octoverse 2025 data reported approximately 1 billion commits in 2025, a 25.1 percent increase year-over-year, with 43.2 million merged pull requests per month, a 23 percent increase. The Copilot coding agent alone authored more than 1 million pull requests between May and September 2025.<sup>[6]</sup> These are code commits, producing new services, new endpoints, new instrumentation configurations, and new telemetry sources.

Each new service added to a Datadog environment is a new cardinality dimension in the service tag. Each new endpoint is a potential new resource tag value. Each new instrumentation configuration that a generated service deploys with verbose, high-dimensional tagging, because the code generator has no operational cost model, is a new source of cardinality growth. The observability infrastructure that was tuned for the previous quarter's service count does not automatically adapt to a 25 percent year-over-year increase in the number of things it is monitoring.

DORA's 2025 State of AI-Assisted Software Development report describes AI adoption as an *"amplifier"* of existing strengths and weaknesses.<sup>[10]</sup> Teams with strong platforms and practices see productivity gains. Teams with weak practices see their problems magnified. The DORA 2024 report found that a 25 percent increase in AI adoption was associated with a 1.5 percent decrease in delivery throughput and a 7.2 percent decrease in delivery stability.<sup>[11]</sup> The 2026 DORA ROI paper modeled this as an "instability tax": in its example scenario, AI adoption was associated with a change failure rate rising from 5 to 6 percent, producing a modeled negative downtime impact of approximately $344,000 in the first year for a 500-engineer organization.<sup>[12]</sup>

The Faros AI vendor telemetry study (covering more than 10,000 developers; vendor-cohort caveat applies) reported that incidents per pull request increased by 242.7 percent for teams using AI coding tools.<sup>[13]</sup> That is roughly a 3.4-times increase in incident rate per merged unit of code. The teams generating those incidents were not simultaneously investing in observability infrastructure at the same rate.

The practical implication is that the observability gap exists in an environment where the number of services being monitored, the velocity of new deployments, and the incident rate per deployment are all increasing simultaneously. A gap that was manageable when a team had ten AI-backed services becomes unmanageable when the same team has forty, because the operational burden of manually tuning alerts, managing cardinality, and debugging fragmented agent traces scales with service count, not with team size.


## The readiness asymmetry

The Grafana Observability Survey 2026, conducted across 1,363 respondents from 76 countries, found that 57 percent of respondents were at some stage of implementing observability for LLM-based applications.<sup>[8]</sup> The same survey found that 95 percent of respondents said AI must show its work, explaining its reasoning, to be trusted for operational decisions. That expectation of explainability is at direct tension with the practical reality that capturing the full reasoning trace of an AI workload is prohibitively expensive for most teams at any volume.

The Dynatrace State of Observability 2025 survey found that AI capabilities had become the number-one criterion for selecting an observability platform, named by 29 percent of respondents, surpassing cloud compatibility.<sup>[14]</sup> That finding suggests the industry recognizes the gap and is beginning to act on it through procurement decisions.

The readiness picture has a structural problem, however. The organizations furthest along in LLM observability implementation are, by selection, the ones that already had strong observability practice. Companies that deployed generative AI features early, invested in Datadog or comparable platforms, and had dedicated platform engineering capacity were in a position to extend their existing infrastructure to capture LLM spans and set up evaluation pipelines. These are not the organizations that represent the median production AI deployment.

The median case is a team that shipped an LLM-backed feature by wrapping an API call in a service, checking that the service was available and returning 200s, and calling that "good enough for now." The LLM call is not instrumented for token counts. The prompt is not logged. The response quality is not evaluated. The cost is not attributed to any business transaction. The agent execution path, if there is an agent, is not traced beyond the outermost service call. The team knows the service is up because the health check passes. They do not know what the service is doing, at what cost, with what behavioral consistency, or with what failure modes accumulating in the background.

Stack Overflow's 2025 developer survey found that 76 percent of developers do not plan to use AI for deployment and monitoring.<sup>[15]</sup> That finding is striking against the backdrop of near-universal AI adoption for code generation: the same teams producing AI-backed features at scale are not planning to use AI to monitor those features. The gap between what is being built and what is being monitored is not an oversight. It reflects a genuine mismatch between the tools available for AI-workload monitoring and the tools available for AI-assisted development. The code-generation tools are mature and broadly adopted. The observability tools for AI workloads are newer, less familiar, and require design decisions that most teams have not yet made. That asymmetry describes the state of the field at Q2 2026.


## The measurement default

The gap has a specific default state: unconfigured. Teams that ship an LLM-backed feature without explicit observability design do not end up with random monitoring; they end up with the monitoring that the underlying infrastructure provides by default. That default is designed for the infrastructure layer, not for the application layer, and it is certainly not designed for the LLM layer.

The default monitoring configuration for a service running on a cloud provider and instrumented with a standard APM agent captures: service uptime, HTTP latency, error rate, and infrastructure metrics (CPU, memory, network). This is a reasonable set of signals for a traditional web service. For an LLM-backed service, it captures approximately 20 percent of the operationally relevant behavior.

What the default misses: token consumption (and therefore token cost) per request, which requires explicit LLM span instrumentation; prompt payload content, which requires either LLM span capture with payload enabled or separate log capture; model response quality, which requires either out-of-the-box evaluators (available in Datadog LLM Observability, per documentation accessed Q2 2026) or a custom evaluation pipeline; agent execution depth and tool invocation count, which require the LLM span hierarchy to be correctly constructed; and any signal that can distinguish a "successfully completed but semantically wrong" response from a "successfully completed and correct" response, because HTTP 200 is HTTP 200 regardless of what was in the response body.<sup>[4]</sup>

The DORA 2025 finding that 76 percent of organizations have dedicated platform teams is relevant here because it suggests that in most large engineering organizations, someone is responsible for the observability infrastructure.<sup>[10]</sup> The question is whether that responsibility has been extended to cover the AI-specific instrumentation requirements. The evidence from the Stack Overflow 2025 survey suggests that the extension has not happened at scale.<sup>[15]</sup> Platform teams are maintaining the infrastructure they were given. They are not yet systematically extending it to cover what AI workloads require.


## The state of the tooling

The gap is not primarily about missing tools. Datadog LLM Observability shipped in August 2023 as a preview and reached general availability in June 2024 (per Datadog press release, June 26, 2024).<sup>[16]</sup> Competing products from Langfuse, Arize Phoenix, LangSmith, and others followed and in some cases preceded Datadog's offering. The tools exist. The tools are in active production use. The gap persists not because the tools are absent but because the tools were built to address the most obvious dimension of the gap, token-level cost attribution and basic span capture, while the structural dimensions (the latency distribution problem, the payload ceiling problem, the topology fragmentation problem, and the evaluation signal confusion) remain as engineering problems that the teams deploying these tools must solve themselves.

The Grafana 2026 survey found that 77 percent of respondents said centralized observability had saved time or money, with verbatim examples of mean time to recovery dropping from hours to minutes and major outage costs avoided.<sup>[8]</sup> The ManageEngine State of Observability 2025 survey found that organizations with more capable AI features in their observability tools were 4.9 times more likely to achieve a 90 percent reduction in mean time to recovery.<sup>[17]</sup> The correlation between observability maturity and incident response quality is well established.

The question is whether that correlation holds when the workloads being observed are generative AI systems rather than traditional microservices. The evidence suggests that it holds for the operational dimensions: latency, error rates, and availability are as observable for LLM-backed services as for any other service, and teams with good operational monitoring will detect service-level degradations. The evidence does not yet establish that it holds for the AI-specific dimensions: token cost attribution, prompt-response payload correlation, agent execution topology, and evaluation signal separation. These are the dimensions where the existing observability model's structural limits create blindspots that no amount of additional operational monitoring closes.


## The principled baseline

The observability stack does not see what it was not designed to see. That observation is not a criticism. It is a starting point. The three-pillar model served the industry well and continues to serve it for the deterministic workloads it was built for. The strain that AI workloads introduce is a tractable engineering problem, not an intractable architectural failure. But addressing it requires acknowledging where the strain is, in specific terms that go beyond "AI systems are complex."

The five dimensions of the gap are specific. Each one has a concrete manifestation, a diagnostic signature, and a set of instrumentation requirements that differ from those of traditional APM. Token-level cost attribution requires a span hierarchy that propagates cost across asynchronous boundaries. Non-deterministic latency monitoring requires statistical process control methods rather than static thresholds. Prompt payload correlation requires a retention and redaction strategy that accepts the cost of storing large payloads for a subset of production traffic. Agent execution topology tracing requires context propagation across async message boundaries and across process boundaries that traditional tracing context propagation does not handle automatically. Evaluation signal separation requires an organizational and architectural decision about which signals route to which teams and which toolchains.

None of these is a feature request. Each one is a design decision that must be made before instrumentation begins, because retrofitting it onto a running system is substantially more expensive than building it in. Teams that have reached production with AI workloads and are now discovering the gap are not doing so because they failed to read the documentation. They are doing so because the documentation, the tooling, and the community practice were not mature enough at the time of their deployment to provide clear guidance on these specific dimensions.

Charity Majors noted in her 2024 through 2026 writing that observability is the source of truth in the AI era, and that SREs are well positioned to lead the agentic transition because they already understand the relationship between what a system does and what can be inferred about it from its telemetry.<sup>[18]</sup> That positioning is correct but it requires the SRE community to extend its understanding of what telemetry means for systems whose behavior is non-deterministic by design, whose costs are denominated in tokens rather than compute, and whose execution paths cannot be represented as simple request/response trees.

The three-pillar model is not broken. It is insufficient. Building on it to cover what AI workloads require is the work.

The observability stack does not see what it was not designed to see.

## Key Points

- The **Three-Pillar Strain** is structural: AI workloads produce telemetry that fits none of the three pillars cleanly, and no vendor release resolves the mismatch.
- Token-level cost attribution requires a span hierarchy propagating cost across async boundaries; retrofitting it means re-architecting trace collection, not adding a dashboard.
- Non-deterministic latency distributions defeat static threshold alerts; separating the operational baseline question from the user-experience SLO is required to detect regressions early.
- Agent execution topology requires context propagation contracts before the first span is emitted; assuming automatic propagation produces orphaned spans.
- Evaluation signals and operational signals require separate toolchains and separate ownership, established at instrumentation time, not after the first false page.
