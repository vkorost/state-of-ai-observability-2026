## Chapter 8. Instrumentation in Practice

The engineering team at a fintech company had spent three weeks building an agentic document-review workflow. Their stack was deliberately modern: a LangGraph orchestrator dispatching to an Anthropic Claude model for extraction, a retrieval step pulling from a Pinecone vector store, and a tool layer writing structured results back to a Postgres database. They had instrumented the application with OpenTelemetry from the start. They were confident the observability was sound.

On the day they pushed to staging, the traces in their backend showed a single span. One span. The orchestrator call. Everything beneath it was invisible. The LangGraph graph traversal, the Claude message turns, the retrieval steps, the database writes: none of it had emitted a span. The team had instrumented at the application boundary but not at the AI library level, because the OpenTelemetry libraries they were using predated the GenAI semantic conventions and had no understanding of what an LLM call was. The trace was technically correct. It was also operationally useless: a single root span with a duration of forty-three seconds and no children to explain where those seconds had gone.

They spent two days retrofitting instrumentation. One developer handled the Anthropic SDK wrapping. Another added the LangGraph hooks. A third sorted out context propagation across the retrieval step, which was running in a separate async task and had broken the parent span association. By the end, they had what they should have had on day one: a trace with typed spans for each workflow stage, token counts on the LLM spans, latency breakdowns for retrieval, and a cost estimate on the root span. The delay cost them a week of the staging cycle.

The story is instructive less for its drama than for what it reveals about the instrumentation landscape at Q2 2026. The conventions exist. The auto-instrumentation libraries exist. The mapping from library calls to typed spans exists in documented form. But the path between "deploy LLM application" and "get a useful trace" still has gaps, ambiguities, and version-dependency requirements that are not always visible until a team encounters them in production.

---

What makes that team's experience relevant to this chapter is what it exposes about the instrumentation layer as a distinct engineering problem. **The Datadog AI Surface**, introduced in Chapter 7 as the portfolio of capabilities built between August 2023 and Q2 2026, is only useful to the degree that data reaches it. The span taxonomy, the hallucination evaluators, the AI Agents Console: none of these features produce value if the application's instrumentation layer does not emit spans in the shape those features expect. Instrumentation is the prerequisite the product surface depends on, and in Q2 2026 that prerequisite is still non-trivial to satisfy correctly.

The chapter examines that prerequisite along three axes. First, what the OpenTelemetry GenAI semantic conventions actually specify at v1.37, how they arrived there, and where they remain contested. Second, what Datadog's auto-instrumentation libraries cover and where the coverage boundary sits. Third, what the Datadog MCP Server does to the relationship between developer tooling and the observability platform, and what governance question that change introduces.

---

**Auto-Instrumentation Coverage.** The set of AI frameworks and providers for which Datadog's ddtrace library and the OpenTelemetry GenAI semantic conventions provide zero-code or low-code span capture, as distinct from the set that requires manual instrumentation. The boundary of this coverage is not stable: it moves with each library release, each new framework adoption cycle, and each update to the GenAI semantic conventions. Understanding where the boundary sits at Q2 2026 requires separating two distinct paths: the ddtrace auto-instrumentation path for teams that have installed the Datadog agent, and the native OTel ingestion path for teams that have not.

### The OpenTelemetry GenAI Semantic Conventions at v1.37

The OpenTelemetry community has been working on a standard vocabulary for generative AI telemetry since 2023, and the effort has been productive, contested, and structurally complicated in roughly equal measure. Version 1.37 of the semantic conventions, which serves as the reference point for Q2 2026 implementations, establishes a unified framework for normalizing the operational footprint of LLM transactions, agentic orchestrations, and external tool interactions (Gemini report, "In-Depth Analysis of OpenTelemetry Generative AI Semantic Conventions," Q2 2026).

The convention's most important structural decision is its use of span attributes to classify operations. The required `gen_ai.operation.name` attribute identifies the transaction type: `chat`, `generate_content`, `text_completion`, `embeddings`, `create_agent`, `invoke_agent`, `invoke_workflow`, or `execute_tool`. The required `gen_ai.provider.name` attribute identifies the underlying model provider, replacing the earlier `gen_ai.system` namespace. Together, these two attributes create the classification layer that backends use to distinguish a standard LLM call from a tool invocation from an agent workflow step. The `gen_ai.conversation.id` attribute provides thread-level correlation for multi-turn interactions. The `gen_ai.request.model` and `gen_ai.response.model` attributes capture what the application asked for and what the provider actually delivered, which differ in practice when providers route requests across model versions (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026).

The token accounting attributes are finalized and production-stable at v1.37. The `gen_ai.usage.input_tokens` and `gen_ai.usage.output_tokens` attributes replace the earlier `prompt_tokens` and `completion_tokens` names. Critically, v1.37 adds `gen_ai.usage.reasoning.output_tokens`, which tracks tokens consumed by internal chain-of-thought reasoning in models that expose this data. It also adds `gen_ai.usage.cache_read.input_tokens` and `gen_ai.usage.cache_creation.input_tokens`, which matter for providers like Anthropic that charge differently for cached input: a team not capturing these fields will see prompt-caching cost reductions appear as unexplained billing drops rather than as a measured optimization outcome.

The operational metrics are also finalized at v1.37. The `gen_ai.client.token.usage` counter, the `gen_ai.client.operation.duration` histogram, and the `gen_ai.client.operation.time_to_first_chunk` histogram give platform teams the basis for SLO definitions against latency and token throughput. The server-side equivalents (`gen_ai.server.request.duration`, `gen_ai.server.time_to_first_token`, `gen_ai.server.time_per_output_token`) allow teams running self-hosted inference to measure performance at the serving layer rather than the client layer. These metrics enable the operational SLO discipline that **the Eval/Ops Split**, introduced in Chapter 5 as the principled separation between production health questions and model quality questions, depends on for the operational side of that boundary.

The providers explicitly documented within the v1.37 specification include OpenAI, Anthropic, AWS Bedrock, Azure AI Inference, Azure AI OpenAI, Google Vertex AI, Google Gemini, DeepSeek, Perplexity, Cohere, and xAI. This list covers the major commercial APIs as of Q2 2026 (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026).

One structural change at v1.37 deserves specific attention because it affects how teams navigate the ecosystem. The GenAI semantic conventions have been extracted from the core OpenTelemetry repository into a dedicated repository: `open-telemetry/semantic-conventions-genai`. This migration is an explicit architectural decision to bypass the review queue bottlenecks of the primary project and allow the GenAI working group to iterate more rapidly (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026). The practical consequence for teams is dual-repository navigation: issues and pull requests that were in the main repository are being migrated to the new workspace. Teams following the GenAI spec development need to track `open-telemetry/semantic-conventions-genai` rather than the parent repository. The split creates a temporary alignment challenge; it is the right structural call given the pace of generative AI development relative to the rest of the OpenTelemetry specification surface.

The opt-in mechanism for v1.37 compatibility is explicit. An application must set the environment variable `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` to instruct compliant instrumentation libraries to emit the v1.37 convention set rather than the pre-v1.36 layout. This is a breaking-change guard: production systems that have not set this variable will emit the older attribute names, and backends that expect v1.37 names will see missing or misnamed fields. Teams adopting the GenAI conventions for the first time should set this variable from the outset. Teams migrating existing instrumented applications should treat the migration as a schema change and verify that downstream queries, dashboards, and alerts account for the attribute renaming before enabling it.

The convention's SIG roadmap for the period immediately following v1.37 focuses on moving core span and event attributes out of experimental tracking and into a finalized stable release in the `semantic-conventions-genai` repository. The practical target is to deprecate the `OTEL_SEMCONV_STABILITY_OPT_IN` opt-in mechanism entirely, so that compliant libraries emit stable v1.37+ attributes by default without any environment variable configuration. This stabilization milestone matters for enterprise adoption: most enterprise security teams resist deploying applications with environment variables labeled `experimental` in CI/CD pipelines, regardless of the stability of the underlying attributes the variable activates. The gap between "experimental" as a label and "stable" as a deployment posture is one of the reasons adoption of the GenAI conventions lags behind their actual technical readiness (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026).

#### What the v1.37 Conventions Leave Open

Two structural problems remain actively contested in the GenAI working group at Q2 2026, and both affect agentic workloads specifically.

The first is the question of span topology for autonomous, multi-step, and multi-agent workflows. The conventions handle basic operations well. They do not yet handle the recursive depth, branching tool invocations, and session-boundary questions that emerge when multiple agents interact across a single user-initiated task. Meta-issue `#2664` in the working group catalogs six relational structures for agent topologies, but the mathematical modeling of these structures within standard distributed tracing limits is still under discussion. The core difficulty is that standard tracing was designed for request-response trees; agentic workflows produce graphs, and graphs do not always serialize cleanly into parent-child span hierarchies. **The Span Topology Problem**, introduced in Chapter 4 as the structural difficulty of representing agentic execution as a parent-child span hierarchy, is the principle-level statement of exactly this contestation (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026). The conventions are moving toward resolution via new attribute sub-families (`gen_ai.task.*`, `gen_ai.action.*`, `gen_ai.agent.skills.*`) but these are in-progress, not stable.

The second is payload handling. The decision to isolate prompt and completion content into separate OTel Events, rather than including them directly in span attributes, remains contested. The rationale is sound: embedding large text payloads directly in span attributes degrades tracing infrastructure performance. The concern is that separating content into events removes spatial and temporal locality from the primary trace, which complicates backend processing and sampling. Issue `#2010` in the working group documents an ongoing push by several observability backend vendors to reconsider this choice. Datadog's own resolution, the per-span payload extraction described in Chapter 7 as **the 1 MB Payload Ceiling**, is a proprietary adaptation to this unresolved convention question: Datadog stores prompt and completion content on the span itself rather than in separate events, subject to the payload size limit.

#### The MCP Protocol Extension

The integration of the Model Context Protocol into the OpenTelemetry GenAI conventions is a recent and significant addition. MCP operates over JSON-RPC layers to establish standard communication channels between models, development tools, and external data sources. Traditional HTTP or generic RPC telemetry tracking fails to capture the nested interactions across a persistent MCP transport connection, because multiple multiplexed JSON-RPC messages may traverse a single session. The GenAI convention addresses this by introducing dedicated metrics: `mcp.client.operation.duration` and `mcp.server.operation.duration` as histograms measuring operational and server execution latency respectively, and `mcp.client.session.duration` and `mcp.server.session.duration` for tracking connection lifespan (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026). The convention enforces that tool calls executed over MCP use `gen_ai.operation.name = "execute_tool"`, aligning them with the standard `execute_tool` span kind, and requires propagation of trace context within the JSON-RPC request dictionary. The practical implication is that MCP-mediated tool calls are traceable with the same span structures as directly-invoked tools, provided both the MCP client and server are instrumented.

A team that runs `ddtrace-run python app.py` with `DD_LLMOBS_ENABLED=1` set in the environment will receive auto-instrumented spans for every supported framework the application uses, without any additional configuration. The fintech team in the opening case would have gotten their LangGraph and Anthropic spans automatically on this path.

Teams using Webpack must explicitly exclude integration modules from bundling (Datadog LLM Observability documentation, accessed Q2 2026).

#### What Falls Outside Auto-Instrumentation

The auto-instrumentation boundary is meaningful. Several frameworks and use patterns require manual instrumentation as of Q2 2026.

LangChain is not listed in the auto-instrumentation documentation independently from LangGraph. Teams using LangChain's lower-level chain and agent APIs directly, rather than through LangGraph's graph execution model, should verify whether their specific call patterns are covered or whether they need manual decorator wrapping.

Microsoft AutoGen does not appear in the published auto-instrumentation list at Q2 2026 (Claude research report, "Datadog in the generative AI era," Q2 2026; listed as uncertain). Teams using AutoGen for multi-agent coordination must instrument manually using the `@workflow`, `@agent`, `@task`, and `@tool` decorators in Python, or the equivalent `llmobs.trace({kind:'...'})` calls in Node.js, combined with `LLMObs.annotate(input_data, output_data, metadata, metrics, tags)` to attach typed attributes to each span. This is not burdensome in absolute terms but it is meaningfully different from zero-configuration: the team must understand the span taxonomy, apply decorators at the correct abstraction boundaries, and handle context propagation across asynchronous execution paths.

LlamaIndex falls into the same category at Q2 2026. The auto-instrumentation list does not include a LlamaIndex integration. Teams using LlamaIndex for RAG pipeline construction can instrument via the OpenTelemetry path if LlamaIndex's OTel integration emits v1.37-compatible spans, or via manual decorator wrapping for the Datadog-native path.

Strands Agents, the AWS-developed agent framework, requires explicit opt-in via `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` because Strands emits older OpenTelemetry GenAI semantic convention fields by default. Without this variable set, the spans Strands emits will not map correctly to the v1.37 attribute names that Datadog's native OTel ingestion expects (Claude research report, "Datadog in the generative AI era," Q2 2026).

The LangChain and LangGraph distinction is worth resolving precisely, because both frameworks are widely used and the instrumentation path differs. LangGraph instruments `CompiledGraph.invoke`, `CompiledGraph.stream`, `Pregel.invoke`, and `Pregel.stream` through both synchronous and asynchronous variants. LangGraph also captures CrewAI memory state (short-term and long-term) and LangGraph's own graph state within the trace. The `@workflow` span wraps the top-level graph invocation; beneath it, each node execution is captured as an `agent` or `task` span; each LLM call from within a node is captured as an `llm` span; each tool call from within a node is captured as a `tool` span. This hierarchy is enforced at SDK level, which is the resolution that **the Span Topology Problem** requires: the parent-child relationships are built at emission time by the SDK's context propagation, not reconstructed by the backend from trace IDs and timestamps (Datadog LLM Observability documentation, accessed Q2 2026).

#### The OTel Native Ingestion Path

For teams that are not using ddtrace, Datadog provides a native OTel ingestion path. Applications instrumented with the OpenTelemetry SDK, emitting v1.37 GenAI semantic convention spans via OTLP, can route directly to the Datadog backend without installing the ddtrace agent. This path covers the same provider list that the convention specifies and handles automatic mapping between OTel field names and Datadog's native schema (Datadog LLM Observability documentation, accessed Q2 2026).

The schema mapping is defined and documented. When a span with `gen_ai.operation.name = "chat"` arrives via OTLP, Datadog's intake engine maps it to an `llm` span kind in the LLM Observability schema. The `gen_ai.provider.name` field maps to the model provider field. The `gen_ai.conversation.id` maps to `session_id` and populates the session tracking interface. Tool-specific attributes map as follows: `gen_ai.tool.name` maps to the span name, `gen_ai.tool.call.id` maps to `metadata.tool_id`, `gen_ai.tool.call.arguments` maps to `input.value`, and `gen_ai.tool.call.result` maps to `output.value` (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026).

For prompt and completion content extraction, the intake engine applies a priority hierarchy. It first checks for direct span attributes: `gen_ai.input.messages`, `gen_ai.output.messages`, and `gen_ai.system_instructions`. If these are absent, it falls back to parsing the associated OTel span event named `gen_ai.client.inference.operation.details`. For embedding spans, `gen_ai.input.messages` maps to `meta.input.documents`, and the output value is set to the static string `[N embedding(s) returned]`.

Three practical limitations of the native OTLP path require explicit attention. The first is tag truncation: any `gen_ai.*` attribute not explicitly defined in Datadog's mapping table falls back to a generic span tag, and Datadog enforces a 256-character maximum per tag value. Custom metadata arrays or long unstructured parameters will be permanently truncated at ingestion. The second is attribute filtering: non-mapped custom attributes are dropped entirely, which means application context tags that do not match the standard specification cannot be carried through the OTel path. The third is access control: OTel traces arriving via OTLP can simultaneously appear in Datadog's general APM Trace Explorer and in LLM Observability. Teams with PII-sensitive prompt content must configure a Restricted Dataset policy in APM to prevent prompt payloads from being visible to engineers who have APM access but should not have access to the conversation content (Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026).

There is also an ID format mismatch between OTel and Datadog's external evaluations API. OTel manages trace and span identifiers as hexadecimal strings. Datadog's external evaluations HTTP API requires numeric identifiers submitted as decimal strings. Engineers implementing custom evaluators against production OTel traces must convert identifiers using the equivalent of `int(hex_id, 16)` before submitting evaluation results. The issue is documented and straightforward to solve, but it is the kind of friction that causes evaluation pipelines to fail silently if the conversion step is omitted.

The OTel path also supports legacy schema compatibility. If an incoming span lacks the standard `gen_ai.operation.name` attribute, the intake engine inspects the legacy `llm.request.type` field used by older instrumentation frameworks such as OpenLLMetry. Chat and completion operations map to `llm`; embedding operations map to `embedding`; rerank and unknown operations fall back to `workflow`. This compatibility layer means teams migrating from early OpenLLMetry instrumentation do not need to re-instrument their applications immediately, but should plan the migration to v1.37 attributes to preserve full feature access.

### The MCP Telemetry Bridge

The Datadog MCP Server, which reached general availability at some point in the first half of 2026 (per the April 2026 "This Month in Datadog" announcement describing it as "now generally available"), introduces a different kind of instrumentation question (DASH talk, "This Month in Datadog: Datadog MCP Server, Experiments, Bits AI Security Analyst," April 29, 2026). The previous sections address instrumentation as the outbound flow: application code emitting spans that travel to the Datadog platform for storage and analysis. The MCP Server addresses the inbound flow in the opposite direction: an AI coding assistant querying the Datadog platform for live production telemetry, receiving that data as structured context, and reasoning about it alongside the application source code the assistant is already analyzing.

**The MCP Telemetry Bridge.** The Datadog MCP Server as an architectural pattern that inverts the normal relationship between the developer tool and the observability platform, allowing AI coding assistants to query live production telemetry rather than requiring engineers to switch contexts to a separate dashboard. The bridge makes observability data available as a tool the AI agent can call, rather than as a separate interface the human engineer must navigate, collapsing the distance between the code a developer is examining and the production evidence about how that code behaves.

The protocol mechanics are worth describing precisely because the MCP architecture is distinctive. The MCP specification, as defined in the 2024-11-05 version, describes a client-host-server architecture where each host can run multiple client instances (MCP Specification, Architecture, 2024-11-05). The host process acts as the container and coordinator: it creates and manages client instances, controls connection permissions, enforces security policies, and handles user authorization decisions. Each client maintains an isolated connection to one server. Each server exposes resources, tools, and prompts via MCP primitives. Communication across the connection is built on JSON-RPC, creating a stateful session protocol.

For the Datadog MCP Server, the host is the AI coding assistant (Claude Code, Cursor, Codex CLI, or Gemini CLI). The MCP client is embedded within the assistant. The Datadog MCP Server is the server process, running either locally or as a remote service, exposing Datadog's telemetry APIs as MCP tools. When the developer asks the assistant a question that requires production evidence, the assistant calls the relevant Datadog tool through the MCP client. The MCP server translates that tool call into a Datadog API request, authenticated via OAuth, and returns the telemetry data as structured content. The assistant incorporates that content into its reasoning context alongside the source code it is analyzing (DASH talk, "This Month in Datadog," April 29, 2026).

The tools exposed by the Datadog MCP Server cover Datadog's primary telemetry surfaces: logs, metrics, traces, monitors, dashboards, and notebook creation. The server automatically determines which API is appropriate for a given query based on the natural language structure of the prompt. A developer asking about error logs for a specific service will receive log search results without needing to specify the Datadog Logs API endpoint explicitly. A developer asking about host utilization will receive metrics query results (DASH talk, "This Month in Datadog," April 29, 2026).

The HIPAA-eligible status of the MCP Server is noteworthy. HIPAA eligibility requires specific data handling guarantees around access control, audit logging, and encryption in transit. The architecture achieves this by routing telemetry queries through OAuth authentication, which means that the AI coding assistant's access to production data is governed by the same Datadog RBAC policies that govern human engineer access. A developer who does not have access to production LLM span payloads in the Datadog UI will also not have access to those payloads through the MCP Server. The same Restricted Dataset policies that protect PII in the APM Trace Explorer apply to MCP queries against those datasets (Claude research report, "Datadog in the generative AI era," Q2 2026).

The notebook creation capability in the MCP Server is particularly relevant to production workflows. A developer who has identified a cost optimization opportunity through MCP-mediated telemetry queries can write the analysis directly to a Datadog Notebook, creating a shared artifact with the supporting graphs and evidence without leaving the development context. This closes a workflow gap that previously required an engineer to investigate in the terminal, switch to the Datadog UI to build a notebook, and then share the result separately.

#### What the MCP Telemetry Bridge Does Not Provide

The MCP Telemetry Bridge changes the access pattern for production observability data without changing the data itself. The AI assistant can query the same telemetry that the human engineer can query. It cannot query telemetry that has not been captured. A team that has not instrumented its LangGraph orchestrator will find that the MCP Server returns empty log results and missing traces when the assistant investigates an agent misbehavior, because the telemetry was never emitted. The instrumentation completeness problem described in the first half of this chapter is not resolved by the MCP bridge; the bridge provides a more accessible query interface against whatever data exists.

The governance gap is the more analytically interesting limitation. When an AI coding assistant calls the Datadog MCP Server to retrieve production telemetry, and then uses that telemetry as context for generating a code change or a PR fix, a human-authorization step is implied but not structurally enforced. The developer reviews the assistant's suggested change, but the review relies on the developer trusting that the assistant correctly interpreted the telemetry, correctly identified the root cause, and correctly proposed the fix. The chain from production observation to code modification now passes through an AI reasoning step that is not itself instrumented or auditable in the same way that the original span is. The assistant's reasoning over the telemetry data is not itself captured as a span in Datadog LLM Observability unless the team has instrumented the assistant's reasoning loop, which most teams have not.

This is a narrow but real instance of **the Span Topology Problem**, in a new context: the trace that explains how a production decision was made now has a gap in it at the AI-reasoning step. The production span shows what the service did. The code change shows what the developer decided to do. The MCP-mediated analysis that connected them is not, by default, captured anywhere. For post-incident review, that gap may matter.

The MCP specification itself addresses authorization at the protocol level. The 2024-11-05 architecture specifies that the host controls client connection permissions, enforces security policies, and handles user authorization decisions. The Datadog implementation honors this by requiring OAuth authentication, which ties MCP access to Datadog identity. But the specification does not define what happens to the AI reasoning that occurs between the telemetry query and the resulting action. That governance layer is outside the current MCP protocol scope and outside the current Datadog product scope at Q2 2026 (MCP Specification, Architecture, 2024-11-05).

### The Instrumentation Coverage Reality at Q2 2026

A team instrumenting a production AI application in Q2 2026 has three choices that are genuinely distinct in their tradeoffs.

The first choice is the ddtrace auto-instrumentation path. The setup requires installing ddtrace, setting four environment variables (`DD_LLMOBS_ENABLED`, `DD_LLMOBS_ML_APP`, `DD_API_KEY`, `DD_SITE`), and launching the application under `ddtrace-run` (Python) or with `NODE_OPTIONS="--import dd-trace/initialize.mjs"` (Node.js). For applications using OpenAI, Anthropic, Bedrock, LangGraph, CrewAI, LiteLLM, Google GenAI, Google ADK, OpenAI Agents SDK, Pydantic AI, Vertex AI, or the Vercel AI SDK, the result is zero-code span capture for all supported call patterns. The spans land in LLM Observability with typed attributes, cost estimates, token counts, and correlation to surrounding APM spans in the same trace ID. The limitation is SDK coupling: the application is instrumented via Datadog-specific library patching, and moving to a different observability backend requires removing or replacing that instrumentation.

The second choice is the OTel GenAI conventions path. The setup requires an OpenTelemetry SDK installation configured to emit spans with v1.37 GenAI attributes, the `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` environment variable, and an OTLP export endpoint pointing to the Datadog API. For applications using frameworks that natively emit OTel GenAI spans, or for teams using auto-instrumentation layers like Arize's OpenInference (supporting over 32 libraries) or Traceloop's OpenLLMetry (supporting over 34 libraries), this path provides broader framework coverage than the ddtrace path in some areas, while sacrificing tight integration with Datadog-specific features like App and API Protection, Continuous Profiler, and the full Ingestion Rules surface (Claude research report, "Datadog in the generative AI era," Q2 2026; Gemini report, "OpenTelemetry GenAI Semantic Conventions," Q2 2026). The benefit is portability: OTel-instrumented applications can switch backends without re-instrumenting.

The third choice is a hybrid. Teams using ddtrace for their service-level APM instrumentation but needing to capture AI spans from a framework not yet covered by ddtrace auto-instrumentation can use the OTel GenAI path for the AI spans and the ddtrace path for the surrounding services. Datadog's OTLP receiver in the agent, enabled via `DD_OTLP_CONFIG_RECEIVER_PROTOCOLS_GRPC_ENDPOINT` or the HTTP equivalent on Agent version 6.32 or later, accepts both ddtrace spans and OTel spans on the same agent instance. The combined trace, stitched together by the shared trace ID in W3C Trace Context headers, provides full-stack correlation without requiring all components to use the same instrumentation library (Claude research report, "Datadog in the generative AI era," Q2 2026).

The practical boundary between these paths is not purely technical. Organizational factors matter. Teams with an existing investment in OpenTelemetry instrumentation across their service mesh will prefer the OTel path to maintain instrumentation consistency. Teams that are starting from scratch with a Datadog subscription and no prior OTel investment will likely find the ddtrace auto-instrumentation path faster to initial coverage. Teams with frameworks that fall outside both auto-instrumentation catalogs, such as AutoGen or custom orchestration systems, will spend manual instrumentation effort regardless of which path they choose for the covered portion of their stack.

#### The Strands and Bedrock Agent Edge Cases

Two instrumentation cases warrant specific treatment because they represent the edge of what is documented and what is uncertain at Q2 2026.

Strands Agents, the AWS-developed agent framework that launched in May 2025, is listed in the Datadog auto-instrumentation coverage. Teams using Strands must set `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` because Strands emits GenAI spans using pre-v1.36 attribute names by default. Without the opt-in variable, the Datadog intake engine will not map Strands' span attributes correctly to LLM Observability schema fields. This is a version-dependency trap: the Strands application will produce traces that appear in APM but will lack the typed LLM Observability attributes, and the team will not receive token counts, cost estimates, or typed span kinds. The fix is a single environment variable, but discovering the need for it requires either reading the integration-specific notes in the documentation or encountering the missing-attribute symptom in production.

The Amazon Bedrock Agents integration has a different specificity requirement. The integration automatically instruments the `InvokeAgent` method, which captures the overall agent call. But intra-agent steps, the individual reasoning and tool-use turns within a Bedrock agent's execution, are not captured by default. To enable intra-agent step tracing, the team must set `enableTrace=True` in the `InvokeAgent` request parameters. This is a call-site configuration, not an SDK configuration: every `InvokeAgent` call that should emit step-level spans must include this parameter. Applications that call `InvokeAgent` without it will produce single-span traces for agent executions that may span dozens of internal steps, reproducing exactly the diagnostic blindness that was the failure mode in the chapter's opening case.

### Context Propagation: The Hidden Fault Line

The auto-instrumentation coverage matrix describes which libraries emit spans. It says nothing about whether those spans arrive at the backend organized as a coherent trace. That second problem is context propagation, and it is where more instrumented AI applications silently break than the framework coverage question.

Context propagation in distributed tracing is the mechanism by which a trace ID and a parent span ID are carried from one component to the next, so that all spans belonging to the same logical operation end up in the same trace tree. In a synchronous HTTP service, context propagation is automatic: the tracing library injects trace context into outgoing HTTP headers and extracts it from incoming headers, and the parent-child relationship is established without any explicit developer action. The W3C Trace Context standard, which Datadog uses as its carrier format, is supported by every major HTTP framework.

AI application architectures break this assumption in at least three common ways.

The first is asynchronous task dispatch. A LangGraph orchestrator invokes a retrieval step by dispatching an async Python coroutine. If the async coroutine is created with `asyncio.create_task()` rather than `asyncio.ensure_future()` with explicit context capture, the trace context from the parent span does not automatically propagate into the new task's execution context. The OpenTelemetry Python SDK uses Python's `contextvars.ContextVar` for context storage, which propagates correctly across `await` boundaries but not across thread boundaries or across tasks created without explicit context copying. The result is that the retrieval span is emitted with a fresh, unparented trace ID, and the two spans that should form a parent-child pair become two separate unrelated traces in the backend. This was one of the two days' worth of work the fintech team in the opening case spent resolving.

The resolution requires explicit context capture at task creation. In Python, the pattern is to capture the current context with `contextvars.copy_context()` and pass it as the `context` argument to `asyncio.get_event_loop().run_in_executor()` or to `asyncio.create_task()`. The ddtrace library handles this automatically for call patterns it instruments, but custom async dispatch code outside the instrumented path requires the developer to manage context explicitly.

The second is thread pool execution. A tool function that calls an external service via a `concurrent.futures.ThreadPoolExecutor` executes in a thread that does not inherit the calling thread's Python context. The trace context from the parent span, stored in a `ContextVar`, is not carried into the thread. The tool's span becomes an orphan. The fix is to use `ddtrace`'s `Pin` API or the OTel `Context.attach()` and `Context.detach()` functions to propagate context explicitly across the thread boundary. This is not obscure knowledge, but it is also not surfaced as a warning in the ddtrace getting-started documentation; teams encounter it when they notice orphaned tool spans in their trace view.

The third is cross-process agent-to-agent communication. In a multi-agent system where a parent agent dispatches work to a sub-agent via an HTTP call or a message queue, trace context propagation depends on the calling code injecting the trace context into the HTTP headers or the message payload, and the receiving code extracting it. If both agents are using ddtrace, the tracing library handles HTTP injection and extraction automatically. If the parent agent is using ddtrace and the sub-agent is using a different instrumentation library, the trace context format must be compatible. The W3C Trace Context format ensures compatibility across conforming implementations, but message-queue-based dispatch requires the calling code to inject context into a message attribute, and the receiving code to extract it from the same attribute. This is framework-specific and is not automatically handled by ddtrace for every queue provider (Claude research report, "Datadog in the generative AI era," Q2 2026).

The practical diagnostic for context propagation failures is the orphaned span pattern that **the Span Topology Problem**, introduced in Chapter 4, defines. When a team opens a trace in the LLM Observability trace view and sees the root `workflow` or `agent` span with fewer child spans than the application's execution should have produced, the likely cause is a context propagation failure at one of the three points described above. The investigation starts by checking whether the missing spans appear as separate, unparented traces in the Trace Explorer; if they do, the propagation boundary is identifiable.

The deeper implication is architectural. AI application instrumentation cannot be treated as a purely operational concern that is retrofitted after the application is built. Context propagation correctness depends on how the application dispatches work: whether it uses async coroutines, thread pools, or message queues; whether those dispatch mechanisms propagate Python context variables; and whether the receiving code on the other side of a process boundary extracts trace context from the carrier. Teams that make these decisions without considering their instrumentation implications will produce incomplete traces. Teams that consider instrumentation as an architectural requirement from the first design session will make dispatch choices that are instrumentable by construction.

### Sub-Patterns of Instrumentation Failure

Three sub-patterns appear repeatedly in the production instrumentation failures documented in the corpus and in publicly available Datadog engineering and support documentation. Each one is distinct enough to name.

**The Version Mismatch Trap.** The failure mode in which an instrumentation library, a framework, and a semantic convention version are each individually current but are mutually incompatible, causing spans to be emitted with incorrect or missing attributes that the backend silently drops or misclassifies. The trap closes when a team upgrades one component without upgrading the others.

The Version Mismatch Trap manifests in two directions. The first is the framework-ahead-of-convention case: a framework ships support for a new model provider before the GenAI semantic conventions have standardized the attribute names for that provider's specific response fields. The framework chooses attribute names that seem reasonable but that conflict with later-standardized names. Teams instrumenting with that framework version lock in the non-standard attribute names. When the convention stabilizes, the backend maps the standardized names but not the framework's earlier choice, and cost estimates or token counts silently disappear from the trace. The second direction is the convention-ahead-of-library case: the GenAI conventions update to v1.37, renaming `prompt_tokens` to `gen_ai.usage.input_tokens`, but the instrumentation library has not yet shipped a release that emits the new names. Teams running the updated convention opt-in variable get the new backend schema but receive data from the old attribute names, which the intake engine does not map. Both directions produce the same symptom: spans that appear in the trace view with missing token counts, cost estimates, or typed attributes.

The resolution discipline requires pinning all three components to a tested combination: the ddtrace version, the framework integration version, and the semantic convention opt-in state. Datadog's documentation for each integration lists the minimum ddtrace version required for that integration's GA support. Teams that pin to a tested combination and update the combination together as a unit, rather than updating individual components independently, avoid most Version Mismatch Trap failures.

**The Evaluation Span Gap.** The failure mode in which a team instruments the operational spans for an AI application (LLM calls, tool calls, retrieval steps) but does not instrument the evaluation signals that the same application produces in production, creating a trace that answers the latency question but not the quality question.

The Evaluation Span Gap is a direct instrumentation-layer consequence of **the Eval/Ops Split** introduced in Chapter 5. The operational spans and the evaluation signals require different instrumentation strategies. Operational spans are emitted by the framework integration automatically. Evaluation signals, such as hallucination detector scores, relevance scores from a retrieval evaluator, or ground-truth comparison results, are computed by application code and require explicit instrumentation. The Datadog LLM Observability SDK provides the `LLMObs.annotate()` method for attaching evaluation scores to existing spans as metadata, and the external evaluations HTTP API for attaching evaluations computed outside the ddtrace process. Both require deliberate implementation.

The gap appears most commonly in RAG pipelines. A team instruments the vector search step with a `retrieval` span and captures the documents returned. The `retrieval` span records latency and document count. It does not automatically record the relevance score of the returned documents unless the application explicitly reads that score from the vector database response and passes it to `LLMObs.annotate()`. A quality regression driven by declining retrieval relevance will be invisible in the operational trace, because the only number that would reveal it is the one the team did not annotate. The LLM call that follows will be slower and more likely to produce a poor response, but the trace will attribute the degradation to the LLM span rather than to the retrieval step that fed it bad context.

The resolution is to treat evaluation signal capture as a first-class instrumentation requirement at design time, not as a post-production enhancement. For each application component that produces a quality signal (relevance score, confidence estimate, evaluation result), the team designates who reads that signal from the component's output and how it is attached to the corresponding span. This is one to three lines of annotation code per component in practice, but it requires a deliberate design decision to ensure those lines exist.

**The Cost Attribution Blindspot.** The failure mode in which a team captures token counts at the span level but does not aggregate those counts into a cost attribution that connects to a business transaction, making the per-span cost visible but the per-user-request cost invisible.

This sub-pattern is the instrumentation-layer manifestation of the cost attribution problem that **the Observability Gap** introduced in Chapter 1 describes at the structural level. The auto-instrumentation libraries emit `gen_ai.usage.input_tokens` and `gen_ai.usage.output_tokens` on each LLM span, and Datadog's LLM Observability computes an estimated cost for each span from these token counts and the provider's published pricing. A team using the auto-instrumentation correctly will see per-span cost estimates in the trace view.

What the auto-instrumentation does not provide is the aggregation of those per-span costs into a cost attributed to the root user interaction. A multi-step agent workflow that makes twelve LLM calls across four models in the course of handling a single user request will produce twelve cost estimates, one per LLM span. The total cost of that user request is the sum of those twelve estimates, but that sum is not automatically computed and associated with the root `workflow` span. Teams that want to know the cost distribution across users, or the cost of their most expensive user requests, need to either implement that aggregation themselves using the `LLMObs.annotate()` method on the root span, or query the trace data with an aggregation expression in the Trace Explorer.

The resolution for the aggregation case is to add a post-processing step at the workflow level: after the workflow completes, sum the cost estimates from child spans and annotate the root span with the total. This requires reading the child span cost metadata from the completed trace, which is accessible via the ddtrace trace context after the child spans close. It is a ten-line implementation for a Python decorator applied to the root workflow function. Teams that do not implement this aggregation will be unable to answer the question "what did this user request cost," which is the unit of cost attribution that pricing decisions, cost governance, and per-customer showback require.

### What the Coverage Matrix Means for the Observability Gap

The auto-instrumentation coverage matrix answers the most immediate question a team instrumenting an AI application will ask: does my framework work out of the box? For most of the widely adopted frameworks at Q2 2026, the answer is yes on the ddtrace Python path. For the Node.js path, the coverage is somewhat narrower but covers the major provider SDKs. For the Java path, the coverage is narrower still, confined to the OpenAI integrations that reached GA with dd-trace-java v1.51.0.

But coverage for the framework does not mean coverage for the application. A team running OpenAI Agents SDK with a custom tool layer that calls internal microservices needs auto-instrumentation for the OpenAI Agents SDK spans and manual instrumentation for the custom tool spans. A team running LangGraph with a retrieval step that calls a vector database through a bespoke client library will get auto-instrumented LangGraph orchestration spans and will need to manually instrument the vector database calls if they want typed retrieval spans with relevance scores. A team building a multi-agent pipeline where agents communicate via message queues will get instrumented spans at each agent boundary but will need to handle context propagation across the queue explicitly.

Coverage, context propagation, and evaluation signal capture are three distinct instrumentation requirements that address three distinct questions: did something happen, are the somethings connected, and did the somethings produce good results. The auto-instrumentation layers, at their best, answer the first question automatically and provide the scaffolding for the other two. They do not answer all three by themselves.

**The Observability Gap**, introduced in Chapter 1 as the collective set of AI-era blindspots in current observability tooling, has a specific instrumentation-layer manifestation: the gap between what a framework emits by default and what a team needs to diagnose a quality or cost regression in production is almost always wider than it appears at the moment of initial instrumentation. The auto-instrumentation layers close the most common part of the gap. They do not close the part that is specific to the team's application architecture, the custom tool implementations, the bespoke retrieval logic, and the proprietary data transformations that sit between the framework boundaries.

The Datadog Agent evolution described in the May 2026 DASH talk on the Agent's transformation into an AI execution layer provides a useful frame for understanding the instrumentation problem at a longer time horizon (DASH talk, "The evolution of the Datadog Agent into an AI execution layer," May 5, 2026). The Agent began as a passive telemetry collector in 2010, then evolved through successive capability additions: Kubernetes context awareness, serverless extension support, edge processing for PII redaction, remote configuration for zero-restart policy updates, and fleet automation for at-scale agent management. The AI-era addition is the Private Action Runner, which is now embedded in Agent version 7.77, making the Agent capable of executing remediation steps inside a private environment rather than just collecting data from it. The instrumentation trajectory follows the same arc: passive collection first, then typed span classification, then evaluation signal capture, then cost aggregation, then autonomous analysis. The convention maturation at v1.37 represents approximately the second stage of that arc. The work that remains in the convention and in the SDK implementations corresponds to stages three through five.

The Shri Subramanian talk from the April 7, 2026 Datadog event on practical AI-enabled observability makes the same point from the evaluation side (DASH talk, "Practical AI-Enabled Observability for Agents and LLMs," April 7, 2026). Subramanian describes the development feedback loop for AI agents as fundamentally different from the classic application development loop: the classic loop iterates on code, the AI agent loop iterates on prompts, models, and tool descriptions. Instrumentation that captures code execution but not the prompt-model-tool decision surface leaves the AI agent development loop without the data it needs to close. The eval-driven development methodology Subramanian describes, in which evaluation scores serve as the CI system for AI agent iterations, requires instrumentation that goes beyond span emission to include evaluation annotation on every agent invocation that matters.

The resolution is not complicated, but it requires a design decision at the application architecture stage rather than a retrofit in production. Teams that identify their manual instrumentation boundaries at design time, add decorator wrapping for custom tools and retrieval functions, enforce context propagation across asynchronous boundaries, annotate evaluation signals as first-class span attributes, and aggregate per-span costs into root-span totals will have complete, useful traces from the first production deployment. Teams that instrument the framework and expect the auto-instrumentation to cover everything will produce the diagnostic incomplete trace that was the first symptom the fintech team encountered, and will spend the next two days fixing it.

The conventions exist. The libraries are improving with each release cycle. The MCP bridge is creating new patterns for how telemetry feeds back into development. What has not changed, and what the instrumentation landscape at Q2 2026 makes clear, is that observability is a design choice that has to be made with the same intentionality as the application architecture. The Cheap Code Effect, introduced in Chapter 3 as the phenomenon in which AI coding assistants lower the marginal cost of code generation while leaving the observability burden unaddressed, applies directly here: AI-assisted development can generate a LangGraph orchestrator and a set of tool functions in an afternoon, but the instrumentation discipline that makes that orchestrator diagnosable in production is still a human design decision that no code generator makes automatically.

The auto-instrumentation ceiling is generous and rising. The manual instrumentation floor remains.

Instrumentation is where observability intent meets engineering execution. The gap between those two things is still a team's responsibility to close.

### Key Takeaways

Auto-instrumentation coverage at Q2 2026 is generous and rising: Datadog's ddtrace library provides zero-configuration span capture for the major AI frameworks on the Python path, with narrower coverage on Node.js and Java (OpenAI integrations only at dd-trace-java v1.51.0). The OTel GenAI semantic conventions at v1.37 provide a portable alternative for teams not using ddtrace. But coverage for the framework does not mean coverage for the application: custom tool layers, bespoke retrieval clients, and message-queue-based agent communication each require manual instrumentation that the auto-instrumentation ceiling does not reach.

- The **MCP Telemetry Bridge** inverts the traditional relationship between developer tool and observability platform: Claude Code and Cursor can query live Datadog telemetry during a coding session through the Datadog MCP Server, enabling real-time evidence-based debugging without context switching to a dashboard.
- The **Version Mismatch Trap** is the most common instrumentation failure in teams using OTel GenAI conventions: incompatibility between the SDK version, the framework integration version, and the semantic convention opt-in state causes spans to emit with missing or misclassified attributes that the backend drops silently.
- The **Evaluation Span Gap** requires deliberate remediation: evaluation signals (retrieval relevance scores, hallucination detector outputs, ground-truth comparison results) are not emitted by auto-instrumentation and must be attached explicitly via `LLMObs.annotate()` or the external evaluations API.
- The **Cost Attribution Blindspot** is resolved by one deliberate design step: after the root workflow span completes, aggregating child span cost estimates and annotating the root span with the session total turns twelve per-span cost estimates into one per-request cost figure that supports user-level cost governance and pricing decisions.
- LangChain, Microsoft AutoGen, and LlamaIndex require manual decorator wrapping at Q2 2026; Strands Agents requires explicit semantic convention opt-in; teams that assume a framework is auto-instrumented without checking the coverage matrix will produce the single-span trace that is the canonical instrumentation failure case.
- Instrumentation is a design decision, not a deployment enhancement: teams that define manual instrumentation boundaries, context propagation contracts, and evaluation annotation requirements at architecture time produce diagnosable systems from their first production deployment.
